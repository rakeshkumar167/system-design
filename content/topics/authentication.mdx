<h2 id="overview">What Authentication Is</h2>

Authentication answers one question: **are you who you claim to be?** A user types a username and a password, or taps "Sign in with Google", or presents a fingerprint — and the system has to decide whether to believe the claim. That is *authentication* (authN): verifying identity. It is distinct from *authorization* (authZ), which answers a different question — **now that I know who you are, what are you allowed to do?** Authentication proves you are Alice; authorization decides whether Alice may delete this record. The two are constantly confused (and even the OAuth 2.0 spec, as we'll see, blurs the line), but they are separate concerns with separate machinery — authorization, with its roles and policies, is [its own topic](/topics/security/authorization). This topic is about proving identity.

The reason authentication is more than a single password check is **HTTP is stateless**. Each request arrives independently, carrying no memory of the last one, so proving your identity *once* at login isn't enough — every subsequent request has to re-establish *who you are* without making you log in again. The entire topic is really about that: take a claim of identity plus some evidence, verify it, and then carry the *proven identity* across a stream of stateless requests securely, at scale, and revocably. Everything that follows — sessions, tokens, JWTs, OAuth, OIDC, MFA — is a different answer to "how do I prove identity and then keep proving it cheaply and safely."

<Callout variant="info" title="Authentication vs Authorization">
**Authentication (authN)** = *who are you?* — verifying identity (this topic). **Authorization (authZ)** = *what may you do?* — evaluating permissions (a separate topic: RBAC, ABAC). A request is first authenticated (identity established), then authorized (permission checked). Conflating them is the source of a surprising number of security bugs — for example, treating "the token is valid" (authN) as if it also meant "this action is allowed" (authZ).
</Callout>

The evidence you present is a **factor**, and factors come in three kinds: something you **know** (a password), something you **have** (a phone, a security key), and something you **are** (a fingerprint, a face). Strong authentication combines factors — that's multi-factor authentication, later in this topic. But it all starts with a single login that exchanges some factor for a credential the system will trust on later requests.

<h2 id="credentials-passwords">Credentials & Password Login</h2>

The oldest and still most common factor is the **password** — something you know. The login flow is deceptively simple: the user submits a username and password, the server verifies them, and on success it issues something (a session or a token) that authenticates subsequent requests. The security lives entirely in that middle step: **how the server stores and verifies the password**.

The cardinal rule is that the server **never stores the password itself**. If the database is breached — and you must assume it eventually will be — plaintext passwords hand the attacker every account instantly, plus every *other* site where the user reused that password. Instead, the server stores a **salted, slow cryptographic hash** of the password. At login it hashes the submitted password the same way and compares hashes; the original is never recoverable. The hash must be **slow** (Argon2, bcrypt, scrypt — deliberately expensive functions) so that an attacker who steals the hashes can't brute-force billions of guesses per second, and **salted** (a unique random value per user) so that identical passwords don't produce identical hashes and precomputed "rainbow table" lookups are useless. The *details* of choosing and tuning those functions are involved enough to be [their own topic](/topics/security/password-hashing) — here the takeaway is simply: **never plaintext; always a salted slow hash.**

Login is also where you meet your first attackers head-on. **Brute force** (guessing one account's password repeatedly) and **credential stuffing** (replaying username/password pairs leaked from other breaches) are constant background noise on any login endpoint. The defenses are layered: [rate-limit and lock out](/topics/security/rate-limiting) repeated failures, monitor for credential-stuffing patterns, and — most effective — require a **second factor** so a correct password alone isn't enough. A password is a weak factor on its own (users pick guessable ones and reuse them everywhere); the modern posture is to treat the password as one factor among several, or to replace it entirely with passkeys (covered under MFA).

<h2 id="sessions-vs-tokens">Sessions vs Tokens</h2>

Once login succeeds, the server must issue something the client sends on every later request to prove "I already authenticated." There are two fundamentally different ways to do this, and choosing between them shapes the whole system. This is **the** central decision in authentication.

**Stateful sessions.** The server creates a **session record** — a row in a session store (often Redis or a database) holding the user's identity and metadata — and hands the client an **opaque session id** (a random string with no meaning of its own), typically in a cookie. On each request the client sends the cookie, and the server **looks the id up in the session store** to recover who the user is. The identity lives on the *server*; the client just holds a pointer to it.

**Stateless tokens.** The server creates a **signed, self-contained token** — a [JWT](#jwt-internals) — that *carries the identity and claims inside it*, cryptographically signed so it can't be tampered with. The client stores the token and sends it on each request, and the server **verifies the signature locally** to trust the contents. There is no lookup: the token *is* the identity, and anyone holding the signing key (or the public half of it) can validate it without consulting a shared store.

The trade-off is sharp and runs through the rest of this topic. Stateful sessions are **trivially revocable** — to log someone out everywhere, you delete the record — but they put a **store lookup on the hot path of every request** and are awkward across many services (each must reach the same session store). Stateless tokens **scale beautifully** — no per-request shared read, and any service can verify them independently, which is why they dominate microservice and API architectures — but they are **hard to revoke**: a signed token is valid until it expires, and you can't "delete" a record that was never stored server-side. The [capacity section](#capacity-estimates) makes that cost concrete, and the [token-lifecycle section](#token-lifecycle) confronts the revocation problem head-on.

<h2 id="capacity-estimates">Capacity: The Cost of Verifying</h2>

The sessions-vs-tokens choice is usually argued on feel ("tokens scale better"). It's more convincing as a number. Every authenticated request must verify identity — that work is *unavoidable*. The design decision is **where** the check happens: against a central datastore (stateful) or locally on the server (stateless). Let's size both for a mid-large service.

The numbers tell the whole story. Fifty million daily users making 200 authenticated requests each is ~**115,740 auth checks/sec** on average, and with a 3× peak factor ~**347,222/sec** at the busy hour. That peak load is identical for both models — verifying identity is mandatory either way. The difference is the **mechanism**. With **stateful sessions**, each of those ~347k requests is a **read against the central session store**, so the store must sustain ~347k reads/sec — about **7 session-store nodes** that sit directly on the hot path of *every* request and must be replicated, sharded, and kept highly available, because if they're down, *nobody* can make an authenticated request. With **stateless tokens**, that per-request central read is **zero**: verification is a local signature check (CPU on the server already handling the request), so there is no shared datastore to size, scale, or keep alive in front of your traffic.

That is the concrete meaning of "tokens scale": they convert a **stateful, central, hot-path read** into **stateless, local CPU**, removing a shared dependency from the critical path of every request. And it previews the catch — the very self-containment that eliminates the lookup is exactly *why you can't revoke a token by deleting a record*. There's nothing to delete. The [token-lifecycle section](#token-lifecycle) is where that bill comes due.

<h2 id="jwt-internals">JWT Internals</h2>

The stateless token in practice is almost always a **JWT** — a JSON Web Token. Understanding its anatomy demystifies most of modern authentication. A JWT is three Base64URL-encoded parts joined by dots: **`header.payload.signature`**.

- The **header** names the signing algorithm, e.g. `{"alg": "RS256", "typ": "JWT"}`.
- The **payload** is a JSON object of **claims** — facts about the user and the token. Standard registered claims include `sub` (subject — the user id), `iss` (issuer — who minted it), `aud` (audience — who it's for), `exp` (expiry timestamp), and `iat` (issued-at). You can add custom claims (roles, email) too.
- The **signature** is computed over the header and payload with a secret or private key, and is what makes the token **tamper-proof**: change a single byte of the payload and the signature no longer matches.

The single most important — and most misunderstood — property: **a JWT is signed, not encrypted.** The payload is merely Base64-encoded, which is *not* secrecy — anyone holding the token can decode and read every claim. The signature guarantees **integrity and authenticity** (it hasn't been altered, and it came from someone holding the key), but it provides **no confidentiality**. Therefore: **never put secrets in a JWT payload** (no passwords, no sensitive PII) — treat it as public, readable data that merely can't be forged.

Signing comes in two flavors, and the choice matters at scale. **Symmetric (HMAC, e.g. HS256)** uses a single shared secret to both sign and verify — simple, but every verifier needs the secret, which means every verifier can also *mint* tokens. **Asymmetric (RSA/ECDSA, e.g. RS256)** uses a **private key to sign** and the corresponding **public key to verify** — so the authorization server holds the private key, and any number of independent services verify with the freely distributable public key, *without* the power to issue tokens. Asymmetric signing is what makes JWTs work cleanly across many services. Verification, then, is: recompute/verify the signature with the right key, and check the claims — crucially `exp` (not expired), `aud` (intended for me), and `iss` (from someone I trust).

<h2 id="oauth2-delegated">OAuth 2.0</h2>

So far identity has been between a user and one application. **OAuth 2.0** solves a different, harder problem: letting a user grant **one application limited access to their data on another service** — *without sharing their password*. When a photo-printing app wants to access your Google Photos, you don't hand it your Google password; you're redirected to Google, you consent, and the app receives a scoped, revocable **access token**. That's OAuth: **delegated authorization**.

It defines four roles. The **resource owner** (the user) owns the data. The **client** (the third-party app) wants access. The **authorization server** (Google's login) authenticates the user and issues tokens. The **resource server** (the Google Photos API) holds the data and accepts the token. The canonical, secure dance between them is the **authorization-code flow**: the client redirects the user's browser to the authorization server; the user logs in and consents; the authorization server redirects back with a short-lived **authorization code**; the client then exchanges that code (server-to-server) at the **token endpoint** for an access token. The indirection exists so the access token never passes through the browser's URL/history. For mobile and single-page "public" clients that can't keep a secret, **PKCE** (Proof Key for Code Exchange) adds a one-time `code_verifier`/`code_challenge` pair so a stolen authorization code is useless to an attacker who lacks the verifier.

There are other grant types — **client credentials** for machine-to-machine access (no user involved), and the legacy **implicit** and **resource-owner-password** grants, both now **discouraged** in favor of authorization-code-plus-PKCE. But the crucial conceptual point, the one interviewers probe: **OAuth 2.0 is an *authorization* framework, not an *authentication* one.** An access token grants *access to an API*; it says nothing reliable about *who the user is*. Apps that tried to use a raw OAuth access token as "proof of login" created real security holes (the token could have been minted for a different app entirely). Plugging that gap is precisely why **OpenID Connect** exists.

<h2 id="oidc">OpenID Connect (OIDC)</h2>

**OpenID Connect** is a thin **identity layer on top of OAuth 2.0**. It reuses the exact same authorization-code flow but adds the missing piece: alongside the access token, the authorization server also returns an **ID token** — a **JWT that asserts who the user is**, with standardized identity claims (`sub` for the stable user id, plus optionally `name`, `email`, etc.) and signed by the provider so the client can verify it. OIDC also defines a `userinfo` endpoint for fetching profile data and a discovery document so clients can configure themselves automatically.

The key distinction to keep straight is the **two tokens** and who each is *for*:

- The **ID token** is for the **client** — it proves *who the user is*. The client verifies its signature and reads `sub` to know the logged-in identity. This is the authentication piece.
- The **access token** is for the **resource server (API)** — it grants *access to call the API*. The client treats it as an opaque bearer credential and forwards it; it should not try to read it as identity.

Mixing them up — using the access token as proof of identity, or sending the ID token to the API as an access credential — is a classic mistake. With this layer, "Sign in with Google / Apple / Microsoft" becomes genuine, standardized **single sign-on**: one identity provider authenticates the user, issues an ID token proving it, and many relying applications trust that provider instead of each maintaining their own passwords. OIDC is the backbone of modern federated login and enterprise SSO.

<h2 id="mfa">Multi-Factor Authentication</h2>

A password is a single factor, and a weak one — it can be guessed, phished, reused, or leaked in a breach. **Multi-factor authentication (MFA)** strengthens the proof by requiring **two or more factors from *different* categories**, so compromising one isn't enough. The three categories, again: something you **know** (password, PIN), something you **have** (a phone, a hardware security key), and something you **are** (fingerprint, face). Two passwords aren't MFA — they're the same category; the security comes from the factors being *independent*, so an attacker would have to defeat two unrelated things at once.

The common second factors, roughly weakest to strongest:

- **SMS one-time codes** — better than nothing, but the **weakest** second factor: vulnerable to SIM-swap attacks (an attacker ports your number) and interception. Avoid where you can.
- **TOTP** (Time-based One-Time Password, RFC 6238) — an authenticator app and the server share a secret seed at setup, and both derive the same 6-digit code from the seed plus the current 30-second time window. No network needed, no SMS to intercept. The ubiquitous "authenticator app" model.
- **WebAuthn / passkeys** — public-key credentials bound to the device (and often a biometric). The device holds a private key; the server holds the public key and verifies a signed challenge. Critically, passkeys are **phishing-resistant**: the credential is cryptographically scoped to the real site's origin, so a lookalike phishing page can't elicit a usable response. This is the direction the industry is moving — passkeys can *replace* the password entirely, not just supplement it.

MFA doesn't have to fire on every action. A common pattern is **step-up authentication**: authenticate normally for routine use, then demand a second factor only for sensitive operations (changing a password, moving money). The principle: combine **independent factors** so a single stolen credential — especially a phishable password — can't by itself grant access.

<h2 id="token-lifecycle">Token Lifecycle: Refresh & Revocation</h2>

Stateless tokens scale because nothing is stored server-side — which is exactly why they're **hard to revoke**, the bill foreshadowed in the [capacity section](#capacity-estimates). If a JWT is stolen, it stays valid until its `exp`. You can't delete what was never stored. The lifecycle design exists to manage that risk.

The standard pattern is **short-lived access token + long-lived refresh token**. The **access token** (the JWT used on every API call) is given a **short lifetime** — minutes — so that a leaked token is useful only briefly. To avoid forcing a fresh login every few minutes, the client also holds a **refresh token**: a longer-lived, *stored, revocable* credential whose only job is to mint new access tokens at the token endpoint when the old one expires. This gets the best of both worlds — the per-request hot path stays stateless (verify the access-token signature, no lookup), while the refresh token, used rarely, *is* tracked server-side and can be revoked to cut off access.

Refresh tokens introduce their own risk (they're long-lived and powerful), mitigated by **rotation with reuse detection**: each time a refresh token is used, it's invalidated and a new one issued. If an *old, already-used* refresh token is ever presented again, that means it was stolen and replayed — so the server **revokes the entire token family**, logging out the attacker and the legitimate user (who can simply log in again). This turns a silent theft into a detectable, contained event.

For revoking **access tokens** before expiry, you have two honest choices, and they trade directly against the [capacity lesson](#capacity-estimates):

- **Short TTLs only** — accept that a token is valid until it expires, and keep that window tiny. No per-request lookup; you tolerate a small revocation lag. This preserves the stateless-scaling win.
- **A denylist or token introspection** — check each token against a server-side list of revoked tokens (denylist) or ask the authorization server "is this still valid?" (introspection) on each request. This gives *immediate* revocation — but it **re-introduces exactly the per-request central lookup** that statelessness was meant to eliminate, partially recreating the stateful model from the [sessions-vs-tokens section](#sessions-vs-tokens).

There's no free lunch: instant revocation costs a lookup. Most systems land on short-lived access tokens (small revocation window, fully stateless) plus revocable refresh tokens (rotation + reuse detection), reserving denylists for high-stakes immediate-kill scenarios.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

Authentication is a domain where small mistakes become full compromises. The recurring ones:

<Callout variant="warning" title="Authentication footguns">
- **`alg: none` / algorithm-confusion attacks.** Early JWT libraries honored an `alg: none` header (no signature) or could be tricked into verifying an RS256 token with the public key as an HMAC secret. **Always pin the expected algorithm** server-side; never let the token's own header choose how it's verified.
- **Storing tokens in `localStorage`.** It's readable by any JavaScript, so a single XSS bug exfiltrates the token. Prefer **`HttpOnly`, `Secure`, `SameSite` cookies** (inaccessible to JS), and pair them with CSRF defenses (`SameSite` plus anti-CSRF tokens) since cookies are sent automatically.
- **Secrets in the JWT payload.** It's signed, not encrypted — the payload is public. No passwords, no sensitive PII.
- **Not validating all the claims.** Always verify the **signature**, and check **`exp`** (expired?), **`aud`** (for me?), and **`iss`** (from a trusted issuer?). A valid signature on a token meant for a different audience is still an attack.
- **Long-lived access tokens.** A leaked token is valid for its whole lifetime; keep access-token TTLs short and lean on refresh tokens.
- **Treating authentication as authorization.** "The token is valid" (authN) is not "this action is allowed" (authZ). Check permissions separately.
- **Weak login hygiene.** Rate-limit and lock out brute force/credential stuffing, offer MFA, and require step-up auth for sensitive actions.
</Callout>
