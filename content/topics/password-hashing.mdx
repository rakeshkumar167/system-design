<h2 id="overview">What Password Hashing Is</h2>

Sooner or later, someone steals your database. A leaked backup, a SQL-injection hole, an over-broad cloud credential — the exact route varies, but you should design as if the `users` table *will* end up in an attacker's hands. Password hashing is the discipline that decides what happens next: whether the breach hands the attacker everyone's password, or a pile of values that are expensive-to-useless. The goal is simple to state and easy to get wrong — **a stolen credential store must not yield usable passwords**.

You achieve that by never storing the password at all. Instead you store a **one-way transformation** of it — a *hash* — computed with a **unique random salt** and a **deliberately slow** function. There is no key that reverses it and no one, not even the people who run the system, can recover the original from what's stored. At login you don't *decrypt* anything; you **recompute** the same transformation on the password the user just typed and check that it matches the stored value.

<PasswordRegistrationSequence />

This one property — irreversibility — is what makes password hashing different from almost everything else in a system, and it drives every decision in this topic: why you hash rather than encrypt, why you add a salt, why you make the function *slow* on purpose, which algorithms are safe, what it costs to run at scale, and the handful of mistakes that turn a "we were breached but passwords are safe" incident into a catastrophe.

<Callout variant="info" title="The one rule">
You never store, log, or transmit the plaintext password. What you persist is a **salted, slow, one-way hash** — plus the salt and the parameters needed to recompute it. Login is always *recompute-and-compare*, never *decrypt-and-check*. Everything else in this topic is a consequence of taking that rule seriously.
</Callout>

<h2 id="hashing-vs-encryption">Hashing vs Encryption</h2>

The single most important distinction is between **encryption** and **hashing**, because confusing them is how passwords end up recoverable.

- **Encryption is reversible.** It transforms plaintext into ciphertext *using a key*, and anyone with the key can transform it back. That reversibility is the point — you encrypt data you need to read again later. (Transport security, the [TLS](/topics/security/tls-https-certificates) topic, is all about encryption.)
- **Hashing is one-way.** A cryptographic hash maps an input to a fixed-size digest with no key and no inverse. You cannot "un-hash" a value; the only way to learn which input produced a digest is to hash candidate inputs and look for a match.

Passwords must be **hashed**, never encrypted, and the reason is about *what an attacker has to steal*. If you encrypt passwords, the decryption key exists somewhere — in config, an HSM, an environment variable — and whoever gets both the database and that key gets **every password in plaintext, instantly**. The key is a single point of catastrophic failure. A hash has nothing equivalent to steal: there is no value an attacker can lift that turns the whole table back into passwords. The best they can do is *guess*, one candidate at a time.

There's a second subtlety: a *password* hash is not the same as a *general-purpose* cryptographic hash. SHA-256 and its relatives are hashes, and they're one-way, but they're designed to be **fast** — which, as the next two sections show, is exactly the wrong property for passwords. So "hash your passwords" is necessary but not sufficient; *how* you hash is what actually protects them.

<h2 id="salting">Salting & Rainbow Tables</h2>

A one-way hash alone isn't enough, because attackers don't have to reverse the math — they can **precompute** it. Take a plain unsalted hash like `SHA-256(password)`: the digest for any given password is always the same, everywhere, forever. So an attacker computes the hashes of billions of common passwords *once*, stores them in a giant lookup table — a **rainbow table** — and then cracks any stolen database by looking up each digest. The expensive work is done ahead of time and reused against every victim.

A **salt** destroys that economy. A salt is a **unique, random value generated per password** and stored — in the clear — right alongside the hash. You hash `salt || password` instead of the password alone:

```
alice:  salt=9f3a…  hash=H(9f3a… || "hunter2")
bob:    salt=b27c…  hash=H(b27c… || "hunter2")
```

Two things change. First, a precomputed table is now worthless: to attack Alice the attacker would need a rainbow table built for salt `9f3a…`, and a *different* table for Bob's `b27c…`, and so on for every user — precomputation no longer amortizes across accounts, so the attacker is forced back to attacking each password individually. Second, notice that Alice and Bob both chose `hunter2` but their stored hashes are completely different — the salt hides the fact that they share a password, which an unsalted scheme would leak in plain sight.

The salt does **not** need to be secret — it's fine that it sits next to the hash — because its job isn't confidentiality, it's *uniqueness*. It just has to be **long, random, and unique per password** (a fresh 16 bytes from a cryptographically secure RNG). What salting does *not* do is stop an attacker who targets a single account from guessing that one password; it defeats *precomputation and bulk attacks*, not brute force. Stopping brute force is the job of the next section.

<h2 id="slow-hashing">Slow Hashing & the Work Factor</h2>

Salting kills *precomputation*, but it doesn't slow down *guessing*. Given the salt (which the attacker has, since it's stored alongside the hash), they can still run through a dictionary — `password`, `123456`, `qwerty`, leaked passwords from other breaches — hashing each candidate with that salt and checking for a match. Whether this attack succeeds comes down to **how many guesses per second** they can make.

This is where general-purpose hashes fail catastrophically. MD5 and the SHA family are engineered to be **fast** — you want to checksum a gigabyte file quickly — so a single GPU can compute *billions* of them per second. Point that at a salted SHA-256 database and common passwords fall in seconds. The very property that makes SHA-256 a good general hash makes it a terrible password hash.

The fix is to make the hash **deliberately slow** and, crucially, **adaptively** slow via a tunable **work factor** (also called the *cost*). Password hashing functions expose a knob — bcrypt's cost, PBKDF2's iteration count, Argon2's time and memory parameters — that you turn up until a single hash takes a target amount of wall-clock time, typically **200–300 ms** on your hardware. That's imperceptible to a user logging in once, but it's a devastating multiplier for an attacker who must try *billions* of candidates: the same guessing that took seconds against SHA-256 now takes years or millennia.

The work factor is a **living setting**, not a constant. Hardware gets faster every year, so a cost that took 250 ms in 2020 might take 60 ms today — meaning you must **raise the work factor over time** to hold the line. (Section 8 shows how you upgrade already-stored hashes when you do.) The guiding idea: **tune the cost to the highest value your login traffic can afford**, because every millisecond you add to your own hashing multiplies straight through the attacker's entire keyspace.

<h2 id="algorithms">The Algorithms: Argon2, scrypt, bcrypt, PBKDF2</h2>

There are four password-hashing functions worth knowing, and the property that separates the modern ones from the legacy ones is **memory-hardness**. A slow hash that uses little memory can still be **massively parallelized** on GPUs and custom ASICs, which pack thousands of cheap compute cores — the attacker just runs a hundred thousand slow hashes at once. A **memory-hard** function deliberately forces each computation to use a large, non-compressible block of **RAM** (tens or hundreds of MiB). Since GPUs and ASICs have comparatively little memory per core, memory-hardness neutralizes their parallelism advantage: the attacker can no longer cheaply fan out.

Reading the table top to bottom is roughly reading history. **PBKDF2** (2000) just repeats an HMAC many times — it's slow but *not* memory-hard, so it's the weakest against GPUs, and you should reach for it only when a compliance regime (FIPS) demands an approved primitive. **bcrypt** (1999) is battle-tested and everywhere; it has a modest fixed memory footprint (so it resists GPUs better than PBKDF2 but isn't strongly memory-hard) and a well-known quirk: it silently **truncates input to ~72 bytes**, which matters for long passphrases. **scrypt** (2009) introduced serious, tunable memory-hardness. **Argon2** (2015), specifically the **Argon2id** variant, won the Password Hashing Competition and is the modern default: it lets you tune **time, memory, and parallelism** independently and is the strongest general recommendation.

The practical guidance is short: **use Argon2id for new systems**; use **bcrypt** if Argon2 isn't available in your stack (it's a perfectly respectable choice); use **PBKDF2** only under FIPS constraints; and never invent your own. Whichever you pick, the *parameters* matter as much as the choice — an Argon2 with trivially low memory is weaker than a well-tuned bcrypt.

<h2 id="capacity-estimates">Capacity: The Cost of Hashing</h2>

Password hashing is the one place in system design where **being slow is the goal**, so its cost is real and has to be planned for. Unlike most work, you can't cache it away or make it cheaper — the whole point is that each hash burns a fixed, deliberate amount of CPU and (for memory-hard functions) RAM. That cost is paid **once per login**, so you size it against your **peak login rate**, and you make sure hashing happens *only* at login — never on every request, and never anywhere an attacker can trigger it cheaply.

At 5,000 logins/sec with a 250 ms hash, the arithmetic is stark: you need ~**1,250 CPU cores** just for hashing, and because a 64 MiB memory-hard function keeps ~1,250 hashes in flight at once, ~**80,000 MiB (~78 GiB)** of RAM at peak. That's a serious, dedicated budget — which is exactly why hashing lives *only* on the login path and why login is **rate-limited**: if an attacker could make you hash freely, that 1,250-core cost becomes a **CPU-exhaustion denial of service** (hash a few thousand fake logins per second and the auth tier falls over).

Now flip to the attacker's side, and the same 250 ms that's a rounding error for you becomes the whole point. Your server pays the cost *once* per real login; an offline attacker pays it for *every one of billions* of guesses. Against a fast general-purpose hash a single GPU manages ~**10 billion guesses/sec**; against this tuned memory-hard hash it manages ~**2,000/sec** — a **5,000,000× slowdown**. That factor is the entire game: it's the difference between cracking a breached database in an afternoon and not cracking it in a human lifetime. You are, quite literally, buying the attacker's time in units of your own login latency.

<h2 id="peppering-storage">Peppering & Credential Storage</h2>

Salting and slow hashing assume the attacker has your whole database — salt and all. A **pepper** adds one more layer for the common case where they *don't* get everything. A pepper is a **secret** value combined with every password during hashing, but — unlike the salt — it is **not stored in the database**. It lives somewhere the database backup doesn't: an HSM, a KMS, or application configuration, ideally applied via a keyed function (an HMAC) rather than simple concatenation.

The threat it addresses is the *most common* kind of breach: a **database-only** compromise — a dumped backup, a SQL-injection read, a snapshot leak — where the attacker gets the `users` table but **not** the application's secret store. Without the pepper, every stolen hash is uncrackable, because the attacker is missing an input they can't guess. Peppering is genuinely **defense in depth**: it does nothing extra once an attacker has fully compromised the app server *and* the database, but it turns the frequent "just the database leaked" incident from a slow-motion password-cracking disaster into a non-event.

How you **store the credential** matters just as much as how you compute it. The established practice is to store a single **self-describing** string that carries the algorithm, its parameters, the salt, and the digest together — the **PHC / Modular Crypt Format** you'll recognize from real hashes:

```
$argon2id$v=19$m=65536,t=3,p=4$c2FsdHNhbHQ$aGFzaGhhc2hoYXNo
   algo    ver     params        salt          digest
```

Because the record names its own algorithm and cost, the system is **self-upgrading**: you can raise the work factor, or migrate from bcrypt to Argon2, and old and new records coexist — each hash is verified with the parameters baked into *it*, and upgraded opportunistically (next section). The one thing you don't put in this string is the pepper: it's applied at compute time from the secret store and never written next to the hash, which is the whole point.

<h2 id="verification-in-practice">Verification in Practice</h2>

Verification is deceptively simple: look up the stored credential record for the username, read the salt and cost parameters *out of that record*, recompute the hash of the submitted password with those exact parameters, and check it against the stored digest. Because the record is self-describing, verification always uses the parameters that hash was created with — which is what lets different costs and even different algorithms coexist during a migration.

<PasswordVerificationSequence />

Three things make this correct rather than merely functional:

- **Constant-time comparison.** Compare the recomputed digest to the stored one with a **constant-time** equality function, never a plain `==`. A naive comparison returns as soon as it finds a mismatched byte, and that early exit leaks — through timing — *how many* leading bytes matched, which an attacker can exploit to reconstruct the target digest byte by byte. Constant-time comparison always examines the full length.
- **Upgrade-on-login (rehashing).** Raising the work factor only helps if stored hashes actually move to the new cost, but you can't rehash a password you don't have. The trick: you *do* have it, briefly, **every time the user logs in successfully**. So after a successful verify, check whether the stored parameters are below current policy; if so, transparently **rehash the plaintext at the new cost and overwrite** the stored record. Active accounts upgrade themselves silently over time.
- **Uniform timing and errors.** If "no such user" returns instantly while "wrong password" takes 250 ms, an attacker learns which usernames exist just by timing responses — a **user-enumeration** leak. Defend it by hashing against a dummy record (or otherwise spending the same time) on the no-user path, and by returning an **identical error message** for bad-user and bad-password. The login result should reveal only "these credentials are wrong," never *which* half.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

Most password-hashing disasters aren't subtle cryptographic breaks — they're a handful of well-known mistakes, made over and over.

<Callout variant="warning" title="Password-hashing footguns">
- **Using a fast/general-purpose hash.** Storing `SHA-256(password)` or `MD5` with no stretching is the single most common breach amplifier — salted or not, a GPU cracks common passwords in seconds. Use a purpose-built slow hash.
- **No salt, or one global salt.** A missing or shared salt puts you right back in rainbow-table territory and leaks which users share a password. Use a **fresh, random, per-password** salt.
- **Home-rolled schemes.** "SHA-256 iterated a thousand times" or "hash-then-hash-again" invariably underperforms a vetted function and often introduces subtle flaws. Never invent your own — use Argon2id/bcrypt/scrypt/PBKDF2.
- **Unbounded input → login DoS.** A memory-hard hash over a 10 MB "password" lets an attacker exhaust your CPU/RAM one request at a time. **Cap password length** (e.g. 128 chars) before hashing and **rate-limit login**.
- **Never raising the cost.** A work factor tuned years ago is now cheap for attackers. Revisit it periodically and use **upgrade-on-login** to migrate.
- **Leaking the plaintext.** Passwords land in request logs, crash dumps, analytics, and error traces. Scrub them everywhere — the safest password is one your logs never saw.
- **Non-constant-time comparison.** A plain `==` on digests is a timing side channel. Always use a constant-time equality.
</Callout>
