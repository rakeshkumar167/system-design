<h2 id="overview">What API Security Is</h2>

Most modern systems are, from the outside, a collection of **APIs** — the HTTP (or gRPC/GraphQL) endpoints that mobile apps, single-page frontends, partners, and other services call. That makes APIs the **primary attack surface**, and securing them is different from securing a traditional server-rendered website in a few decisive ways. The client is **untrusted machine code**: there's no browser enforcing same-origin rules for you, no human to slow down with a CAPTCHA, and nothing stopping an attacker from scripting millions of requests and reading your published API docs to find every endpoint. Worse, APIs tend to **expose business logic and data directly** — an endpoint often maps straight onto a database object — so a single missing check leaks records at scale.

The good news is that the principles are the ones you already know from the rest of this Security series: authenticate the caller, authorize every request, validate input, encrypt in transit, and limit consumption. API security is largely about applying those rigorously at the API layer, where the human-facing mitigations don't exist. The field's reference is the **OWASP API Security Top 10**, a list deliberately *separate* from the [general web-app Top 10](/topics/security/owasp-top-10) because APIs fail in their own characteristic ways — and its **#1 risk is authorization**, specifically failing to check that a caller may access the specific object they asked for.

<Callout variant="info" title="Assume a hostile machine">
An API endpoint is a promise to a **machine**, not a person. Assume every client is **hostile, scripted, and reading your documentation** — it will send fields you didn't expect, iterate every id, and replay requests a million times. There's no UI to hide behind and no human to reason with, so every control has to live on the **server**. The whole topic follows from taking that assumption seriously.
</Callout>

<h2 id="authenticating-clients">Authenticating API Clients</h2>

The first question an API asks is *who is calling?* There are three common answers, and choosing well means knowing what each actually proves.

- **API keys** — a long random secret string the client sends on each request. A key identifies an **application** (or a project/account), not an end user, and it's simple to issue. But it's a **bearer credential** with no built-in expiry and weak guarantees: anyone who obtains the key *is* the client, and keys leak notoriously (checked into repos, embedded in mobile apps and JavaScript). Keep them secret, scope them narrowly, and rotate them.
- **OAuth2 bearer tokens / JWTs** — the standard for **delegated user access**. The client presents a token (often a JWT) that the API validates; the token encodes who the user is and what scopes were granted. This is the mainstream choice for user-facing APIs (the [Authentication](/topics/security/authentication) topic covers the flows and token internals).
- **mTLS (mutual TLS)** — both client and server present certificates, so each cryptographically proves its identity. It's strong **service-to-service** authentication inside a trusted network or mesh (the [TLS](/topics/security/tls-https-certificates) topic covers certificates).

The interview-level point: **an API key is identification, not strong authentication**. It tells you which application is calling, not that the call is trustworthy — and by itself it authorizes nothing. Whatever mechanism you choose, authentication only establishes *who*; the far more commonly botched step is deciding *what they may do*.

<h2 id="authorizing-requests">Object- & Function-Level Authorization</h2>

The two most damaging API risks are both **authorization** failures — the app authenticates the caller correctly but then fails to check what they're allowed to do.

**BOLA (Broken Object-Level Authorization)** is the **#1** API risk. The endpoint verifies the caller is logged in, looks up the object by the id in the request, and returns it — **without checking that the object belongs to the caller**. So an attacker with a perfectly valid token changes `GET /accounts/1001` to `/accounts/1002` and reads someone else's data. Because APIs expose objects by id and attackers can iterate ids trivially, this one flaw leaks entire datasets.

<BolaAttackSequence />

**BFLA (Broken Function-Level Authorization)** is its sibling for *actions*: a normal user invokes an **administrative** function — `DELETE /users/5`, `POST /admin/config` — that was merely hidden from their UI but never protected on the server. The client isn't limited by what buttons it was shown; it can call any endpoint it discovers.

Both fixes are the same discipline, and it's exactly the [Authorization](/topics/security/authorization) topic: enforce authorization **on the server, on every request**, checking both **object level** (does this caller own or have a relationship to *this specific* record?) and **function level** (is this caller's role permitted *this operation*?), and **deny by default**. Scopes from an OAuth token are coarse ("this app may read orders") and are *not* a substitute for the per-object check.

<h2 id="data-exposure">Data Exposure & Mass Assignment</h2>

Two mirror-image risks come from the API being careless about *which fields* cross the boundary — the read side and the write side.

**Excessive data exposure** (the read side): the endpoint serializes the **whole object** and trusts the client to display only the relevant parts. But "the mobile app only shows the name" is irrelevant — the full JSON, including `ssn`, `passwordHash`, `internalNotes`, and `isAdmin`, went over the wire, and an attacker just reads it straight from the response. The fix is to **return only the fields the caller needs**, via an explicit response schema/DTO, never the raw database row.

**Mass assignment** (the write side): the endpoint **binds client-supplied JSON directly onto an internal object**, so the client can set fields the developer never intended to expose. A registration request that includes `"role": "admin"` or `"isVerified": true` silently escalates the account if those properties are blindly assigned. The fix is to **allowlist exactly which properties a client may set** and ignore the rest.

Both are really the same principle one level deeper than object authorization: **object property-level authorization** — controlling not just *which objects* a caller can touch but *which fields* of them they may **read** and **write**. Whitelist inputs, and shape outputs deliberately.

<h2 id="resource-consumption">Rate Limiting & Resource Consumption</h2>

APIs also fail when they let a single client consume **unbounded resources** — the OWASP API risk of **unrestricted resource consumption**. Without limits, one caller (malicious or just buggy) can exhaust CPU, memory, bandwidth, database connections, or a metered third-party dependency you pay for per call. It's also the front line against **brute-force** and **denial-of-service**.

The controls stack together:

- **Rate limits** — cap requests per client per time window (the classic token-bucket / sliding-window algorithms, covered in the forthcoming rate-limiting topic).
- **Quotas** — longer-horizon caps (daily/monthly), often per pricing tier.
- **Payload-size limits** — reject oversized request bodies before you parse or hash them (recall the login-DoS from the [Password Hashing](/topics/security/password-hashing) topic).
- **Pagination caps** — never return an unbounded list; enforce a maximum page size so `?limit=10000000` can't be used to exhaust memory.
- **Timeouts** — bound how long any single request may run.
- **Cost-based limiting** — weight expensive endpoints (a report, a GraphQL query that fans out) more heavily than cheap ones, so limits reflect actual work.

The unifying idea is that **every dimension of consumption needs a ceiling**. Rate limiting is the most visible, but payload size, page size, and execution time are equally exploitable if left unbounded.

<h2 id="capacity-estimates">Capacity: The Cost of Securing Every Request</h2>

Enforcing security on every request isn't free — verifying a token and checking a rate limit costs CPU — but that cost is small, and it *pays for itself* by rejecting abusive traffic before it reaches the expensive backend. A little arithmetic shows why doing security **early and at the edge** is also a capacity optimization.

<ApiSecurityCapacity
  assumptions={{
    requestsPerSec: 200000,

    backendCpuMs: 5,
    msPerCorePerSec: 1000,

/>

At 200,000 requests/sec, the edge security checks — verify the token, check the rate limit — cost about **50 cores**. That's the security tax, and it looks like pure overhead until you account for what it prevents. Suppose **half** the incoming traffic is abusive or unauthenticated (scrapers, credential stuffing, junk). If the gateway waved everything through, the backend would have to *serve* all 200k req/s at 5 ms each — **1,000 cores**. Reject that abusive half at the edge and only the legitimate 100k req/s reach the backend, which now needs just **500 cores**. So the 50-core gateway investment removes 500 cores of backend load — a **net saving of ~450 cores**.

Two lessons. First, **do cheap security checks early**: authentication and rate limiting at the gateway cost a fraction of serving a request, and every abusive request they drop is one the expensive backend never pays for — security is a capacity optimization, not just protection. Second, "early and cheap" applies only to the *coarse* checks; the **object-level authorization** that actually prevents BOLA can't be done at the gateway (it doesn't know who owns record 1002), so it stays deep in the service. Coarse and cheap at the edge, precise and correct in the service — the next section makes that split explicit.

<h2 id="api-gateway">The API Gateway as Enforcement Point</h2>

Rather than have every service reinvent authentication, rate limiting, and TLS, mature architectures put a hardened **API gateway** in front of everything and centralize the **coarse, cross-cutting** security there: **TLS termination**, **authentication** (validate the token/key), **coarse authorization** (does this token's scope cover this route?), **rate limiting and quotas**, request validation, a **Web Application Firewall**, and consistent **logging**. One well-audited front door beats dozens of inconsistent ones, and — as the capacity model showed — it's where you cheaply shed bad traffic.

<GatewayEnforcementSequence />

But the gateway is **necessary, not sufficient**. It sees the request in transit; it does **not** know your domain model, so it cannot answer "does *this* user own *this* order?" — the object-level check that stops BOLA. That must run **in the service**, where the data and ownership live. The correct design is **defense in depth**: coarse, cheap enforcement at the edge (authenticate, rate-limit, scope-check, reject early) and precise, correctness-critical **object- and function-level authorization** inside the service. Treating the gateway as the *only* line of defense is how "we have an API gateway" coexists with a wide-open BOLA vulnerability. (The [API Gateway](/learn/api-gateway) tutorial builds one end to end.)

<h2 id="transport-misconfig">Transport Security & Misconfiguration</h2>

The remaining risks are hygiene — unglamorous, and responsible for a large share of real incidents.

- **Always HTTPS, never plaintext.** Every endpoint over **TLS**, with **HSTS** to prevent downgrade, and no `http://` listener at all (the [TLS](/topics/security/tls-https-certificates) topic). An API token sent over plaintext is a token handed to anyone on the path.
- **Correct CORS.** Cross-Origin Resource Sharing should be an **allowlist of trusted origins**, never `Access-Control-Allow-Origin: *` combined with credentials — that hands your authenticated API to any website.
- **Security headers and safe errors.** Return **generic error messages**; a stack trace, a raw SQL error, or an internal hostname in a 500 response is free reconnaissance for an attacker. Log the detail server-side, return an opaque error to the client.
- **API inventory management.** You can't secure what you don't know you run. **Shadow APIs** (undocumented endpoints someone shipped) and **zombie APIs** (old, unpatched `/v1` versions still live after `/v2` launched) are dangerous precisely because they're forgotten — unmonitored and unpatched. Maintain an inventory and **retire old versions**.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

The API-security failures are consistent and, once named, easy to guard against.

<Callout variant="warning" title="API security footguns">
- **Trusting the client.** Hidden fields, client-side filtering, "our app won't send that" — every request is attacker-controllable. Enforce on the **server**.
- **API keys as authentication.** A key identifies an app, not a trustworthy request; leaked keys in mobile apps and front-end JavaScript are everywhere. Scope, rotate, and back them with real authorization.
- **No object-level authorization (BOLA).** The #1 API risk: check ownership of the specific object on **every** request; never trust a client-supplied id.
- **Returning whole objects / mass assignment.** Shape outputs to explicit schemas and allowlist writable fields — control which properties are read and written.
- **No resource limits.** Unbounded rate, payload size, or page size invites DoS and runaway cost. Cap every dimension.
- **Verbose errors and HTTP endpoints.** Generic errors only; TLS everywhere with HSTS.
- **Forgotten versions.** Zombie `/v1` endpoints outlive their patches. Inventory and retire.
</Callout>
