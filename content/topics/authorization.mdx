<h2 id="overview">What Authorization Is</h2>

Once a system knows **who** you are, it still has to decide **what you're allowed to do**. That second decision is **authorization** (authz), and it is a distinct concern from **authentication** (authn), which proves identity. The two are constantly confused because they run back-to-back on every request, but they answer different questions: authn answers *"who is this?"*, authz answers *"may this principal do this?"*. Authorization always runs **after** a verified identity — you cannot meaningfully authorize an anonymous, unproven caller, so a broken or skipped authn step poisons every authz decision downstream.

Every authorization decision is a verdict — **allow** or **deny** — over a small tuple:

- a **subject** — the authenticated principal (a user, a service, an API key);
- an **action** — the verb (`read`, `transfer`, `delete`);
- a **resource** — the object acted on (`invoice:8815`, the `/admin` route, a row in a table);
- a **context** — the surrounding conditions (time of day, source IP, tenant, whether MFA was used).

Framed that way, authorization is a pure function: given `(subject, action, resource, context)`, return `allow | deny`. The whole topic is about *how* systems compute that function — the **models** for expressing the rules (RBAC, ABAC, ReBAC), the **architecture** for evaluating them without scattering `if` statements through the code, the **cost** of doing it on every request, and the **failure modes** (chief among them forgetting to check that *this* user owns *this* object).

<Callout variant="info" title="authn vs authz">
**Authentication** proves *who you are* (see the [Authentication](/topics/security/authentication) topic — passwords, tokens, MFA). **Authorization** decides *what you may do* — and it always runs **after** authn, because you can't authorize an unknown caller. A single request does both: first verify the identity in the token, then decide whether that identity is permitted the action on the resource. Confusing the two — treating "logged in" as "allowed" — is the root of a whole class of security bugs.
</Callout>

<h2 id="access-control-models">Access-Control Models</h2>

There are four dominant ways to express *who may do what*, and picking the right one is the first design decision. They trade **granularity** against **administrative cost**.

- **ACL (Access Control List)** — each **object** carries a list of who-can-do-what: `file.txt → {alice: read-write, bob: read}`. Dead simple and intuitive (it's how Unix file permissions work), but the rules are **scattered across every object**, so "what can Alice access?" and "revoke Bob everywhere" become expensive scans. ACLs don't scale to millions of objects and thousands of users.
- **RBAC (Role-Based Access Control)** — users are granted **roles**, and roles carry **permissions**: `alice → billing-admin → {invoice:read, invoice:refund}`. The indirection collapses the user×permission matrix into a handful of roles, which makes assignment, audit, and reasoning tractable. It is the workhorse of enterprise authorization.
- **ABAC (Attribute-Based Access Control)** — decisions are **computed** by evaluating **policy rules** over **attributes** of the subject, resource, action, and environment: *allow `read` if `subject.department == resource.department`*. Fine-grained and dynamic, at the cost of harder-to-author policies and a need for attribute data at decision time.
- **ReBAC (Relationship-Based Access Control)** — permissions are derived from a **graph of relationships**: *"a user who is an **editor** of a folder can edit every document **in** that folder."* This is how consumer sharing (Google Drive, GitHub) works, where access flows through ownership, membership, and containment.

<h2 id="rbac">Role-Based Access Control (RBAC)</h2>

RBAC is built from three concepts. A **permission** is a specific action on a resource type — `invoice:read`, `user:delete`. A **role** bundles permissions into a meaningful job function — `billing-admin` might hold `{invoice:read, invoice:refund, report:read}`. **Users are assigned roles**, and a user's effective permissions are the union of their roles' permissions. The indirection is the whole point: you manage a handful of roles instead of a permission list per user, and onboarding someone is "give them the `support-agent` role" rather than wiring up twenty individual grants.

**Role hierarchies** reduce duplication further: `admin` *inherits* `editor`, which inherits `viewer`, so a permission added to `viewer` automatically flows up. **Separation of duties (SoD)** is a governance constraint expressed in RBAC — no single role (or user) should be able to both *create* a payment and *approve* it, so fraud requires collusion. RBAC's strengths are exactly what auditors want: permissions are **explicit, enumerable, and stable**, so "who can refund an invoice?" is a lookup, not an investigation.

Its defining weakness is **role explosion**. The moment permissions need to depend on *which* object or *what* context — "an editor of documents **in the EU region**", "a manager, but only for **their own** team, during **business hours**" — RBAC has no natural home for the condition, so teams mint ever-more-specific roles (`editor-eu`, `editor-us-finance-readonly`) until there are more roles than users. That's the signal to add **ABAC** conditions or model the relationships with **ReBAC**, rather than creating role number 501.

<h2 id="abac">Attribute-Based Access Control (ABAC)</h2>

ABAC replaces static role-to-permission mappings with **policy rules evaluated over attributes** at decision time. The inputs are attributes of the **subject** (department, clearance, employment status), the **resource** (owner, classification, tags, region), the **action**, and the **environment** (time, source IP, device posture, whether MFA was satisfied). A policy is a rule over those attributes, for example:

> *Allow `read` on a document if `subject.department == resource.department` **and** `resource.classification != "restricted"` **and** `environment.time` is within business hours.*

Because the decision is **computed** rather than looked up, one policy covers cases that would need a role each under RBAC — it kills role explosion for contextual rules and enables **fine-grained, dynamic** access ("only the owner", "only from a managed device"). The cost is real: policies and their attribute sources are **harder to author, test, and reason about**, and the decision point must have the **attributes available** when it evaluates — which is why the architecture needs a place to fetch them (the PIP, next section). Policy languages like **Rego** (Open Policy Agent) and **Cedar** (AWS) exist precisely to make these rules declarative, testable, and reviewable.

In practice ABAC and RBAC are **complementary**: a role is just one attribute of the subject. A common, powerful pattern is RBAC for the coarse "what job function" and ABAC conditions for the "but only when/where/whose" — the baseline role gates the door and the attribute policy narrows it.

<h2 id="policy-architecture">Policy Architecture: PEP, PDP & Engines</h2>

The biggest operational mistake in authorization is **scattering decisions through the code** — an `if (user.role === "admin")` here, a hand-rolled ownership check there. It's unauditable, inconsistent, and impossible to change safely. The fix is to **separate deciding from enforcing**, a structure formalized by four roles:

- **PEP — Policy Enforcement Point.** The chokepoint that *intercepts* the request and *enforces* the verdict (an API gateway, a service middleware, a sidecar). It asks the question and obeys the answer; it contains no policy logic.
- **PDP — Policy Decision Point.** The *engine* that evaluates the policy against the request and returns `allow | deny`. This is where the rules actually run.
- **PAP — Policy Administration Point.** Where policies are *authored, versioned, and managed* — the source of truth the PDP loads.
- **PIP — Policy Information Point.** The *attribute source* the PDP consults for data it doesn't have in hand (the subject's department, the resource's owner).

<AuthorizationDecisionSequence />

**Externalized authorization** puts the PDP in a dedicated **policy engine** that evaluates declarative policy — **OPA (Open Policy Agent)** with its **Rego** language, or **AWS Cedar** — rather than bespoke code. The engine is deployed one of two ways. As a **local sidecar** (a co-located process next to each service): decisions are a **local, sub-millisecond call** with no network hop, which is the common high-performance choice; the policies and data are distributed to every sidecar. Or as a **central service**: one authoritative decision point and one place to see every policy, at the cost of a **network round trip per check** and a hot dependency in the request path. The trade-off — local speed versus central consistency — sets up the capacity story next.

<h2 id="capacity-estimates">Capacity: The Cost of Permission Checks</h2>

Authorization runs on **every request**, and usually **more than once** per request — a coarse check at the gateway, a business-rule check in the service, and an object-level check at the data layer. So a busy API makes **several times more authz checks than it serves requests**, and if each check evaluates policy from scratch (or worse, makes a network hop to a central PDP), the decision tier can rival the application tier in cost. The dominant lever is a **decision cache**: most checks repeat the same `(subject, action, resource)` within a short window, so caching the verdict turns repeated evaluations into cheap lookups.

<AuthorizationCapacity
  assumptions={{
    requestsPerSec: 200000,
    checksPerRequest: 3,
    decisionCpuMs: 0.5,
    cacheHitRate: 0.9,
    msPerCorePerSec: 1000,
  }}
/>

At 200k requests/sec with 3 checks each, that's **600k checks/sec**. Evaluating every one costs ~**300 cores**; a **90%-hit decision cache** drops it to 60k real evaluations and ~**30 cores** — a **10× reduction**. Two lessons follow. First, authz capacity is sized by **checks per second**, not requests per second, so cutting redundant checks and caching decisions matters as much as the raw traffic. Second, the cache buys speed at the price of **staleness**: a permission revoked *now* keeps passing until its cached verdict expires, so the TTL is kept **short** (seconds), and **sensitive or destructive actions bypass the cache** and re-evaluate live. This is the same amortization shape as TLS session resumption — the expensive computation is done once and reused — with the same caveat that reuse means a window of stale state.

<h2 id="relationship-based">ReBAC & Google Zanzibar</h2>

Some products don't have roles so much as **relationships**: *the owner of a folder can share it; members of that share can view every document inside; a comment inherits the visibility of its parent.* Expressing this in RBAC or ABAC is painful because permission flows through **chains of ownership, membership, and containment** that change constantly. **Relationship-Based Access Control** models it directly as a **graph of relationship tuples** of the form `(object, relation, subject)`:

```
(doc:123,   viewer, user:alice)
(doc:123,   viewer, group:eng#member)   // everyone who is a member of group:eng
(folder:9,  editor, user:bob)
(doc:123,   parent, folder:9)           // doc:123 inherits folder:9's grants
```

A permission question becomes a **`check(user, relation, object)`** call that **walks the graph**: is Alice a `viewer` of `doc:123`? Directly yes — or transitively, because she's a member of a group that is a viewer, or because she can edit the parent folder. **Google Zanzibar** is the system that popularized this at planet scale (it powers Google Drive, YouTube, and Calendar permissions); its design centralizes the tuples in one strongly-consistent store and returns a boolean from a simple check API. Because a UI must not show a document the user just lost access to (or hide one they were just granted), Zanzibar issues **zookies** — consistency tokens that let a read request demand "evaluate against a snapshot at least as fresh as this change" — giving **read-your-writes** consistency without making every check globally linearizable. Open-source implementations — **SpiceDB** and **OpenFGA** — bring the model to everyone.

<RelationshipCheckSequence />

The trade-off is a **central, consistent, and very hot** permission store: every page view can fan out into many `check` calls, so ReBAC systems invest heavily in caching, denormalized "reverse indexes" (to answer "list every doc Alice can view"), and the zookie machinery. It's the right tool for **shared, hierarchical, consumer-scale** resources — and overkill for a back-office app that RBAC handles fine.

<h2 id="enforcement-in-practice">Enforcement in Practice</h2>

Authorization must be enforced **in depth**, not at a single gate, because each layer sees different information:

- **At the API gateway** — coarse checks: is the caller authenticated, is the **OAuth scope** sufficient for this API, is the tenant active? This stops obviously-unauthorized traffic early but knows nothing about specific objects.
- **In the service** — business rules: is this user's role and context allowed to perform this operation? This is where RBAC/ABAC policy typically runs.
- **At the data layer** — the **object-level** check that's easiest to forget and most dangerous to skip: does *this* subject actually own or have a relationship to *this specific* record? A gateway that confirms you're a logged-in user does **not** confirm that order `#124` is yours.

Two practices deserve emphasis. **OAuth scopes are coarse authorization, not fine-grained access control**: a scope like `orders:read` says an *application* may read orders on your behalf — it does **not** say which orders, so per-object authz still has to run behind it. And in **multi-tenant** systems, every data access must be **tenant-scoped**: a `WHERE tenant_id = ?` filter is an authorization control, and forgetting it on one query leaks one tenant's data to another. The unifying rule is **deny by default** — an unmatched request is denied, never allowed — so a missing rule fails **closed**, not open.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

The failure modes of authorization are well-known and, unlike subtle crypto bugs, mostly about discipline:

<Callout variant="warning" title="Authorization footguns">
- **Broken Object-Level Authorization (BOLA / IDOR).** The **#1 API vulnerability**: the code checks that you're authenticated but not that the object is *yours*, so changing `GET /orders/123` to `/orders/124` returns someone else's data. **Always enforce object-level ownership at the data layer** — never trust a client-supplied ID.
- **Failing open.** A bug or timeout in the authz path that returns "allow" is catastrophic. **Deny by default**: unmatched or errored decisions must fail **closed**.
- **Over-broad grants.** Wildcard roles, `admin` handed out for convenience, service accounts with far more than they use. Apply **least privilege** — grant the minimum, and prefer **time-bound elevation** (just-in-time access) over standing admin.
- **Role explosion.** Reaching for the 500th narrow role instead of adopting **ABAC** conditions or **ReBAC** relationships. The sprawl itself is the bug.
- **Confused deputy.** A privileged service performing an action on **attacker-controlled input** without re-checking authorization in the *caller's* context — the service's own high privilege gets abused. Pass and check the **original subject's** authority, don't act on the service's.
- **Cache staleness vs revocation.** Decision caches make revocation lag. Keep TTLs **short**, and **bypass the cache** for sensitive/destructive actions so they re-evaluate live.
</Callout>
