<h2 id="overview">What TLS & HTTPS Are</h2>

Every time you see the padlock in a browser, **TLS** is doing the work. TLS — Transport Layer Security, the modern successor to SSL — is the protocol that turns an ordinary, eavesdroppable network connection into a secure one. **HTTPS is not a separate protocol; it is simply HTTP running over TLS.** The application speaks the same HTTP it always did; TLS sits underneath it, wrapping every byte. Swap the transport and `http://` becomes `https://` with no change to the application's logic.

TLS provides three guarantees, and it's worth naming them precisely because they're often blurred together:

- **Confidentiality** — the connection is **encrypted**, so anyone watching the wire (a malicious Wi-Fi hotspot, an ISP, an attacker on the path) sees only ciphertext, not your passwords, cookies, or data.
- **Integrity** — the connection is **tamper-evident**, so a man-in-the-middle can't silently alter a message — injecting a script, flipping a transaction amount — without the change being detected and the connection torn down.
- **Authentication** — you're talking to the **real server** you intended, not an impostor. This is what **certificates** prove, and it's the guarantee people forget: encryption to the *wrong* server is worthless, so verifying identity is as important as hiding the data.

<Callout variant="info" title="HTTPS = HTTP + TLS">
HTTPS is HTTP carried inside a TLS-secured connection. TLS delivers three things at once: **confidentiality** (encryption — eavesdroppers see ciphertext), **integrity** (tamper-evidence — messages can't be altered undetected), and **authentication** (the certificate proves you reached the genuine server, not a man-in-the-middle). Note the scope: TLS authenticates the *server and the connection*, not the *user* — proving who the human is at login is a separate concern (see [Authentication](/topics/security/authentication)).
</Callout>

The rest of this topic unpacks how TLS delivers all three: the crypto it's built on, the handshake that bootstraps a secure channel over an insecure one, the certificates and chain of trust behind authentication, and how all of it is operated in practice.

<h2 id="symmetric-asymmetric">Symmetric vs Asymmetric Crypto</h2>

TLS rests on two fundamentally different kinds of cryptography, and understanding why it needs *both* is the key that unlocks the handshake.

**Symmetric encryption** uses a **single shared key** to both encrypt and decrypt (AES is the workhorse). It is **fast** — modern CPUs encrypt gigabytes per second — which makes it ideal for the actual bulk data of a connection. But it has a fatal bootstrapping problem: both parties must already share the secret key, and if you have to *send* the key over the network to share it, an eavesdropper simply grabs it in transit and the whole scheme collapses. How do two strangers agree on a secret key over a channel an attacker is **watching every byte of**, without ever transmitting the key itself?

**Asymmetric (public-key) encryption** solves exactly that. Each party has a **key pair**: a **public key** anyone may know, and a **private key** kept secret. Data encrypted with the public key can only be decrypted with the matching private key (and a private key can *sign* data that anyone verifies with the public key). This breaks the bootstrapping problem — you can publish your public key to the world, and someone can use it to send you something only you can read. The catch is that asymmetric operations are **slow** (orders of magnitude slower than symmetric), far too expensive to encrypt every byte of a real connection. In practice TLS uses a key-exchange algorithm (**Diffie-Hellman**) built on the same public-key math to let both sides *derive* a shared secret without ever sending it.

So TLS is a **hybrid**, and this is the whole trick: use the **slow asymmetric** crypto **once**, during the handshake, to securely establish a shared symmetric key; then use the **fast symmetric** crypto for **all the actual data**. You pay the expensive asymmetric cost a single time per connection to bootstrap a cheap, fast, encrypted channel. The [capacity section](#capacity-estimates) shows just how lopsided that cost is.

<h2 id="tls-handshake">The TLS Handshake</h2>

Before a single byte of HTTP flows, TLS runs a **handshake** to set up the secure channel. Three things happen in it. First, **negotiation**: the client offers the TLS versions and **cipher suites** it supports, and the server picks one — so both agree on exactly which algorithms to use. Second, **authentication**: the server presents its **certificate**, which (as the [next section](#certificates-pki) details) proves the server really owns the domain you're connecting to. Third, and most cleverly, **key exchange**: client and server run an **ephemeral Diffie-Hellman** exchange — swapping public values from which each side *independently derives the same shared secret*, while an eavesdropper who saw every exchanged value still cannot compute it. That shared secret becomes the symmetric session key.

The word **ephemeral** matters: a fresh key pair is generated for each session and thrown away afterward. This gives **forward secrecy** — even if the server's long-term private key is stolen *later*, an attacker who recorded today's traffic still can't decrypt it, because the session key was never derivable from the long-term key alone. Forward secrecy is now a baseline expectation.

Versions matter for performance. **TLS 1.3** (the modern default) streamlined the handshake to a **single round trip** (1-RTT), and supports **0-RTT** resumption where a returning client can send data immediately. **TLS 1.2** needed two round trips and is slower; older SSL/TLS versions are insecure and disabled. The handshake is the costly part of TLS — once it completes, the connection is just fast symmetric encryption.

<h2 id="capacity-estimates">Capacity: The Cost of TLS</h2>

People worry that "encryption is expensive." The truth is more specific: **the handshake is expensive; the data encryption is cheap.** Symmetric encryption of the actual bytes runs at line rate on modern hardware. What costs real CPU is the **asymmetric** work in the handshake — and the dominant lever on TLS capacity is therefore **how often you have to do a full handshake** versus resuming an existing session. Let's size a busy edge.

The numbers make the point. At **50,000 new connections/sec**, if *every* connection did a full asymmetric handshake (~2 ms of CPU each), you'd burn **100,000 ms of CPU per second — 100 cores** just on handshakes. But most connections aren't truly new: with **session resumption** (TLS 1.3 resumption / 0-RTT, session tickets) and HTTP **keep-alive** reusing one connection for many requests, ~80% of handshakes are *resumed* at ~20× lower cost (~0.1 ms). That collapses the bill to ~**24,000 ms/sec — 24 cores**, roughly a **4× reduction**. The takeaway threefold: TLS-termination capacity is sized by **handshakes per second, not bytes transferred**; **session resumption and keep-alive are the dominant optimization** (don't re-handshake when you can resume); and because handshake CPU is a concentrated, optimizable cost, it pays to **terminate TLS at a shared, tuned edge** (a load balancer or CDN with hardware acceleration) rather than paying it redundantly on every backend. The [termination section](#tls-termination) builds on exactly that.

<h2 id="certificates-pki">Certificates & the Chain of Trust</h2>

The handshake's authentication step rests on the **certificate** — and certificates are where most people's mental model gets fuzzy. A **certificate** (an **X.509** document) **binds a domain name to a public key**, and is **digitally signed by a Certificate Authority (CA)** vouching that the binding is genuine. When the server presents its certificate, it proves it holds the matching private key, so you know the public key you're about to use really belongs to `example.com` and not an impostor.

But that just moves the question: why trust the CA's signature? The answer is a **chain of trust**. Your server's certificate (the **leaf**) is signed by an **intermediate CA**, whose certificate is in turn signed — directly or through more intermediates — by a **root CA**. The roots are the anchor: a curated set of **root certificates ships pre-installed** in your operating system and browser, in what's called the **trust store**. These are the parties you trust *a priori*, without being told to. To validate, the client walks the chain from the leaf upward, verifying each certificate's signature against the next one up, until it reaches a root that is **already in its trust store**. If the chain terminates at a trusted anchor — and the certificate's **domain matches** the site and **nothing has expired** — the server's identity is accepted. This whole system of CAs, certificates, and chained trust is **Public Key Infrastructure (PKI)**.

The chain exists so CAs don't sign leaf certificates with their precious, offline root keys directly; the root signs a small number of intermediates, and intermediates do the day-to-day signing. It also means trust is only as strong as the CA system: a compromised or careless CA can issue a certificate for a domain it shouldn't, which is why mechanisms like Certificate Transparency logs and pinning exist as backstops.

<h2 id="certificate-lifecycle">Issuance, Expiry & Revocation</h2>

Certificates have a life cycle, and operating it correctly is where real systems succeed or fail. **Issuance** starts with proving you control the domain. **Domain Validation (DV)** — the common case — just verifies domain control (e.g. you can place a file or DNS record the CA checks) and is fully **automated**: **ACME**, the protocol behind **Let's Encrypt**, issues and renews DV certificates with no human in the loop, which is why HTTPS is now free and ubiquitous. **Organization (OV)** and **Extended Validation (EV)** certificates add vetting of the legal entity, at more cost and manual effort, for cases where that matters.

Certificates **expire** — deliberately, and lifetimes have been getting **shorter** (often 90 days or less). Short lifetimes limit the damage window of a mis-issued or compromised certificate, but they make **automated renewal non-negotiable**: the single most common TLS outage is a **certificate that quietly expired** because nobody automated its renewal, taking a site down for everyone at once. Automate it (ACME) and monitor expiry dates.

Sometimes a certificate must be **revoked before it expires** — say its private key leaked. Three mechanisms address this, each fixing the last one's flaws:

- **CRL (Certificate Revocation List)** — the CA publishes a list of revoked certificates. Simple, but the lists grow large and are fetched infrequently, so revocation is slow to take effect.
- **OCSP (Online Certificate Status Protocol)** — the client asks the CA, in real time, "is this specific certificate still valid?" Fresher, but it adds latency to the handshake and leaks the user's browsing to the CA, and clients often "fail open" if the CA is unreachable.
- **OCSP stapling** — the **server** periodically fetches a fresh, CA-signed status statement and **staples** it to its certificate during the handshake. The client gets up-to-date revocation info with no extra round trip and no privacy leak — the best of both, and the modern recommendation.

<h2 id="https-in-practice">HTTPS in Practice</h2>

Beyond the cryptography, a handful of practical mechanisms make HTTPS work at web scale:

- **SNI (Server Name Indication).** A single IP address often hosts many HTTPS sites, but the server must present the *right* certificate before it knows which site you want — a chicken-and-egg. SNI fixes it: the client includes the **hostname it wants in the ClientHello**, so the server can select the matching certificate. It's what makes virtual hosting and shared CDN edges possible. (The hostname is visible in classic TLS; Encrypted Client Hello is the evolving fix.)
- **ALPN (Application-Layer Protocol Negotiation).** During the handshake, client and server **negotiate which application protocol** to speak — HTTP/2, HTTP/1.1, etc. — so upgrading to HTTP/2 costs no extra round trip.
- **HSTS (HTTP Strict Transport Security).** A response header telling the browser "**always** use HTTPS for this site, for the next N months." It defends against **downgrade and SSL-stripping** attacks, where an attacker tries to force the victim back onto plaintext HTTP. With HSTS the browser refuses to even try HTTP.
- **The http→https redirect.** Sites still listen on port 80 only to immediately **redirect to HTTPS** (and then HSTS keeps the browser on HTTPS thereafter).
- **Mixed content.** An HTTPS page that pulls a script or stylesheet over plain `http://` reopens the very hole HTTPS closed — that asset can be tampered with. Browsers **block active mixed content**; everything a secure page loads should itself be secure.

<h2 id="tls-termination">TLS Termination & mTLS</h2>

**Where** does TLS actually end? Rarely on the application server itself. In most architectures TLS is **terminated at the edge** — a **load balancer, reverse proxy, or CDN** decrypts the incoming TLS connection and forwards the request to the backends. This **centralizes certificate management** (one place to install and rotate certs) and **concentrates the handshake CPU** (the expensive part from the [capacity section](#capacity-estimates)) on a tier built and tuned for it, often with hardware acceleration. There are three patterns:

- **Termination** — TLS ends at the edge; traffic to backends is plaintext on a trusted internal network. Simplest and fastest.
- **Passthrough** — the edge forwards the encrypted bytes untouched and the backend terminates TLS itself, for cases where even the edge shouldn't see plaintext.
- **Re-encryption** — the edge terminates the client's TLS, then opens a *new* TLS connection to the backend, so traffic is encrypted end to end while the edge still inspects/routes it.

So far TLS authenticated only the **server** to the client. **Mutual TLS (mTLS)** makes it bidirectional: **both** sides present certificates, so the server also verifies the **client's** identity cryptographically. This is the backbone of **service-to-service** authentication in **zero-trust** architectures and **service meshes** — every service proves who it is with a certificate, so the network itself is never implicitly trusted. mTLS is overkill for public web traffic (you can't issue every user a certificate) but ideal *inside* a system, where you control both ends.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

TLS is mature, but the operational failure modes are common and avoidable:

<Callout variant="warning" title="TLS footguns">
- **Expired certificates.** The number-one TLS outage: a cert quietly expires and the whole site goes down at once. **Automate renewal** (ACME/Let's Encrypt) and **monitor expiry dates** with alerts well ahead of time.
- **Weak protocol versions and ciphers.** Disable SSLv3, TLS 1.0, and TLS 1.1, and drop weak cipher suites. **Prefer TLS 1.3** and require **forward secrecy** (ephemeral key exchange).
- **Self-signed or untrusted chains.** A certificate that doesn't chain to a trusted root (or is missing its intermediates) breaks validation; ship the full chain and use a real CA in production.
- **Not validating the hostname.** A valid certificate for the *wrong* domain is an attack; always check the cert covers the host you connected to (a classic bug in hand-rolled clients).
- **Downgrade / SSL-stripping attacks.** Force HTTPS with **HSTS**, redirect http→https, and avoid **mixed content**.
- **Protect the private key.** It's the crown jewel — store it securely (an HSM or secrets manager), restrict access, and rotate if exposure is suspected. Short cert lifetimes and forward secrecy limit the blast radius if it leaks.
</Callout>
