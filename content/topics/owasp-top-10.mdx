<h2 id="overview">What the OWASP Top 10 Is</h2>

The **OWASP Top 10** is the single most recognized document in application security: a ranked list of the ten most critical risks to web applications, published by the **Open Worldwide Application Security Project**. It exists because most breaches aren't exotic — they exploit the same handful of well-understood weaknesses over and over, and a shared, prioritized list of those weaknesses lets teams focus on what actually gets exploited in the wild. It's refreshed every few years as the data shifts; the **2021** edition is current.

Two things make it more than a listicle. First, it's **data-backed**: the ranking is built from analyzing hundreds of thousands of real applications and CVEs, tempered by a practitioner survey for emerging risks the raw data hasn't caught up to yet. Second, it lists **categories of risk**, not individual bugs — "Injection" or "Broken Access Control," not "the SQL bug in the login form." That's what makes it a durable framework rather than a changelog.

The critical thing to get right is **how to use it**. The Top 10 is an **awareness baseline** — a floor, a shared vocabulary, a "are we at least covered against the obvious?" It is emphatically *not* a complete checklist, a certification, or a substitute for **threat modeling** your specific system. Teams that treat "we passed the Top 10" as "we are secure" miss everything specific to their design. It's where you *start* thinking about security, not where you stop.

<Callout variant="info" title="A floor, not a ceiling">
The OWASP Top 10 is a **baseline for awareness**, not a compliance checklist and not a threat model. It tells you the risks that hit *everyone*, so you can make sure you're not obviously exposed — a floor to clear before you reason about the threats specific to your own system. "We covered the Top 10" is the beginning of a security program, never the end of one.
</Callout>

<h2 id="broken-access-control">A01: Broken Access Control</h2>

At number one is **Broken Access Control**: the application fails to enforce what an authenticated user is actually allowed to do. Authentication proved *who* the user is; access control decides *what they may do* — and when that enforcement is missing, weak, or client-side, users act outside their permissions. The most common concrete form is **IDOR / BOLA** (Insecure Direct Object Reference / Broken Object-Level Authorization): the app checks that you're logged in but not that the object is *yours*, so changing `GET /orders/123` to `/orders/124` returns someone else's order.

The failure has several faces: **missing function-level authorization** (a normal user calling an admin-only endpoint that was merely hidden from the UI), **forced browsing** to unprotected pages, privilege escalation by tampering with a role in a token or cookie, and metadata manipulation. The through-line is trusting the client — the URL, the hidden field, the JWT claim — instead of enforcing on the server.

The defense is exactly the [Authorization](/topics/security/authorization) topic: enforce access control **server-side, on every request, at the object level**, and **deny by default** so anything not explicitly allowed is refused. There's no client-side shortcut — hiding a button is not a control.

<h2 id="cryptographic-failures">A02: Cryptographic Failures</h2>

Number two, **Cryptographic Failures**, covers everything that leaves sensitive data unprotected because cryptography was absent, weak, or misused. The 2021 rename — from "Sensitive Data Exposure" to "Cryptographic Failures" — deliberately shifts attention from the *symptom* (exposed data) to the *cause* (the crypto that should have protected it and didn't). It's a catch-all that ties directly to three other topics in this series.

The failures cluster into a few kinds: **no encryption at all** (sensitive data in plaintext, at rest or in transit), **weak or fast password hashing** (storing passwords with MD5 or unsalted SHA-256 instead of a slow, salted function), **broken transport security** (old TLS versions, invalid certificates, downgrade-able connections), **weak algorithms or modes** (MD5, DES, ECB), and **key mismanagement** (hardcoded keys, keys committed to source control, no rotation).

Each maps cleanly to a dedicated topic: protecting data confidentiality and managing keys is [Encryption & Key Management](/topics/security/encryption-key-management); storing credentials safely is [Password Hashing](/topics/security/password-hashing); securing data in transit is [TLS/HTTPS](/topics/security/tls-https-certificates). If those three are done right, A02 is largely handled — which is why "Cryptographic Failures" is best read as a pointer to getting the fundamentals of each right.

<h2 id="injection">A03: Injection</h2>

**Injection** — long the reigning #1, now third — happens when an application takes untrusted input and lets an interpreter treat it as **code** instead of **data**. The classic is **SQL injection**: user input is concatenated into a query string, so a value like `' OR 1=1 --` changes the *meaning* of the SQL and the database runs the attacker's logic. The same root cause produces **cross-site scripting (XSS)** (attacker input becomes `<script>` executed in a victim's browser), **OS command injection**, and LDAP or NoSQL injection.

<InjectionAttackSequence />

The single root cause is **mixing code and data**, and so is the single fix: **keep them separate**. For databases, use **parameterized queries / prepared statements**, which send the query structure and the values on different channels so input can never be parsed as command syntax. For XSS, use **context-aware output encoding** (encode data for the HTML, attribute, or JavaScript context it lands in) and a Content Security Policy. Input validation (allowlists) is valuable defense in depth, but it is *not* a substitute for parameterization — the reliable fix is architectural: the interpreter must never see attacker input as code.

<h2 id="capacity-estimates">Capacity: Defense in Depth by the Numbers</h2>

A recurring truth across all ten risks is that **no single control is perfect** — every defense has bypasses, gaps, and bad days. What makes systems actually secure is **layering independent controls** so that an attack has to defeat *all* of them, and because the layers are independent, their effects **multiply**. A little arithmetic makes the payoff vivid.

  assumptions={{
    attackAttemptsPerDay: 1000000,
    blockRatePerLayer: 0.9,
    layers: 3,

/>

Suppose a million automated attacks hit your app each day and each control you deploy blocks 90% of them. One control sounds strong — until you notice it still lets **100,000** attacks through. Add a second independent layer and 1% get through; a third and only 0.1% do — **1,000** per day. That's a **100× improvement** over a single control and a **1,000× reduction** versus none, purely from stacking three imperfect defenses. This is why the Top 10's fixes are meant to **stack**, not compete: for injection you validate input *and* parameterize queries *and* run the database with least privilege *and* monitor for anomalies — each layer catches what the previous one missed.

Two lessons follow. First, **defense in depth is multiplicative**, so there's enormous leverage in adding independent layers rather than perfecting one. Second — and this is why **A09 Security Logging & Monitoring** is itself a Top 10 item — the residual never reaches zero. Even excellent layering leaves ~1,000 attacks a day getting through, so you *must* be able to see and respond to what slips past. Layering shrinks the problem; monitoring catches the remainder.

<h2 id="design-config-components">A04–A06: Design, Configuration & Components</h2>

The middle of the list covers three "whole-system" risks that live above any single line of code.

- **A04 Insecure Design** is a flaw in the *design itself* — a control that should have existed but was never conceived, as opposed to a control that was implemented buggily. No amount of careful coding fixes a design that, say, never planned for account-recovery abuse. The remedy is proactive: **threat modeling**, secure design patterns, and shifting security **left** into the design phase where missing controls are cheap to add.
- **A05 Security Misconfiguration** is the gap between "software installed" and "software hardened": default credentials left in place, verbose error messages leaking stack traces, an open S3 bucket, unnecessary features and ports enabled, missing security headers. The fix is **secure defaults**, systematic hardening, and least functionality — turn off what you don't need and don't ship development settings to production.
- **A06 Vulnerable and Outdated Components** is running dependencies with *known* vulnerabilities — the **Log4Shell** class, where a flaw in a ubiquitous library exposes everyone who hasn't patched. Modern apps are mostly third-party code, so the fix is **software composition analysis (SCA) / dependency scanning** to inventory and flag vulnerable packages, plus a process to patch promptly.

<Callout variant="warning" title="An insecure design is not an insecure implementation">
A01–A03 are often *implementation* bugs — the control exists but was coded wrong. **A04 Insecure Design** is different and can't be patched away: the control was never there. You can implement a flawed design flawlessly and still be insecure. That's why security has to enter at design time (threat modeling), not just code review — you can't test or patch your way to a control that was never designed.
</Callout>

<h2 id="remaining-risks">A07–A10: Auth, Integrity, Logging & SSRF</h2>

The final four round out the list, and each connects to a broader theme.

- **A07 Identification and Authentication Failures** — weaknesses in proving identity: permitting weak or breached passwords, no **MFA**, exposed session IDs, broken logout or session fixation. This is the [Authentication](/topics/security/authentication) topic's domain — verify identity robustly and manage sessions and tokens correctly.
- **A08 Software and Data Integrity Failures** — trusting code or data whose integrity you never verified: **insecure deserialization** of attacker-controlled objects, auto-updates pulled without signature checks, and compromised **CI/CD** pipelines — the **supply-chain** class exemplified by **SolarWinds**. The defense is verifying integrity with digital signatures and trusted sources before you execute or ingest anything.
- **A09 Security Logging and Monitoring Failures** — you cannot detect, respond to, or investigate what you never recorded. Inadequate logging lets breaches **dwell undetected for months**; it's the reason the residual attacks from the capacity model matter. Log security-relevant events, monitor them, and alert.
- **A10 Server-Side Request Forgery (SSRF)** — an attacker makes the **server** issue requests on their behalf, classically to reach an internal-only address the attacker couldn't touch directly, such as the **cloud metadata endpoint** that hands out instance credentials.

<SsrfAttackSequence />

SSRF's rise tracks the cloud: a server that fetches a user-supplied URL becomes a proxy into the internal network. The defense is to never fetch arbitrary URLs — use an **allowlist** of permitted destinations and block requests to internal and link-local ranges.

<h2 id="pitfalls-best-practices">Cross-Cutting Defenses & Best Practices</h2>

Step back from the ten and the same handful of mistakes — and the same handful of defenses — recur across all of them.

<Callout variant="warning" title="Recurring mistakes behind the Top 10">
- **Trusting client input.** URLs, hidden fields, headers, and tokens are all attacker-controllable. Validate and authorize on the **server**, always.
- **Blocklists instead of allowlists.** Trying to enumerate "bad" input is a losing game; permit a known-good set and reject the rest.
- **Security by obscurity.** A hidden endpoint or an unlinked admin page is not a control. Assume the attacker knows your design.
- **"We'll add security later."** Bolted-on security misses design-level flaws (A04). Build it in from the start.
- **No logging or monitoring.** If you can't see attacks, you can't stop the ones that get through (A09).
</Callout>

Against those, a small set of **cross-cutting defenses** knocks out whole classes of the Top 10 at once:

- **Validate input and encode output** — the antidote to injection (A03) and much of XSS.
- **Least privilege and deny-by-default** — contains broken access control (A01) and limits blast radius everywhere.
- **Defense in depth** — layer independent controls so no single failure is fatal (the capacity model's lesson).
- **Secure defaults and hardening** — closes misconfiguration (A05).
- **Keep components patched** — closes vulnerable-component risk (A06); automate dependency scanning.
- **Log, monitor, and alert** — the detection layer for everything that slips through (A09).
- **Shift security left** — threat model and design securely (A04), so controls exist before code does.
