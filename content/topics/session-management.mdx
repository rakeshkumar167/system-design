<h2 id="overview">What Session Management Is</h2>

**HTTP is stateless**: every request arrives as if the server has never seen the caller before. That's fine for fetching a public page, but the moment a user logs in you need the server to remember, on the *next* request and every one after, that this is the same authenticated person — otherwise it would demand a password on every click. **Session management** is how that memory works. After [authentication](/topics/security/authentication) succeeds *once*, the server establishes a **session** and hands the client a **session id**; the client presents that id on subsequent requests, and the server uses it to recognize the returning user.

The single most important thing to internalize is what that session id *is*: a **bearer credential**. Whoever presents it is treated as the user, no questions asked, for as long as the session lives. It is not a name or a lookup key you can hand out casually — it is, functionally, a **temporary password**. Steal it and you *are* the victim until it expires. Almost everything in this topic follows from taking that seriously: the id must be unguessable, transmitted only over encrypted channels, unreadable to scripts, changed at login, and expired promptly.

<Callout variant="info" title="The session id is a temporary password">
A session id is a **bearer token**: possession is authentication. Anyone who obtains it — by guessing it, sniffing it off the wire, reading it from JavaScript, or having you use one they planted — becomes you for the rest of the session. Protect it with the same seriousness you'd protect the password itself: high entropy, TLS-only, HttpOnly, regenerated at login, and short-lived.
</Callout>

<h2 id="session-ids-cookies">Session IDs & Cookies</h2>

Because the id is a credential, its **unpredictability** is non-negotiable. It must be generated from a **cryptographically secure random number generator (CSPRNG)** with enough entropy — at least **128 bits** — that an attacker can neither guess a valid one nor brute-force the space. A short or sequential id (`session=1002`) is a catastrophe: an attacker just tries adjacent values. This is the framework's job, not yours; the failure mode is rolling your own with a weak source.

The id is almost always carried in a **cookie**, and the cookie's **security attributes** are a remarkably cheap, powerful line of defense — set them correctly and whole attack classes disappear:

- **HttpOnly** — the cookie is invisible to JavaScript (`document.cookie` can't read it). This blunts **session theft via XSS**: even if an attacker injects a script, it can't exfiltrate the session id.
- **Secure** — the cookie is sent only over **HTTPS**, never plaintext HTTP, so it can't be sniffed off the wire (pairs with the [TLS](/topics/security/tls-https-certificates) topic).
- **SameSite** (`Lax` or `Strict`) — the browser won't attach the cookie to **cross-site** requests, which blunts **CSRF** (below).
- **Domain / Path** scope and a sensible **Max-Age / Expires** — limit where the cookie is sent and how long it lives.

<h2 id="server-vs-client-side">Server-Side vs Client-Side Sessions</h2>

There are two fundamentally different places to keep the session's *state*, and the choice drives everything about scale and revocation.

- **Server-side (stateful) sessions.** The cookie holds only an **opaque random id**; the actual session data (user id, roles, cart) lives in a **session store** on the server — memory, a database, or (typically) **Redis**. The server can **revoke instantly** by deleting the store entry, and the client never sees the data. The cost: the server must **look the id up on every request**, and it must hold every active session in memory.
- **Client-side (stateless) sessions.** The state *itself* is packed into a **signed (and often encrypted) cookie** or a **JWT** that the client holds and presents. The server just **verifies the signature** — **no lookup, no session store** — which makes horizontal scaling trivial (any server can validate any request). The costs are real, though: you **can't easily revoke** a token before it expires (there's nothing central to delete), the payload is **size-limited** and **visible** to the client (if only signed, not encrypted), and any change to the data waits for re-issue.

<h2 id="lifecycle-timeouts">Session Lifecycle & Timeouts</h2>

A session has a life: it's **created at login** (a fresh id issued), used across many requests, and must eventually **end**. Ending it well is where two distinct timeouts matter, and confusing them is a common gap:

- **Idle (inactivity) timeout** — the session expires after a period of *no activity* (say 15–30 minutes). This limits the window on an **abandoned** session — a user who walks away from a shared computer isn't left logged in indefinitely.
- **Absolute timeout** — a **hard cap** on total session lifetime regardless of activity (say 8–24 hours). This bounds the usefulness of a **stolen** session: even a continuously-used hijacked session is forcibly ended, forcing re-authentication.

You want **both** — the idle timeout catches inactivity, the absolute timeout caps the total blast radius. **Sliding renewal** extends the idle window on each request, keeping active users logged in without lengthening the absolute cap. And **logout must actually invalidate the session server-side**: for a stateful session, delete the store entry so the id is instantly dead; for a stateless token, you can only add it to a **blocklist** or wait for it to expire — which is exactly why stateless logout is weaker.

<h2 id="session-attacks">Session Attacks: Hijacking, Fixation & CSRF</h2>

Because the session id is a credential, the attacks all revolve around **obtaining or abusing** it.

- **Session hijacking** — stealing a valid id and replaying it. The two classic routes are **XSS** (a script reads the cookie — defeated by **HttpOnly**) and a **plaintext channel** (sniffing the id off unencrypted HTTP — defeated by **TLS + the Secure flag**).
- **Session fixation** — a subtler trick where the attacker chooses the id *in advance*. They obtain a valid session id, plant it on the victim (e.g. via a crafted link), and wait for the victim to log in. If the server keeps that same id after authentication, the attacker — who already knows it — is now inside the victim's authenticated session.

<SessionFixationSequence />

  The fix for fixation is simple and mandatory: **regenerate the session id on login** (and on any privilege change), so whatever id existed before authentication is discarded and worthless.

- **CSRF (Cross-Site Request Forgery)** — this one doesn't steal the id at all; it *abuses the browser's automatic cookie behavior*. Because the browser attaches the session cookie to **every** request to your domain, a malicious page can trigger a state-changing request (`POST /transfer`) and the victim's session rides along. The defenses are **SameSite cookies** (the browser won't send the cookie cross-site) and **anti-CSRF tokens** (a secret the attacker's page can't know).

<h2 id="capacity-estimates">Capacity: The Cost of Session Lookups</h2>

The storage choice from earlier has a concrete price. **Stateful sessions require a store lookup on every authenticated request**, so the session store's load scales directly with your total request rate — and you must hold every active session in memory somewhere.

<SessionCapacity
  assumptions={{
    requestsPerSec: 300000,
    activeSessions: 50000000,
    sessionBytes: 1000,
    cacheHitRate: 0.95,
    bytesPerGb: 1000000000,

At 300,000 authenticated requests/sec, a naive stateful design sends **300,000 lookups/sec** straight at the session store, and holding 50 million active sessions at 1 KB each takes **50 GB** of memory. The lookup load is the danger — the session store becomes a hot dependency on every request's critical path. A **session cache** (most requests reuse the same session within seconds) with a 95% hit rate cuts store lookups **20×**, to **15,000/sec**, keeping the store from buckling.

This is the stateful-vs-stateless trade in hard numbers. **Stateful** buys you **instant revocation** and central control, but you pay with a **per-request lookup and gigabytes of memory** — costs you soften with caching and cheap key-value stores. **Stateless** sessions eliminate *both* the lookup and the memory (the state rides in the cookie), which is why they scale so effortlessly — but you give up the ability to revoke a session before it expires. Neither is free; you're choosing which cost to bear.

<h2 id="distributed-sessions">Sessions at Scale</h2>

Once you have more than one application server, a new question appears: when a request lands on server B but the session was created on server A, how does B know about it? There are three answers.

- **Sticky sessions (session affinity).** The load balancer pins each user to one server, which keeps their session in local memory. Simple and lookup-free — but **fragile**: if that server restarts or dies, every session on it is lost and those users are logged out, and it undermines even load distribution and smooth deploys. A stopgap, not a foundation.
- **Shared session store.** All servers read and write sessions in a central store (Redis). Any server can handle any request, revocation is central, and deploys don't drop sessions — at the cost of the network lookup from the capacity section. This is the **standard** approach for stateful sessions.
- **Stateless sessions.** With the state in the cookie, there's **no server-side session state to share at all** — every server validates every request independently. This is the cleanest horizontal scale, and the reason stateless tokens are popular for large distributed systems — accepting the revocation trade-off.

<Callout variant="warning" title="Sticky sessions are a trap at scale">
Sticky sessions feel easy — no shared store, no lookup — but they bind a user's session to a single server's memory. That server dying, restarting, or being drained for a deploy logs those users out, and traffic can't rebalance freely. For anything beyond a small deployment, use a **shared session store** (stateful) or go **stateless**; reserve stickiness for cases that genuinely can't avoid it.
</Callout>

<h2 id="secure-sessions">Secure Sessions in Practice</h2>

Putting it together, a secure session is mostly a checklist applied at two moments — login and every subsequent request:

<SecureLoginSessionSequence />

At **login**: verify credentials, then **generate a fresh, high-entropy session id** — crucially **regenerating** it rather than reusing any pre-login id (this kills fixation) — write the session to the store, and return it in a cookie set **HttpOnly + Secure + SameSite**. On **every subsequent request**: the browser returns the cookie automatically, and the server validates the opaque id against the store to recover the identity. Around this, enforce the lifecycle controls: **idle and absolute timeouts**, **regenerate the id again on any privilege elevation**, **always TLS** so the cookie is never exposed, and **invalidate server-side on logout** so the id is instantly dead. None of these is expensive; skipping any one of them reopens a whole attack class from the previous sections.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

Session bugs are consistent and, once you know them, easy to avoid.

<Callout variant="warning" title="Session management footguns">
- **Predictable or short session ids.** Sequential or low-entropy ids can be guessed or enumerated. Use a CSPRNG with ≥128 bits.
- **Session id in the URL.** Ids in query strings leak via browser history, server logs, and the `Referer` header. Keep them in cookies, never the URL.
- **Not regenerating on login.** Reusing a pre-login id enables session fixation. Always issue a fresh id at authentication and on privilege change.
- **Missing cookie flags.** No HttpOnly (XSS can steal it), no Secure (sniffable), no SameSite (CSRF). Set all three.
- **No idle or absolute timeout.** Sessions that never expire maximize the value of a stolen id. Enforce both.
- **Not invalidating on logout.** "Logout" that only clears the client cookie leaves the session alive server-side. Delete it in the store.
- **Plaintext transport.** A session cookie over HTTP is a credential in the clear. TLS everywhere.
</Callout>
