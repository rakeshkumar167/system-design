<h2 id="overview">What Encryption & Key Management Is</h2>

**Encryption** makes data confidential: it transforms plaintext into ciphertext so that only a holder of the right **key** can read it, and gibberish to anyone who steals the disk, the backup, or the network packets. It's the last line of defense — the assumption that, sooner or later, someone gets to your bytes, and when they do those bytes should be worthless. Where the [Password Hashing](/topics/security/password-hashing) topic was about *one-way* storage (you never recover the input), encryption is deliberately **reversible** by design: the whole point is that the legitimate owner *can* get the data back, with the key.

Which is exactly why this topic is really about **keys**, not ciphers. The algorithms — AES, RSA, ECC — are standardized, vetted, and a single library call away; getting them right is mostly "don't roll your own." The hard, system-design part is everything around the key: **generating** it from good randomness, **storing** it so it's not sitting next to the data it protects, **distributing** it to the services that need it without leaking it, **rotating** it on a schedule, and **destroying** it safely — all while never losing it, because a lost key means permanently unreadable data. That asymmetry of difficulty is the theme of the whole topic.

<Callout variant="info" title="The maxim">
**Encryption is easy; key management is hard.** The cipher is a library call. Protecting, distributing, rotating, and *not losing* the key is the system. Most real-world "encryption" failures aren't broken math — they're a key hardcoded in a repo, a key stored beside the data it protects, or a key nobody can rotate.
</Callout>

<h2 id="symmetric-asymmetric">Symmetric vs Asymmetric Encryption</h2>

There are two families of encryption, and they solve different halves of the problem.

- **Symmetric encryption** uses **one shared secret key** for both encrypting and decrypting. The standard is **AES**, and it's **fast** — modern CPUs have dedicated instructions for it, so it encrypts gigabytes per second. The catch is the *key distribution problem*: everyone who needs to read the data needs the same secret key, and getting that key to them securely is its own challenge.
- **Asymmetric encryption** uses a **keypair** — a **public key** that anyone can use to encrypt, and a mathematically-linked **private key** that only the owner holds and that alone can decrypt. **RSA** and **ECC** are the common choices. This elegantly solves distribution (publish the public key freely), but it's **orders of magnitude slower** and works only on small payloads.

In practice you don't choose one — you combine them with **hybrid encryption**: use *asymmetric* to securely exchange (or wrap) a fresh *symmetric* key, then use the fast symmetric key for the actual data. That's precisely what [TLS](/topics/security/tls-https-certificates) does in its handshake, and — as the envelope-encryption section shows — it's the same shape used to protect data at rest. Symmetric does the heavy lifting; asymmetric solves getting the symmetric key where it needs to be.

<h2 id="data-states">Data at Rest, in Transit, in Use</h2>

"Is the data encrypted?" is an incomplete question — you have to say *which state* and *against which threat*, because data lives in three, each with its own defense.

- **At rest** — data sitting in storage: databases, object stores, disks, backups. Protected by **full-disk or volume encryption** (defends against a stolen drive), **transparent database encryption (TDE)** (the database encrypts files transparently), or **field-level / application-layer encryption** for the most sensitive columns (so even a compromised database process sees ciphertext for those fields). The threat is someone getting the storage medium or a dump.
- **In transit** — data moving across a network. Protected by **TLS** (the [TLS/HTTPS](/topics/security/tls-https-certificates) topic covers this in full). The threat is an eavesdropper or man-in-the-middle on the wire.
- **In use** — data live in a running process's memory, necessarily decrypted to be operated on. The hardest to protect; addressed by **confidential computing** — **secure enclaves / trusted execution environments** that keep data encrypted even in RAM, shielded from the host OS and hypervisor. The threat is a compromised host or a malicious cloud operator.

The interview-level insight: these are independent. Full-disk encryption does nothing against a network eavesdropper; TLS does nothing against a stolen backup; both are useless against a compromised host reading process memory. A serious design names the threat and picks the layer — often all three.

<h2 id="envelope-encryption">Envelope Encryption & the Key Hierarchy</h2>

You might think you'd encrypt each object directly with your master key. You don't — that would mean handing the master key to every service that touches data, and rotating it would require re-encrypting everything. **Envelope encryption** solves both with a layer of indirection:

- A per-object **data encryption key (DEK)** encrypts the actual data — fast symmetric AES.
- A **key encryption key (KEK)**, the master key held in the KMS, **encrypts (wraps) the DEK**.
- You store the **wrapped DEK right next to the ciphertext**, and use-then-discard the plaintext DEK.

<EnvelopeEncryptSequence />

The elegance is in what this buys you. Only the **KEK is precious** — one key to guard instead of millions of DEKs, and it never leaves the KMS. The **wrapped DEK can safely sit beside the data** it protects, because it's worthless without the KEK to open it (so "key next to data" is fine here — for the *wrapped* key, not the master). And you can **rotate the KEK by simply re-wrapping the DEKs** — a tiny operation on small keys — without touching a single byte of the bulk ciphertext. This generalizes into a **key hierarchy**: a root key protects KEKs, KEKs wrap DEKs, DEKs encrypt data — each level guarding the one below, with the root anchored in hardware.

<h2 id="kms-hsm">KMS & HSMs</h2>

If the KEK is the one key that matters, where does it live? In a **Key Management Service (KMS)** — crypto-as-a-service. The defining property is that the KMS **generates, stores, and uses keys so the key material never leaves it**: you don't fetch the KEK and unwrap a DEK yourself; you send the wrapped DEK to the KMS and it returns the plaintext DEK (or performs the encrypt/decrypt on your behalf). The master key is used inside the boundary and never exported, so there's no moment where it's sitting in your application's memory or logs.

Underneath a serious KMS sits an **HSM (Hardware Security Module)** — tamper-resistant hardware, certified to **FIPS 140-2 / 140-3**, that physically guards key material and performs crypto operations. It's the **root of trust**: keys are generated inside it and, by design, cannot be extracted even by an administrator; a physical attack triggers tamper-response that destroys the keys. Cloud providers package all of this as managed services — **AWS KMS, Google Cloud KMS, Azure Key Vault** — and **HashiCorp Vault** offers it self-hosted, typically HSM-backed.

The mental model to carry into an interview: **exactly one component ever touches the master key — the KMS/HSM — and everything else just asks it to wrap and unwrap.** That single chokepoint is what makes the rest of the system's key handling tractable: it's the one place to enforce access control, log every use, and anchor rotation.

<h2 id="capacity-estimates">Capacity: The Cost of Encryption</h2>

The instinct is to worry about the cipher's cost, but **AES is nearly free** — hardware-accelerated to gigabytes per second per core. The real cost, and the real bottleneck, is the **KMS call**. A naive design that asks the KMS to unwrap a data key on *every* encrypt/decrypt would generate one KMS call per operation — a network round trip each, against a service with a hard requests-per-second quota. Envelope encryption is what saves you: because one DEK protects many objects, you **cache the plaintext DEK** in memory for a short window and reuse it, so the KMS is consulted per *key*, not per *operation*.

At 500,000 operations/sec, a KMS-call-per-op design needs **500,000 KMS calls/sec** — wildly beyond any KMS quota, and a 20 ms network hop on the critical path of every single operation. A **99.9%-hit DEK cache** collapses that to **500 KMS calls/sec** — a **1,000× reduction** — while the actual AES work is a mere ~**5 cores** of nearly-free local compute. Two lessons follow. First, encryption throughput is sized by **KMS calls, not cipher speed** — the cipher is a rounding error, the key service is the constraint, and DEK caching is the lever that makes it scale. Second, that cache is the familiar amortization-with-a-catch: a cached plaintext DEK is a live key in memory, so its **blast radius** (a leaked cache exposes every object that DEK protects, until the TTL expires) is exactly why the TTL is short, the KEK never leaves the KMS, and the most sensitive data gets narrower key scopes.

<h2 id="key-rotation-lifecycle">Key Rotation & Lifecycle</h2>

A key is not a set-and-forget value; it has a lifecycle: **generate → activate → rotate → expire → revoke → destroy**. **Rotation** — periodically replacing a key with a fresh one — matters because it bounds two things: the *amount of data* encrypted under any single key (limiting the damage if one key is ever broken or leaked) and the *time window* a compromised key is useful. But rotating naively sounds impossible: re-encrypting petabytes every rotation is a non-starter. Envelope encryption makes it cheap again — you **rotate the KEK and re-wrap the DEKs** (small, fast) while leaving the bulk ciphertext untouched; individual DEKs roll over naturally as data is rewritten.

<Callout variant="warning" title="Crypto-shredding: deleting data by destroying its key">
Because encrypted data is unreadable without its key, **destroying the key reliably destroys the data** — even if copies are replicated across regions and buried in backups you can't practically reach. This is **crypto-shredding**, and it's the pragmatic answer to "delete this user's data everywhere" (e.g. GDPR erasure): give each user (or tenant) their own key, and to erase them, **destroy that key**. The ciphertext may physically remain, but it becomes permanently unrecoverable noise. The flip side is the standing risk of the whole topic — **lose a key you didn't mean to, and you've crypto-shredded your own data.** Key durability and backup are as vital as key secrecy.
</Callout>

<h2 id="key-management-in-practice">Key Management in Practice</h2>

Sound key management is mostly operational discipline around that single KMS chokepoint. On the read path, decryption runs the envelope in reverse — fetch the wrapped DEK, have the KMS unwrap it, decrypt locally, and cache the DEK:

Around that flow sit the practices that separate a real system from "we called AES":

- **Access control on keys.** Keys are protected resources with their own IAM policies — who may *use* a key to decrypt is distinct from who may *manage* (rotate, delete) it (cross-ref [Authorization](/topics/security/authorization)). Grant the minimum.
- **Separation of duties.** The people and services that *use* keys shouldn't be able to *export* them; key administrators shouldn't be able to *read* the data. The KMS enforces this split so no single actor holds both halves.
- **Audit logging.** A KMS logs **every** key use — each decrypt is a recorded event. That log is gold for investigations and anomaly detection: a service suddenly decrypting ten thousand records an hour is a visible signal.
- **Never hardcode keys.** Keys in source control, container images, or config files are the most common leak of all. Keys belong in the KMS; code asks for them at runtime.
- **BYOK / HYOK and per-tenant keys.** *Bring/Hold Your Own Key* lets customers supply or retain control of the key protecting their data — a common enterprise-trust requirement. **Per-tenant keys** give isolation (a compromise of one tenant's key exposes only that tenant) and make crypto-shredding a single customer trivial.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

As with password hashing, the disasters here are rarely broken cryptography — they're a handful of predictable mistakes around keys and modes.

<Callout variant="warning" title="Encryption & key-management footguns">
- **Hardcoded / committed keys.** A key in source control, a Docker image, or a config file is the single most common key leak. Keys live in the KMS; code fetches them at runtime and never logs them.
- **ECB mode.** Encrypting each block independently means identical plaintext blocks produce identical ciphertext, leaking structure (the infamous "ECB penguin"). Use an **authenticated mode like AES-GCM**, which also detects tampering.
- **Nonce / IV reuse.** Reusing a nonce (initialization vector) with the same key is catastrophic for stream and GCM modes — it can destroy confidentiality and forge integrity. Use a fresh, unique nonce per encryption.
- **Home-rolled crypto.** "We XOR with a secret" or a custom scheme is almost always broken. Use vetted libraries and standard modes; never invent your own.
- **No rotation.** One key, forever, encrypting everything, gives an unbounded blast radius. Rotate on a schedule (re-wrapping DEKs makes it cheap).
- **KEK in the same blast radius as the data.** The *wrapped* DEK beside the data is fine; the *master key* must live in the KMS/HSM, never in the application database or config.
- **Losing the key.** No key means permanently unrecoverable data. Key durability and backup are as critical as secrecy — crypto-shredding by accident is still data loss.
</Callout>
