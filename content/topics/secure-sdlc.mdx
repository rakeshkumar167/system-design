<h2 id="overview">What a Secure SDLC Is</h2>

Every other topic in this Security category is a *control* — a mechanism like [authorization](/topics/security/authorization), [encryption](/topics/security/encryption-key-management), or [session management](/topics/security/session-management). This one is about the **process** that makes sure those controls actually get designed, built, and maintained correctly. A **Secure Software Development Lifecycle (Secure SDLC)** integrates security into *every phase* of building software — requirements, design, implementation, testing, deployment, and operations — instead of treating it as a one-time check.

The anti-pattern it replaces is **security as a gate at the end**: build the whole thing, then run a penetration test a week before launch. That fails for a simple reason — by then the flaws are baked into the architecture and the code, so fixing them is expensive, slow, and disruptive, and the schedule pressure means many get shipped anyway. A Secure SDLC instead follows the principle of **shifting left**: move every security activity as early in the timeline as possible, because that's where problems are cheapest to fix (the [capacity section](#capacity-estimates) quantifies exactly how much cheaper). The modern name for doing this with automation woven through the delivery pipeline is **DevSecOps** — security as a continuous, shared, automated responsibility rather than a separate team's final sign-off.

<Callout variant="info" title="Build it in, don't test it in">
Security is a **property you build in continuously**, not a test you run at the end. The later in the lifecycle a flaw is found, the more it costs to fix — a missing control is free to add on a design whiteboard and ruinous to fix after a breach. A Secure SDLC pushes threat modeling, code analysis, and dependency scanning as early as they'll go, and automates them so they run on every change.
</Callout>

<h2 id="requirements-design">Requirements & Design: Threat Modeling</h2>

Security begins *before a line of code is written*. In the **requirements** phase, capture **security requirements** as first-class alongside functional ones — what data is sensitive, what authentication and authorization are needed, what compliance obligations apply — and write **abuse cases** (or misuse cases): for each feature, ask "how would an attacker misuse this?" A password-reset feature isn't done when it resets passwords; it's done when it resists account takeover.

In the **design** phase, the central activity is **threat modeling**: systematically walking the design to enumerate what can go wrong and deciding on controls before building. You map the system's **data flows** and **trust boundaries** (where data crosses from less-trusted to more-trusted), then enumerate threats against each — frameworks like **STRIDE** (Spoofing, Tampering, Repudiation, Information disclosure, Denial of service, Elevation of privilege) give you a checklist so you don't miss a category. This is the direct answer to OWASP's [**Insecure Design**](/topics/security/owasp-top-10) risk: a missing control is a *design* flaw no amount of careful coding fixes, and the cheapest place to catch it is on a whiteboard. (Threat modeling has enough depth for its own topic.)

<h2 id="secure-implementation">Implementation: Secure Coding & SAST</h2>

As code is written, three practices keep flaws out. First, **secure coding standards** — the recurring defenses from across this category: validate input, encode output, use parameterized queries, apply least privilege, fail closed. These are the concrete habits that prevent injection, broken access control, and the rest of the [OWASP Top 10](/topics/security/owasp-top-10). Second, **peer code review** with security in mind, so a second set of eyes catches what the author missed — and specifically that **secrets never get committed to source** (they belong in a secrets manager, not the repo).

Third, and the phase's signature automated control, **SAST — Static Application Security Testing**. SAST is **white-box** analysis: it reads the **source code** (without running it) looking for vulnerable patterns — a tainted input reaching a SQL string, a weak crypto call, a hardcoded credential. Its great virtue is that it runs **early and fast**, right in the developer's IDE and on every commit in CI, so a flaw is flagged **seconds after it's written** — the cheapest possible moment. Its limits: it can't see runtime or configuration behavior, and it produces **false positives** that need tuning. Used well, it's a fast feedback loop that stops whole classes of bug at the keyboard.

<h2 id="security-testing">Testing: SAST, DAST, IAST & SCA</h2>

The testing phase adds several complementary tools, and knowing when each applies is a common interview topic. No single one is sufficient — they see different things.

<TradeoffTable
  columns={["Tool", "How it works", "Finds", "Blind spot"]}
  rows={[
    ["SAST", "White-box: analyzes source code, not running", "Insecure code patterns, early", "Runtime/config issues; false positives"],
    ["DAST", "Black-box: attacks the running app from outside", "Runtime, config, and integration flaws", "Later; no source view; shallow coverage"],
    ["IAST", "Instruments the running app from inside", "Runtime flaws with code-level context", "Needs instrumentation; runtime only"],
    ["SCA", "Scans dependencies against known-CVE databases", "Known-vulnerable third-party components", "Only known CVEs, not novel bugs"],
  ]}
  recommendedRow={3}
/>

**SAST** (from the last section) reads source early. **DAST — Dynamic Application Security Testing** is the opposite: **black-box**, it attacks the **running** application from the outside like a real attacker would — finding runtime and configuration flaws SAST can't see, with no source access needed — but it runs later and its coverage is shallower. **IAST** instruments the app from the inside to combine both views. And crucially, **SCA — Software Composition Analysis** scans your **dependencies** against databases of known vulnerabilities: since modern apps are mostly third-party code, this is how you catch the **[vulnerable-components](/topics/security/owasp-top-10)** risk (the Log4Shell class) before it ships. **Fuzzing** rounds it out, hurling malformed input at the app to find crashes and edge cases. The point is to layer them — each covers another's blind spot.

<h2 id="supply-chain">CI/CD & Supply-Chain Security</h2>

The **CI/CD pipeline** is where a Secure SDLC becomes real, because it's where the automated checks run as **enforcement gates**: SAST, SCA, secret-scanning, and DAST execute on every change and can **fail the build**, so security isn't optional or forgotten.

<SecureSdlcPipelineSequence />

But the pipeline is also a **high-value target**, which raises **supply-chain security**. A compromised build system or a single malicious dependency poisons **everything downstream** — every customer who installs the resulting software — which is exactly the **[Software and Data Integrity Failures](/topics/security/owasp-top-10)** risk, the class of the **SolarWinds** attack. The defenses are about **provenance and integrity**: **sign build artifacts** so consumers can verify they're genuine and untampered; generate an **SBOM (Software Bill of Materials)** so you know *exactly* what components you ship (and can answer "are we affected?" the day a new CVE lands); verify build **provenance** with frameworks like **SLSA**; pin and verify dependencies; and lock down the pipeline's own credentials. You're securing not just your code but the entire chain that produces and delivers it.

<h2 id="capacity-estimates">Capacity: The Economics of Shifting Left</h2>

Everything in this topic rests on one empirical fact: the cost to fix a flaw rises **steeply the later it's found**. A bug caught while threat modeling costs almost nothing — you erase a box and redraw it. The *same* bug found in production costs dramatically more: an emergency patch, an incident response, possibly a breach with regulatory and reputational fallout. The classic industry figure is roughly **100×** between the earliest and latest phases, and that multiplier is the entire business case for shifting left.

<SecureSdlcCapacity
  assumptions={{
    vulnsPerRelease: 100,
    costFixInDesign: 100,
    prodCostMultiplier: 100,
    shiftLeftCatchRate: 0.9,
  }}
/>

Put numbers on it. Say a release introduces 100 flaws, each costing about **$100** to fix if caught early but **$10,000** (100×) if it escapes to production. If your only security check is a gate at the end and all 100 surface in production, that's **$1,000,000**. A Secure SDLC that catches **90%** early — threat modeling, SAST and SCA in the IDE and CI, code review — pays $100 for those 90 and $10,000 only for the 10 that slip through: a total of **$109,000**, saving **$891,000**, an **~89% reduction**. The lesson is blunt: shift-left isn't only safer, it's *far cheaper*, and the savings come precisely from moving detection into the earliest, most automated phases. Every dollar spent on early security tooling is leveraged against that 100× multiplier — which is why the investment is in prevention and early detection, not just late-stage testing.

<h2 id="deploy-runtime">Deployment, Runtime & Operations</h2>

Shipping isn't the finish line. At **deployment**, apply **secure configuration and hardening** — secure defaults, least functionality, no debug endpoints or default credentials (the OWASP **[misconfiguration](/topics/security/owasp-top-10)** risk) — and **inject secrets at deploy time** from a secrets manager rather than baking them into images. Run a final **DAST** pass against the deployed app in **staging**.

Then comes **runtime**, where two operational disciplines matter. **Monitoring and logging** provide the detection layer for whatever slips through every earlier gate (the OWASP **logging & monitoring** risk — you can't respond to what you can't see). And **vulnerability management** handles a subtle but critical fact: **your code doesn't have to change to become vulnerable.** A dependency that was clean at ship time becomes exploitable the day a new CVE is disclosed against it — so you must **continuously rescan** dependencies and infrastructure against fresh vulnerability data and **patch promptly**. Security operations is a standing loop, not a launch checklist.

<Callout variant="warning" title="Safe at ship time is not safe forever">
A component with no known vulnerabilities when you deploy can become critically vulnerable overnight — not because your code changed, but because a **new CVE was disclosed** against it (Log4Shell hit millions of unchanged apps this way). This is why a Secure SDLC includes **continuous** vulnerability management — rescanning your dependency inventory (your SBOM) against fresh CVE feeds and patching on a clock — rather than a one-time pre-launch scan.
</Callout>

<h2 id="verification-response">Verification & Response</h2>

Automation catches the known and the mechanical; **skilled humans** catch what it misses. Independent verification comes in escalating forms: **penetration testing** (experts actively try to break a specific application), **red teaming** (a simulated adversary attacks the whole organization — people, process, and technology — to test detection and response, not just find bugs), and **bug bounty** programs (paying a crowd of external researchers to report vulnerabilities continuously). Each surfaces creative, chained, and business-logic flaws that scanners can't reason about.

When something *is* found — by a scanner, a pentester, or a bounty report — the work is to **close the loop**, which is where **incident response** and a **vulnerability-management process** come in.

<VulnerabilityResponseSequence />

Findings are **triaged** and assigned a **severity**, **prioritized** against everything else competing for engineering time, then **fixed, tested, deployed, and verified** as resolved. Doing this reliably — with severity-based **remediation SLAs** so a critical flaw is patched in days, not someday — is what separates a program that *finds* vulnerabilities from one that actually *reduces risk*. A finding that sits unremediated in a backlog protects no one.

<h2 id="pitfalls-best-practices">Pitfalls & Best Practices</h2>

The failure modes of a security program are as consistent as the technical ones.

<Callout variant="warning" title="Secure SDLC footguns">
- **Security as a last-minute gate.** A pentest the week before launch finds flaws when they're most expensive and least likely to be fixed. Shift left.
- **No automation.** Manual security reviews don't scale and get skipped under deadline pressure. Bake SAST/SCA/secret-scanning into CI as gates.
- **Ignoring dependencies.** Most of your code is third-party. Without SCA and an SBOM you're blind to the vulnerable-components risk.
- **Secrets in code or images.** Keys committed to repos or baked into containers are a top leak. Inject them at runtime from a secrets manager.
- **Unsigned artifacts, no SBOM.** Supply-chain blindness — you can't verify what you ship or answer "are we affected?" when a CVE drops.
- **Scanning but never fixing.** Findings without remediation SLAs are theater. Prioritize and close them.
- **Treating it as one-time.** New CVEs land against unchanged code. Vulnerability management is continuous.
</Callout>
