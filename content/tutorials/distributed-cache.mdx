<h2 id="interview-framing">Interview Framing</h2>

A distributed cache is a fleet of in-memory nodes that sits in front of a slower backing store and answers reads in well under a millisecond. An application asks for a key; if the cache has it, the request never touches the database; if it doesn't, the app loads it from the store and remembers it for next time. The payoff is enormous — a high hit ratio turns a flood of reads into a trickle the database can actually serve, at a fraction of the latency — which is why a cache sits in front of nearly every large read-heavy system. The trap is to picture a single big hash map behind a socket. The moment one machine can't hold the data or serve the load, you need *many* nodes, and that one word — **distributed** — is where all the difficulty lives.

The hard part is not get/set; it's everything that "many nodes" forces. Keys must spread evenly across nodes, and crucially, when a node is added or dies, only a *small slice* of keys should move — because in a cache, a moved key is a **miss**, and moving all of them at once is a stampede on the database. That rules out the obvious `hash(key) % N` and demands **consistent hashing**. Memory is bounded and deliberately smaller than the dataset, so the cache must **evict**, and *which* items it keeps determines the hit ratio that justifies its existence. And two failure modes haunt every production cache: a single **hot key** that melts one node, and a popular key expiring into a **stampede** of concurrent misses. The reframe to lead with: *it looks like a networked hash map, but the hard part is distributing it — partition keys so churn moves few of them, evict the right items, and survive hot keys and stampedes — all while staying a lossy layer in front of the real source of truth.*

Spend the opening minutes scoping. Good questions to ask:

- What's the read/write mix and the QPS? Caches exist for read-heavy workloads; the numbers set the node count.
- How big is the hot working set versus the full dataset? That decides how much you can cache and your hit ratio.
- How stale can a read be? A cache is usually allowed to lag the store slightly — that latitude buys a lot of simplicity.
- What's the consequence of a miss — a slow query, or a catastrophic one? It changes how hard you defend against stampedes.
- Is this a look-aside cache the app manages, or a read-through cache that loads on its own?

**5 min** framing and the "distributed makes it hard" reframe · **5 min** requirements and the hit-ratio goal · **5 min** capacity (working set, node count, and store offload fall out here) · **5 min** entity model and the get/set API · **10 min** consistent hashing and eviction — the centerpiece · **10 min** replication, hot keys, and stampede defenses · **5 min** write policies, failure modes, and wrap-up. Say your assumptions out loud — the interviewer is grading whether you know why `hash % N` is fatal under churn and can defend against hot keys and stampedes, not whether you can name every eviction policy.
</Callout>

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** consistent hashing and virtual nodes, eviction policies, replication and tunable consistency, hot-key mitigation, cache stampede protection, and write policies/invalidation. **Out of scope:** the backing store's own design, durable cache persistence, transactions and scripting, secondary indexes and rich queries, the wire protocol details, and security/auth — name them so the interviewer sees the boundary, then set them aside. This problem also revisits the caching touched on in the URL Shortener and Pastebin tutorials, but from the inside: here the cache *is* the system.

Three targets shape every later decision. First, **a high hit ratio** is the cache's reason to exist — it's what turns a database-melting read load into a manageable one — and it's set by how much you can hold (memory) and how well you evict (policy). Second, **minimal key movement on membership change**: because a relocated key is a guaranteed miss, adding or losing a node must disturb only a small fraction of the keyspace, which is precisely what consistent hashing delivers and `hash % N` destroys. Third, **tunable, slightly-stale consistency**: a cache is allowed to lag the source of truth a little, and embracing that — rather than fighting for strict consistency — is what keeps it fast and simple. The cache is a lossy accelerator, not the system of record.

<h2 id="capacity-estimates">Capacity Estimates</h2>

Estimates here pin down three things: how much you *can't* cache (so eviction matters), how many nodes you need (from memory), and how much the hit ratio shields the backing store. We size all of it from one set of assumptions.

The takeaways. First, the cache holds a **fraction**, not the whole: the dataset is ~10 TB but the hot working set we keep is ~2 TB — so the cache is *deliberately* smaller than the data, and **eviction quality is what decides the hit ratio**. Cache everything and you've just built an expensive copy of the database; cache the right 20% and you serve 95% of reads. Second, **node count comes from memory**: holding a 2 TB working set at 64 GB per node needs ~**32 nodes**, and the keyspace is partitioned across them — which is exactly why consistent hashing matters. Third, and most important, **the hit ratio is the whole economic argument**: 50M reads/sec at a 95% hit ratio means only ~**2.5M/sec** reach the slow backing store — a 20× reduction — while each cache node fields ~**1.5M reads/sec** from memory. Push the hit ratio from 95% to 97.5% and you *halve* the load on the database again. A cache's capacity story is "hold the right fraction, and a couple of points of hit ratio is the difference between calm and catastrophe."

<h2 id="entity-model">Entity Model</h2>

The model is deliberately tiny — a cache is a key/value map — but the metadata around each entry is what makes eviction and expiry work. The center is the **CacheEntry**, and the most important conceptual point is what's *not* in the model: durability. The cache is lossy, derived state; the backing store is the source of truth.

Three concepts complete the picture. The **hash ring** is the logical structure that maps every key to a node (and, with virtual nodes, spreads keys evenly); a **node** owns an arc of the ring. A **replica set** holds copies of a node's entries on other nodes, for availability and read scale. And the **backing store** — a database or object store — is the source of truth the cache accelerates: every entry in the cache is a derived copy that can be regenerated, which is exactly what lets the cache be lossy, evictable, and fast.

<h2 id="api-design">API Design</h2>

The interface is the simplest in the curriculum — three verbs on a key — and that simplicity is the point: a cache trades rich semantics for speed. The defining behavior isn't in the request shape but in the response to a `GET`: **hit or miss**, where a miss is a normal, expected outcome the caller handles by loading from the store.

Keeping the surface this thin is deliberate: every feature you add to a cache is latency you can't afford on a sub-millisecond path.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture partitions keys across in-memory nodes, replicates each node for safety, and falls back to the backing store on a miss — with a membership service watching for node churn.

<DistributedCacheArchitecture />

- **Client (with cache library).** The application. Its cache client hashes a key onto the ring and sends the request straight to the owning node — so there's no central bottleneck on the hot path.
- **Cache router / consistent-hash ring.** The logic (in the client library or a thin proxy) that maps every key to its node and replicas. This is where minimal-movement-under-churn lives.
- **Cache nodes (shards).** The in-memory key/value stores, each owning an arc of the ring. They serve hits in sub-millisecond time, enforce TTLs, and evict under memory pressure.
- **Replicas.** Copies of a node's data on other nodes, for availability (a node can die without losing the shard's service) and read scale (reads can fan out to replicas).
- **Backing store.** The database behind the cache — the source of truth. On a miss, the value is read from here and populated into the cache; the cache exists to keep traffic *off* it.
- **Membership / gossip.** Tracks which nodes are alive via heartbeats/gossip, so the ring can be updated when a node joins or fails and routing stays correct.

The decisive boundary is **the cache is a fast, lossy front for a durable store**. Keys are spread by the ring so load is even and churn is cheap; replicas keep a shard available when a node dies; and a miss simply degrades to a (slower) read-through rather than an error. Every section that follows elaborates one piece: how the ring moves few keys (consistent hashing), how a bounded node decides what to keep (eviction), how replicas stay consistent enough (replication), and how the two signature failures — hot keys and stampedes — are tamed.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: a hit, a miss that reads through and populates, a node leaving the ring, and a stampede being collapsed. The first two are the everyday path; the last two are where "distributed" earns its keep.

**Cache hit.** The client hashes the key, sends the GET to the owning node, and the node returns the value from memory — one network hop, sub-millisecond, the database never involved. This is the path that should happen ~95% of the time.

<CacheHitSequence />

**Cache miss.** The key isn't cached (or expired). The node reports a miss, the value is loaded from the backing store, written back into the cache with a TTL, and returned — so the *next* read for that key is a hit.

<CacheMissSequence />

**Node rebalance.** A node fails (or joins). The membership service detects it and the ring reassigns only *that node's* arc of keys to its successor; every other node's keys stay exactly where they were. The relocated keys simply miss once and reload. This minimal movement is the entire reason for consistent hashing.

<NodeRebalanceSequence />

**Stampede.** A popular key expires and thousands of requests miss it at the same instant. Instead of all of them hammering the backing store, the node **coalesces** them into a single load (single-flight): one request fetches and populates, the rest wait and share the result.

<h2 id="consistent-hashing">Consistent Hashing</h2>

This is the centerpiece, and the reason a distributed cache is more than a sharded hash map. The naive way to spread keys across `N` nodes is `node = hash(key) % N`. It distributes evenly — and it is catastrophic under churn. The instant `N` changes (a node added or removed), the modulus changes for **almost every key**, so nearly the entire keyspace remaps to a different node at once. In a database that's an expensive reshuffle; in a *cache* it's far worse, because every remapped key is now a **miss** — so a single node failure converts into a near-total cache wipe and a thundering herd onto the backing store at the worst possible moment. Membership changes must be cheap, and `% N` makes them maximally expensive.

**Consistent hashing** fixes this by hashing both keys and nodes onto the same circular space — a **ring**. A key is owned by the first node found walking clockwise from the key's position. Now adding or removing a node only affects the **arc between that node and its predecessor**: when a node leaves, just its keys fall to the next node clockwise; when one joins, it takes over only the slice of keys in its new arc. Everyone else is undisturbed. Instead of remapping ~100% of keys, a membership change moves only about **1/N** of them — the minimal possible disruption, and exactly the property the requirements demanded.

<Callout variant="info" title="Virtual nodes make the ring even and smooth">
A plain ring has two problems: with few nodes, keys land unevenly (some nodes get a much bigger arc), and when a node dies, its *entire* arc dumps onto a single successor — an instant hotspot. **Virtual nodes** solve both: each physical node is placed on the ring at many positions (say a few hundred), so the keyspace is sliced into many small arcs spread across all nodes. Load evens out by the law of large numbers, and when a node fails, its many small arcs redistribute across *many* successors instead of crushing one. Virtual nodes also make heterogeneous hardware easy — a bigger node simply gets more virtual positions. This is how real systems (Dynamo, Cassandra, Memcached/Redis clients) actually deploy consistent hashing.
</Callout>

<h2 id="eviction-policies">Eviction Policies</h2>

Because the cache is deliberately smaller than the dataset, every node eventually fills up, and a `set` into a full node must **evict** something to make room. The eviction policy — which entry to drop — directly determines the hit ratio, so it's not a detail; it's the lever that decides whether the cache is worth its memory. The goal is to keep the items most likely to be requested again and discard the ones that won't be.

The common policies trade simplicity against accuracy. **LRU (least recently used)** evicts the entry untouched for the longest, betting that recently-used items will be used again — a strong default for most workloads, and cheap to approximate. **LFU (least frequently used)** evicts the entry with the fewest accesses, which better protects items that are popular over the long haul but slow to react when popularity shifts. **TTL/expiry** removes entries after a fixed lifetime regardless of use — the simplest way to bound staleness — and is usually layered *underneath* LRU/LFU as a correctness mechanism rather than a capacity one. **Random** eviction is crude but nearly free and surprisingly competitive at scale.

At very high throughput, exact LRU (a precisely ordered list updated on every access) is too much bookkeeping, so production caches use **approximations** — sampling a handful of entries and evicting the coldest of the sample, or coarse "recently used" bits — which get nearly the hit ratio of true LRU at a fraction of the cost. The lesson: pick the policy that matches your access pattern, and approximate it to keep the hot path fast.

<h2 id="replication-consistency">Replication & Consistency</h2>

Replication copies each node's entries onto one or more other nodes, and it buys two things. **Availability:** if a node dies, a replica already holds its shard, so the keys stay served instead of becoming a wall of misses against the backing store. **Read scale:** popular keys can be read from any replica, spreading hot-key load across machines. The cost is keeping the copies consistent enough — and "enough" is the operative word, because a cache has latitude a database doesn't.

The central choice is **synchronous vs. asynchronous** replication. Synchronous (write to the replica before acknowledging) keeps replicas perfectly in step but adds latency to every write and stalls if a replica is slow. Asynchronous (acknowledge immediately, copy in the background) keeps writes fast but means a replica can briefly serve a slightly stale value. For a cache, **async is usually right**: the cache is *already* an approximate, slightly-stale view of the source of truth, so a replica being a few milliseconds behind is well within the staleness the system already tolerates. This is the heart of **tunable consistency** — you dial freshness against latency and scale per workload, and most caches happily sit on the relaxed end.

<h2 id="hot-keys">Hot Keys & Hotspots</h2>

Consistent hashing spreads keys evenly, but it can't spread *one* key — and a single wildly popular key (a celebrity's profile, a viral post, a global config value) routes all its traffic to one node, which can saturate that node's CPU or network while the rest of the cluster sits idle. This **hot key** problem is a hotspot that even sweet partitioning can't fix, because the unit of distribution is the key, and you can't shard below a single key by hashing alone.

The mitigations stack. **Replicate the hot key** across several nodes and read from any of them, spreading its load — the most direct fix. **Client-side (near) caching** keeps the hottest keys in the application process itself, so most reads never even reach the cache cluster; the trade is a little extra staleness from the local copy. **Key splitting** turns one hot key into several (`config` → `config:0..9`) so the load fans across nodes, where the value tolerates it. And **request coalescing** (next section) keeps a hot key's *misses* from amplifying. The detection side matters too: track per-key request rates so a hot key is spotted and promoted to one of these defenses before it melts a node.

<h2 id="cache-stampede">Cache Stampede</h2>

A **cache stampede** (or dogpile / thundering herd) is the cache's most famous failure, and it strikes precisely when a key is *popular*. A heavily-read key expires or is evicted; in the next instant, thousands of concurrent requests all look for it, all miss, and all rush to the backing store to recompute the same value at the same time. The store, sized for the *cached* read rate of that key (≈ zero, since the cache was absorbing it), is suddenly hit by its full uncached load in a spike — and may buckle, which slows every request, which lengthens the window, which makes it worse. One expiry can cascade into an outage.

The defenses tame the herd at the moment of the miss:

- **Request coalescing / single-flight.** When many requests miss the same key at once, let exactly **one** of them load from the store while the others wait for its result. The store sees a single load instead of thousands — the most important defense, and the one shown in the stampede flow.
- **TTL jitter.** If many related keys are set with the same TTL, they all expire together and stampede in lockstep. Adding a small random offset to each TTL **spreads expiries over time** so misses arrive as a trickle, not a wall.
- **Early / asynchronous recompute.** Refresh a hot key *before* it expires (in the background, while still serving the old value), so it never goes cold under load.
- **Locks / leases.** A miss grabs a short lease to recompute; others briefly serve stale or wait, ensuring only the lease-holder hits the store.

<Callout variant="warning" title="Popularity is what makes a stampede dangerous">
The cruel irony of a stampede is that it scales with success: the more a key is read, the bigger the herd when it expires, and the harder the backing store is hit by a load it was being shielded from. So the keys you most need the cache for are exactly the ones that can take the database down when they expire. That's why a serious cache design treats stampede protection — coalescing plus TTL jitter — as mandatory for hot keys, not optional. It's the same single-flight idea the URL Shortener tutorial used to collapse concurrent cache-miss loads; here it's load-bearing.
</Callout>

<h2 id="write-policies">Write Policies & Invalidation</h2>

So far reads dominate, but writes raise the hardest question in caching: when the underlying data changes, how do you keep the cache from serving a stale copy? The answer is a **write policy**, and the four common ones trade freshness, latency, and complexity differently.

- **Cache-aside (lazy loading).** The application owns the cache: on a read it checks the cache, loads from the store on a miss, and populates; on a write it updates the store and **invalidates** (deletes) the cached entry, so the next read reloads fresh. Simple, resilient (a cache outage just means more store reads), and the most common pattern — at the cost of a guaranteed miss after every write.
- **Write-through.** Writes go *through* the cache, which updates itself and the store synchronously. The cache is always fresh and reads after a write hit, but every write pays the cache-plus-store latency, and you cache data that may never be read.
- **Write-back (write-behind).** Writes update the cache and are flushed to the store asynchronously. Writes are very fast and absorb bursts, but an unflushed node crash can lose writes — so it trades durability for speed and is used only where that's acceptable.
- **Write-around.** Writes go straight to the store, bypassing the cache; the entry is cached only on a later read. This avoids polluting the cache with write-once data, at the cost of a miss on the first read after a write.

The recurring hazard across all of these is **invalidation** — making sure a changed value doesn't linger in cache. TTLs provide a cheap upper bound on staleness (the entry self-expires), and explicit deletes on write provide immediacy; most systems use both. As the saying goes, cache invalidation is one of the genuinely hard problems, because every write path must remember to invalidate, and a missed invalidation is a silently stale read.

/>

The progression tracks the failure you're buying out of. A single node proves the cache earns its keep, but it has no resilience — lose it and you've wiped the cache and herded the store. Sharding with a **consistent-hash cluster** gives capacity and cheap churn; adding **replicas plus hot-key and stampede defenses** removes the two ways a cache takes down the very store it protects — and is where most systems should live. Only a global product needs **multi-region or tiered caching** (a small near-cache in front of a larger remote cluster), and it pays for it in the genuine difficulty of cross-region invalidation. Most distributed caches are happy at "consistent-hash cluster, replicated, with stampede protection."

<h2 id="resiliency-failure-modes">Resiliency & Failure Modes</h2>

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. A cache's failures are unusual because the cache itself is *expendable* — losing cached data isn't data loss — so the real danger isn't the cache breaking; it's the cache failing in a way that **stampedes the backing store**.

<FailureMatrix
  rows={[
    { failure: "Single cache node fails", impact: "That shard's keys miss until reloaded; not data loss", detection: "Membership heartbeat / gossip timeout", mitigation: "A replica takes over the shard; consistent hashing reassigns only that node's arc; reads fall through to the store meanwhile", recovery: "Replica promoted or node rejoins; keys repopulate on miss" },
    { failure: "Cold start / mass cache wipe", impact: "Near-zero hit ratio drives a herd onto the store", detection: "Hit-ratio drop; store load spike", mitigation: "Warm the cache (preload hot keys); coalesce misses; rate-limit store loads; gradual ramp", recovery: "Hit ratio climbs as the working set repopulates" },
    { failure: "Hot-key meltdown", impact: "One node saturates on a single popular key", detection: "Per-key/per-node rate skew", mitigation: "Replicate the hot key across nodes, near-cache it client-side, split the key", recovery: "Load spreads across nodes/clients; node recovers" },
    { failure: "Cache stampede on expiry", impact: "Concurrent misses dogpile the store, risking an outage", detection: "Spike of identical misses; store latency", mitigation: "Request coalescing / single-flight; TTL jitter; early async recompute", recovery: "One load repopulates; the herd is collapsed to a single fetch" },
    { failure: "Replication lag / split-brain", impact: "A replica serves a stale or divergent value", detection: "Replica version skew; partition alarms", mitigation: "Bounded staleness is accepted; on partition, prefer one primary per shard; reconcile by version", recovery: "Replicas converge; stale entries expire by TTL" },
    { failure: "Memory pressure / mass eviction", impact: "Hit ratio drops as entries are evicted early", detection: "Eviction rate; memory high-water", mitigation: "Add nodes/memory; tune eviction; cap item sizes; shed low-value keys", recovery: "Working set fits again; hit ratio recovers" },
    { failure: "Backing store outage", impact: "Misses can't be served — the cache can't shield a dead store", detection: "Store error rate on read-through", mitigation: "Serve stale cached values where acceptable; coalesce and rate-limit retries; fail fast", recovery: "Store returns; misses succeed and repopulate" },
  ]}
/>

The theme: **a cache can't lose data, but it can stampede the store.** Because every entry is a regenerable copy, node loss and eviction degrade to *misses*, not data loss — annoying but recoverable. The failures to design hardest against are the ones that convert a cache hiccup into a **load spike on the backing store**: a cold start, a hot key, and an expiry stampede. Coalescing, replication, near-caching, and TTL jitter all exist to keep a cache problem from becoming a database outage.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Partition keys across in-memory nodes with consistent hashing and virtual nodes, evict with (approximate) LRU plus TTLs, replicate shards asynchronously for availability and read scale, manage the cache with the cache-aside pattern (invalidate on write), and defend the backing store with hot-key replication/near-caching and stampede protection (request coalescing + TTL jitter)."
  rationale="Consistent hashing is what makes the cache distributable: it spreads keys evenly and, crucially, moves only ~1/N of them when a node joins or fails, so churn doesn't wipe the cache. LRU with TTLs keeps the right hot fraction in bounded memory, setting the hit ratio that justifies the system. Async replication matches a cache's already-tolerated staleness while adding availability. And because the cache's worst failures are the ones that stampede the store, coalescing, near-caching, and TTL jitter are treated as core, not optional."
  alternatives="hash%N sharding — even distribution but catastrophic churn, disqualified; LFU instead of LRU — better for stable popularity, slower to adapt; synchronous replication — fresher replicas but slower writes and stall risk; write-through/write-back instead of cache-aside — fresher or faster writes but more coupling and (write-back) durability risk; no replication — simpler but a node loss wipes a shard and herds the store."
  revisitWhen="The access pattern shifts (favoring LFU or a different working-set size), staleness tolerance tightens (pushing toward sync replication or write-through), the workload develops persistent hot keys (more near-caching/splitting), or global latency/residency forces multi-region tiered caching with its harder invalidation."
/>

Three axes capture the real tension:

- **Consistent hashing vs. simpler sharding.** `hash % N` distributes just as evenly and is trivial, but it remaps almost everything on churn — fatal for a cache. Consistent hashing (with virtual nodes) costs a little complexity to make membership changes cheap; for a cache that's non-negotiable.
- **LRU vs. LFU (and exact vs. approximate).** LRU adapts fast to changing hotness and is cheap to approximate; LFU protects long-term-popular items but reacts slowly. Default to (approximate) LRU and reach for LFU only when popularity is stable and long-lived.
- **Freshness vs. speed.** Synchronous replication and write-through keep the cache fresh but slow writes and add coupling; async replication and cache-aside keep it fast and simple at the cost of slight staleness. Because a cache is a slightly-stale accelerator by nature, default to the fast, relaxed end and tighten only where correctness demands.

<h2 id="interview-summary">Interview Summary</h2>

<Callout variant="interview" title="A 60-second end-to-end answer">
"A distributed cache is a fleet of in-memory nodes in front of a slower store, and its whole value is a high hit ratio — at 95%, fifty million reads a second become two and a half million on the database. It looks like a networked hash map, but 'distributed' is the hard part. First, partitioning: hash%N is fatal because changing the node count remaps almost every key, and in a cache a remapped key is a miss, so one node failure wipes the cache and herds the store — so I use consistent hashing on a ring with virtual nodes, which moves only about 1/N of keys on a membership change and keeps load even. Second, eviction: the cache is deliberately smaller than the data, so it evicts with approximate LRU plus TTLs, and that policy sets the hit ratio. Third, replication: async replicas give availability and read scale, and async is fine because a cache is already a slightly-stale view of the source of truth. Then the two signature failures: a hot key all routes to one node, mitigated by replicating it, near-caching it client-side, or splitting it; and a stampede, when a popular key expires and thousands of misses dogpile the store, mitigated by request coalescing so only one load happens, plus TTL jitter. Writes use cache-aside — update the store, invalidate the entry. The cache can't lose data since it's all regenerable, but its worst failures are the ones that stampede the store, so coalescing and replication are core."
</Callout>
