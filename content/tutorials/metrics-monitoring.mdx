<h2 id="interview-framing">Interview Framing</h2>

A metrics and monitoring system collects numeric measurements from a fleet of machines and services — CPU, request rate, error count, queue depth, latency — stores them over time, draws the dashboards engineers stare at, and pages someone when something breaks. Prometheus, Datadog, InfluxDB, and Google's Monborg/Monarch are all this system. From the outside it looks like "store some numbers and graph them," which undersells it badly: the numbers arrive as a **continuous firehose from millions of sources forever**, every dashboard is a range-aggregation over huge volumes, and an alerting engine must evaluate rules over that data every few seconds without missing a beat.

The hard part is that this is a **write-dominated, time-series problem with a cardinality bomb inside it**. Three forces dominate. First, **ingest throughput**: every monitored series produces a sample every scrape interval, so the system takes in millions of samples per second, continuously, and can never push back on the producers (a monitoring system that drops data or slows down the thing it monitors is worse than useless). Second, **storage economics**: that firehose would be tens of terabytes a day raw, so the design lives or dies on **time-series compression** and **downsampling + retention** to make keeping the data affordable. Third, **cardinality**: a "time series" is identified by a metric name plus a set of **labels** (`http_requests_total{method="GET", status="200", host="web-42"}`), and the number of distinct label combinations — the cardinality — is what actually bounds memory and index size; one careless high-cardinality label (a user ID, a request ID) can multiply your series count by millions and take the whole system down. The reframe to lead with: *graphing numbers is trivial; the design is absorbing a relentless write firehose without ever backpressuring producers, compressing regular time-series data ~8× and downsampling the long tail so storage is affordable, answering range-aggregation queries over recent data fast, and evaluating alert rules continuously — all while keeping cardinality under control, because cardinality, not raw sample volume, is what kills a metrics system.*

Spend the opening minutes scoping. Good questions to ask:

- What are we collecting — **metrics** (regular numeric time series) specifically, as opposed to **logs** (text events) or **traces** (request spans)? They're different systems with different storage; this is metrics.
- **Push or pull** ingestion — does the server scrape targets, or do agents push? It shapes discovery, target-health, and the short-lived-job story.
- What **resolution and retention** — samples every 10s kept how long? Raw forever is infeasible; this drives downsampling.
- How **fresh** must alerts be, and can we tolerate approximate/aggregated history? Freshness sizes alert evaluation; approximate old data enables downsampling.
- What's the **cardinality budget** — how many active series, and are labels bounded? This is the question that most often separates a system that scales from one that falls over.

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** the time-series data model and cardinality, push vs pull ingestion, the compressed time-partitioned TSDB, downsampling and retention, range-aggregation querying, and alert-rule evaluation with notification handoff — plus scaling, failure modes, and trade-offs. **Out of scope:** **logs** and **traces** (the other two pillars of observability — logs are the **Distributed Logging** tutorial; distributed tracing is its own system), the visualization/dashboard front-end beyond the query API it calls, and the anomaly-detection/ML layer some products build on top. This problem shares DNA with the **Distributed Logging** platform (both are write-firehose ingestion pipelines that must never backpressure producers) but diverges sharply on storage: logs are high-cardinality text searched ad hoc, metrics are regular numeric series aggregated over time — which is exactly why metrics get their own compressed time-series store.

Three targets shape every later decision. First, **it's overwhelmingly a write problem**: the system continuously ingests every series at every scrape, so the ingestion path and the storage engine must be built for sustained high-throughput **appends**, and — critically — must **never backpressure the producers**, buffering through a WAL/queue instead, because a monitoring system that degrades the systems it watches is unacceptable. Second, **storage must be bounded without losing usefulness**: raw high-resolution data forever is infeasible, so the design leans on aggressive **time-series compression** (points are regular and slow-changing, so they compress ~8×) plus **downsampling and tiered retention** (keep recent data fine-grained, roll up old data to coarse resolution) — because a dashboard showing last year only needs hourly points, not per-second ones. Third, **reads are modest and recent-biased**: dashboards and alerts almost always query the last minutes-to-hours, so queries touch recent, contiguous data and can be fast, and heavy dashboards can be precomputed. Note what's **relaxed**: metrics are **eventually consistent** and **lossy-tolerant at the margin** (a missed scrape leaves a gap, which is fine — you don't retry individual samples forever), and old data is deliberately **approximate** (downsampled). The system trades perfect fidelity of ancient data and per-sample durability guarantees for the ability to absorb an enormous write stream affordably.

The takeaways. First — the signature lesson — **it's a write firehose**: 1 million targets each exposing ~200 series is **200 million active series**, and scraping every one every 10 seconds is ~**20 million samples per second**, ingested continuously and forever. That's the number that dictates the architecture: the ingestion and storage tiers are built for sustained append throughput, and the read side is a comparative afterthought. Second, **compression is the enabling trick**: at 16 raw bytes per sample the firehose is ~**27.6 TB/day**, which is untenable to retain — but time-series data is extraordinarily compressible because **timestamps are regularly spaced and values change slowly**, so **delta-of-delta timestamp encoding** plus **XOR float compression** (the Facebook "Gorilla" scheme) squeezes each sample to ~**2 bytes** (~**8×**), dropping it to ~**3.5 TB/day**; downsampling and retention then bound the long tail. Third — the one that isn't in the sample math — **cardinality is the real memory limit**: those 200 million *active series* each need an in-memory head chunk and an index entry, so RAM is governed by the *number of distinct series*, not the sample rate. Double the scrape frequency and you double samples but not memory; add one unbounded label and you can 1000× the series count and OOM the box. The capacity story: size for **ingest throughput and storage** (with compression and downsampling as the levers), and guard **cardinality** as the constraint that actually takes systems down — query latency barely registers by comparison.

Two ideas complete the picture. The **inverted (posting-list) index** is what makes label queries possible: for each `key=value` label it stores the set of series that have it, so a selector like `{method="GET", status="200"}` is a set intersection of posting lists — the same structure a text search engine uses, here over labels. And **metric type** drives interpretation: a **counter** is monotonically increasing (you almost never read it raw — you compute its **rate**), a **gauge** is a free-moving value (temperature, queue depth — read directly), and a **histogram** is really a family of bucket-counter series that let you estimate percentiles at query time. The discipline: a series' **identity is its full label set** (so labels must be **bounded**, low-cardinality dimensions), the **cardinality** of distinct series is the memory constraint to respect, and samples are stored **append-only, time-ordered, and compressed** per series.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture is a **write pipeline** feeding a **compressed time-series store**, with a **query engine** and an **alerting engine** reading from it and a **downsampler** keeping storage bounded.

<MetricsArchitecture />

- **Monitored Targets.** The hosts, containers, and services exposing metrics — via an exporter/agent that the server scrapes, or that pushes samples.
- **Ingestion Service.** Receives (scrapes or accepts) samples, validates labels, and appends them durably. Built for sustained append throughput; never backpressures producers.
- **Time-Series DB.** The heart: samples land in an in-memory **head block** (backed by a WAL) and are periodically flushed to immutable, time-partitioned, **compressed** blocks, indexed by an inverted label index.
- **Query Engine.** Parses queries, selects series via the index, range-scans blocks, and applies aggregation functions — recent-biased and fast.
- **Dashboards.** The read clients issuing PromQL-style queries.
- **Alerting Engine.** Continuously evaluates rule expressions over recent windows and fires deduplicated, grouped notifications.
- **Downsampler.** A background job that compacts old raw blocks into lower-resolution rollups and expires data by retention policy.

The decisive boundary is that **the write path is optimized for sustained compressed appends while the read path serves cheap recent-range scans** — the store is deliberately write-and-compression-optimized, the opposite of a general database. Every section that follows elaborates one piece: the data model and cardinality, the ingestion model, the compressed storage engine, downsampling and retention, querying and aggregation, and alerting.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: ingesting the firehose, serving a range query, evaluating an alert, and downsampling old data. The first two are the hot request paths; the last two are the continuous background work that makes monitoring useful and affordable.

<h2 id="data-model-cardinality">The Data Model & Cardinality</h2>

Everything starts with how a measurement is identified. A metric like `http_requests_total` isn't one number — it's a **family of time series**, one per unique combination of **labels**: `http_requests_total{method="GET", status="200", host="web-42"}` and `http_requests_total{method="POST", status="500", host="web-13"}` are two entirely separate series. The metric name plus the exact label set **is** the identity; change any label value and you've named a different series. This dimensional model is what makes metrics powerful — you can slice error rate by status, host, region, endpoint — and it's also the source of the system's single most dangerous failure mode.

That failure mode is **cardinality explosion**. The total number of distinct series — the **cardinality** — is the product of the distinct values across labels, and it directly determines index size and, worse, the number of in-memory head chunks the TSDB must hold (one active chunk per series). Bounded labels are fine: `method` has ~5 values, `status` ~a few dozen, `host` maybe thousands — their product is large but manageable. The catastrophe is putting an **unbounded, high-cardinality value in a label**: `user_id`, `request_id`, `email`, a full URL with query params, a raw error message. Now every user or request mints a *new* series that lives forever in the index and memory, and cardinality jumps from millions to billions — the classic way a metrics system OOMs and falls over, often taking down monitoring exactly when you need it during an incident. The rules that follow: labels must be **bounded, low-cardinality dimensions**; identifiers and free-form text belong in **logs or traces**, not metric labels; and the system should **guard cardinality** actively (limits per metric, alerts on series growth, rejecting or dropping offending labels). The principle: the metric+label model gives dimensional slicing, but **cardinality — the count of distinct series — is the true resource constraint**, so keeping labels bounded is the most important discipline in operating a metrics system.

<h2 id="ingestion-pipeline">Ingestion: Push vs Pull</h2>

How samples get from targets into the system is a genuine design fork, and the classic answer surprises people: the dominant model, popularized by Prometheus, is **pull (scraping)** — the monitoring server periodically fetches a `/metrics` endpoint from each target on a fixed interval. Pull has real advantages: the server owns **service discovery** (it decides what to scrape, integrating with Kubernetes/Consul/etc.), a **failed scrape is itself a health signal** (if the target doesn't answer, you know it's down — you get liveness for free), there's no risk of a runaway client flooding you, and targets stay dumb (just expose current values, no delivery logic). The cost: the server needs network reachability to every target, and **short-lived jobs** (a batch job, a serverless function, a CI run) may finish before they're ever scraped.

**Push (remote-write)** inverts it: agents on the targets send samples to the server. Push shines exactly where pull struggles — **short-lived and serverless workloads** (push before you exit), **network-isolated targets** that the server can't reach (they push out through a gateway), and event-driven sources. Its costs mirror pull's benefits: you lose the free liveness signal (silence is ambiguous — down, or just quiet?), and you must defend against a misbehaving client flooding the ingest path. Many production systems support **both**, and use a **push gateway** to bridge short-lived jobs into a pull-based world. Whichever direction, the ingestion tier shares one non-negotiable property from the requirements: it **must never backpressure producers**. It absorbs bursts through an in-memory buffer and a **write-ahead log**, and if it's truly overwhelmed it **sheds load** (drops samples, returns 429) rather than slowing down — because the cardinal rule is that monitoring must not degrade the systems it monitors. The principle: **pull** gives you discovery and free target-health at the cost of reachability and short-job gaps; **push** covers short-lived and isolated workloads at the cost of the liveness signal and flood risk; and either way ingestion **buffers and sheds, never backpressures**.

<h2 id="tsdb-storage">Time-Series Storage & Compression</h2>

The storage engine is purpose-built and looks nothing like a general database. Incoming samples for each series first land in an in-memory **head block** — the current, still-growing chunk — with every write also appended to a **write-ahead log** on disk so a crash can be replayed with zero loss. Periodically (say every couple of hours) the head is cut and **flushed to an immutable, time-partitioned block** on disk: data is organized into non-overlapping time windows, each block self-contained with its own index and chunks. Immutability is a gift — blocks are written once and never modified, so there's no in-place update complexity, compaction just merges adjacent blocks, and retention is "delete the old block." This is the same **LSM-ish, append-only, time-partitioned** shape you'd want for any write-heavy store, specialized for time.

The magic is the **compression**, and it's the reason a metrics store is affordable. Time-series data has two properties that make it extraordinarily compressible. First, **timestamps are regularly spaced** — a scrape every 10 seconds means deltas are almost always exactly 10s, so instead of storing 8-byte timestamps you store the **delta-of-delta**: the change in the interval, which is usually **zero** and encodes in a single bit. Second, **consecutive values change slowly** — CPU is 40.1% then 40.2% — so the **XOR of successive float values** has mostly leading/trailing zero bits, which pack tightly. This pair of techniques — delta-of-delta timestamps + XOR floats — is the **Facebook "Gorilla"** scheme, and it takes a 16-byte (timestamp, value) sample down to **~1.3–2 bytes on average**, an ~8–12× reduction, with no loss. That's what turns ~27.6 TB/day of raw firehose into ~3.5 TB. The principle: store samples **append-only in immutable, time-partitioned blocks** (WAL for durability, head block in memory for the hot window), and exploit the **regularity of time series** with **delta-of-delta timestamp** and **XOR value** compression to shrink the data ~8×, because keeping the firehose is only affordable once it's compressed.

<h2 id="downsampling-retention">Downsampling & Retention</h2>

Compression makes a day of data affordable; **downsampling and retention** make *years* of data affordable. The insight is that **resolution needs decay with age**: you want per-second detail for the last few hours (to debug a live incident), but for a dashboard showing the last year, hourly points are not just enough — they're all the screen can render anyway. So the system keeps **raw, high-resolution** data only for a recent window, and **downsamples** older data into progressively coarser **rollups**: raw 10-second points get aggregated into 1-minute rollups, then 5-minute, then 1-hour, each storing summary statistics (min, max, sum, count, and often specific quantiles) per interval so the aggregations dashboards ask for still work.

<DownsampleRetentionSequence />

**Retention** is the policy layer on top: each resolution tier has a lifetime — raw for, say, 15 days, 5-minute rollups for 90 days, 1-hour rollups for 2 years — after which blocks are dropped. Because storage is immutable time-partitioned blocks, both downsampling (read old raw blocks, write rollup blocks) and expiry (delete blocks past their tier's lifetime) are clean batch operations. Long-term rollups are often shipped to cheaper **object storage** (a remote long-term store) while recent hot data stays on fast local disk — a storage tiering that mirrors the **Distributed Logging** platform's hot/warm/cold tiers. The trade-off is **fidelity vs cost**: downsampling permanently loses the fine detail of old data (you can't recover the exact per-second spike from last March), which is precisely the bargain you want, since nobody debugs a live incident with year-old per-second data. The principle: **decay resolution with age** — keep recent data raw, roll up old data into coarser summaries, and expire by tier — so total storage stays bounded no matter how long you retain history, trading irrecoverable fine detail of the distant past for affordability.

<h2 id="query-aggregation">Querying & Aggregation</h2>

Queries are where the dimensional model pays off, and they have a characteristic shape: **select series by labels, over a time range, then aggregate**. A query like `sum(rate(http_requests_total{status="500"}[5m])) by (host)` reads as: take the `http_requests_total` series with `status="500"`, compute each one's **per-second rate** over 5-minute windows, then **sum** them **grouped by host**. Three families of function dominate. **Rate/increase** turn a monotonic **counter** into the meaningful thing — requests *per second* — because the raw counter value (ever-climbing, reset on restart) is useless directly. **Aggregations** (sum, avg, min, max, count) collapse many series into fewer, `by (label)` controlling the grouping. And **percentiles** come from **histograms**: since you can't average percentiles, latency is recorded as bucketed counter series and `histogram_quantile` estimates p50/p99 from the buckets at query time.

Two properties keep queries fast. First, **reads are recent-biased**: dashboards and alerts overwhelmingly ask about the last minutes to hours, so a query selects a handful of series via the inverted index and range-scans their **recent, contiguous** compressed chunks — a sequential read of a small amount of data, not a scan of the whole store. Second, **heavy queries are precomputed**: a dashboard that aggregates millions of series (`sum by (region)` across a fleet) is expensive to run live on every refresh, so a **recording rule** evaluates the expression on a schedule and writes the result into a **new, low-cardinality series** that the dashboard reads cheaply — trading a little staleness and storage for a massive query speedup. The guardrail is the **cardinality of the query itself**: a selector that matches millions of series (a too-broad label match) can blow up the query engine, so the system limits or rejects queries that would fan out too far. The principle: queries **select-by-label, range-scan recent compressed chunks, and aggregate** (rate for counters, histograms for percentiles); keep them fast by exploiting **recency** and by **precomputing** heavy aggregations with recording rules — while guarding against **high-cardinality queries** that fan out over too many series.

<h2 id="alerting">Alerting</h2>

Dashboards are for humans looking; **alerting** is the system watching so humans don't have to. An alerting engine holds a set of **rules**, each an expression plus a threshold (`rate(errors[5m]) > 10`), and **evaluates every rule on a fixed interval** (say every 30 seconds) against the recent data — effectively running the same range queries continuously and checking the condition. The subtlety that makes alerting *usable* rather than maddening is the **`for` duration**: an alert doesn't fire the instant its condition is true, but only after it has stayed true for a specified time (e.g. 10 minutes). This **suppresses flapping** — a momentary spike, a single bad scrape, a brief blip won't page anyone; only a sustained problem does.

<AlertEvaluationSequence />

Once a rule fires, the raw firing isn't sent straight to a human — a separate stage handles **notification management** (in Prometheus this is Alertmanager): **deduplication** (the same alert firing from many replicas becomes one notification), **grouping** (twenty hosts failing the same rule become one grouped page, not twenty), **routing** (severity and labels decide who's paged, emailed, or Slacked), **silencing** (mute known issues or maintenance windows), and **inhibition** (a "datacenter down" alert suppresses the hundred "host unreachable" alerts it obviously causes). The actual delivery — the push notification, SMS, or email — is then handed off to notification channels, which is the **Notification Service** problem (retries, dedup, multi-channel fan-out) from earlier in the curriculum. The trade-off woven through alerting is **sensitivity vs noise**: shorter `for` durations and tighter thresholds catch problems faster but page on noise; longer ones are quieter but slower — and an alerting system that cries wolf gets ignored, which is its own kind of outage. The principle: evaluate rules **continuously over recent windows**, use a **`for` duration to suppress flapping**, and **dedup, group, route, silence, and inhibit** firing alerts before handing delivery to a notification service — tuning sensitivity so alerts are trustworthy, because a noisy alerting system is a useless one.

<h2 id="scalability-evolution">Scalability & Evolution</h2>

Don't build the globally-federated, long-term-tiered system on day one. Each stage answers a specific pressure and adds a specific cost — name the trigger before taking on the complexity.

  columns={["Stage", "Trigger", "Cost"]}

    ["HA pair (replicated scrape)", "Need monitoring to survive a node loss", "Two nodes scrape/ingest the same targets; dedup at query time; 2× cost"],

The progression tracks the wall you're hitting. A **single TSDB** is genuinely enough for a lot — a compressed store on one beefy box handles millions of series and is how many teams run Prometheus — but it caps ingest and cardinality at one machine and is a single point of failure with only local retention. **HA pairs** (two identical servers scraping the same targets, deduplicated at read time) buy availability so monitoring survives a node loss — table stakes for production. The center of gravity for scale is **sharding series across nodes with a remote long-term store**: distribute the cardinality and ingest load, and offload downsampled history to cheap object storage (Thanos/Cortex/Mimir-style), keeping each node's working set bounded while retaining years affordably. Only genuinely large orgs need **global federation** — per-cluster TSDBs with a global query layer stitching them into one view — paying cross-cluster query complexity for a single pane of glass. Most systems are well-served by "HA pair, shard when cardinality demands it, tier old data to object storage."

<h2 id="resiliency-failure-modes">Resiliency & Failure Modes</h2>

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. The monitoring system's guiding rule is uncomfortable but essential: **monitoring must be more reliable than the things it monitors, and must never take them down** — so ingestion degrades by shedding load, never by backpressuring, and the scariest failure is the system killing itself with cardinality.

The theme: **degrade by shedding, never by backpressuring; and monitor the monitor.** Ingest crashes are survived by the WAL (replay the head) and an HA peer covering the gap. Cardinality explosion — the signature self-inflicted outage — is contained *before* it happens with per-metric series limits, relabeling to drop bad labels, and rejecting offending series at ingest, because once the box OOMs you've lost visibility exactly when you need it. A failed scrape isn't really a failure — it's the **liveness signal** the pull model gives you for free, and gaps are expected rather than retried forever. Overload on either the write or read side is met by **load-shedding and cardinality guards**, never by slowing producers. And the subtlest failure is the **alerting engine silently dying** — if the watcher stops watching, nothing tells you — so you need **meta-monitoring**: a **dead-man's switch** (a heartbeat alert that fires *unless* the pipeline is healthy) and an HA alerting setup. Because the store is append-only with a WAL and immutable blocks, data recovery is usually "replay the WAL or fail to the peer," while the operational discipline is keeping cardinality bounded and watching the watcher.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"

  rationale="Monitoring is a write-firehose problem, so the store is optimized for compressed appends and recent-range reads, not general queries; time-series regularity makes ~8× compression the lever that keeps the firehose affordable, and downsampling/retention bound the long tail.   alternatives="A general-purpose SQL/NoSQL database for metrics — works small but can't match a TSDB's compression or ingest throughput and struggles with the dimensional model.   revisitWhen="Cardinality or ingest outgrows one node (shard series; enforce stricter label limits); retention requirements grow (add a remote long-term object store and more downsampling tiers); you go multi-cluster/region (federate with a global query layer); alert noise or misses appear (tune for-durations, thresholds, and add a dead-man's switch); or you need logs/traces correlation (integrate, but keep them as separate stores)."
/>

Four axes capture the real tension:

- **Pull vs push ingestion.** Pull gives service discovery and free target-health but needs reachability and misses short jobs; push covers short-lived/isolated workloads but loses the liveness signal. Most systems do both.
- **Compress/downsample vs keep raw.** Compression + downsampling make retention affordable at the cost of losing fine detail of old data; keeping everything raw is faithful but financially impossible. Decay resolution with age.
- **Bounded vs rich labels.** Bounded labels keep cardinality (and the system) alive; rich high-cardinality labels give more slicing but risk OOM. Bound labels; put identifiers in logs/traces.
- **Sensitive vs quiet alerting.** Short for-durations catch issues fast but page on noise; long ones are trustworthy but slower. Tune for signal, because a noisy alerting system gets ignored.

<Callout variant="interview" title="A 60-second end-to-end answer">
"A metrics system looks like storing numbers and graphing them, but it's really a write-firehose problem with a cardinality bomb inside it. Every monitored series emits a sample each scrape interval, so I'm ingesting on the order of twenty million samples a second continuously — about two hundred million active series scraped every ten seconds — and the ingestion path must never backpressure the producers, so it buffers through a write-ahead log and sheds load if overwhelmed rather than slowing anything down. That firehose would be twenty-eight terabytes a day raw, so the store is purpose-built for time series: samples hit an in-memory head block backed by the WAL, then flush to immutable, time-partitioned, compressed blocks. Compression is the whole game — because timestamps are evenly spaced and values change slowly, delta-of-delta timestamps plus XOR-encoded floats, the Gorilla scheme, shrink a sample from sixteen bytes to about two, roughly eight times, which is what makes keeping the data affordable. Then downsampling and tiered retention bound the long tail: recent data stays high-resolution, old data rolls up to one-minute, five-minute, hourly, and expires by age. The thing I never stop watching is cardinality: a time series is a metric plus its label set, so an unbounded label like a user or request ID explodes the series count and OOMs the store — cardinality, not sample volume, is what actually kills a metrics system, so labels stay bounded and I put identifiers in logs, not metrics."
