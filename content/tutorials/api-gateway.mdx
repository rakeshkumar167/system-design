<h2 id="interview-framing">Interview Framing</h2>

An API gateway is the single front door to a fleet of backend services. Instead of letting clients call dozens of microservices directly — each having to implement its own authentication, rate limiting, TLS, logging, and routing — you put one service at the edge that every request passes through, and it handles those **cross-cutting concerns** once, for everyone. Kong, NGINX, Envoy, AWS API Gateway, and Apigee are all this. From the outside it looks like the most ordinary thing in infrastructure — a reverse proxy that forwards a request to the right backend — and that framing produces a design that works fine until you notice it's now on the critical path of *every* request in the system, which makes it simultaneously the most trafficked, most latency-sensitive, and most dangerous-to-fail component you own.

The hard part is being a **transparent, low-latency, always-on control point**. Four problems fall out of "every request goes through it." First, **latency is a tax on the whole platform**: whatever the gateway does — verify a token, check a limit, match a route — it adds to every single call, so its overhead has to be single-digit milliseconds or it slows down everything at once. Second, it must be **stateless and horizontally scalable**: it holds no per-request state, so any instance can serve any request and you scale by adding boxes behind a load balancer — and shared state like rate-limit counters lives in a fast external store, not in the proxy. Third, you must **separate the control plane from the data plane**: the data plane forwards traffic at high throughput while a separate control plane manages the configuration — routes, API keys, rate-limit policies — and **pushes** it out, so an operator can change routing without redeploying the proxies. Fourth, because **if the gateway is down, everything is down**, it must be replicated, degrade gracefully (fail-open or fail-closed per concern), and protect the backends behind it with timeouts, retries, and circuit breakers. The reframe to lead with: *it looks like a reverse proxy, but the hard part is being a stateless, single-digit-millisecond control point on the path of every request — centralizing auth, routing, and rate limiting without becoming a latency tax or a single point of failure, which forces a stateless data plane that scales horizontally, a separate control plane that pushes config without redeploys, cached auth and limit state, and graceful degradation.*

Spend the opening minutes scoping. Good questions to ask:

- What does the gateway own versus the backends — auth, rate limiting, TLS, routing, transformation, all of it? This sets the filter chain.
- How many services and how much total traffic does it front? The gateway is sized for *aggregate* load, not one service.
- North-south (client-to-service) only, or also east-west (service-to-service)? East-west is usually a **service mesh**, a different shape.
- What's the latency budget the gateway is allowed to add? This decides how much you cache and how much work you do inline.
- How do auth and tokens work — JWTs the gateway can verify locally, or opaque tokens needing introspection? This drives the auth design.

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** the request lifecycle / filter chain, routing and service discovery, authentication and authorization at the edge, rate limiting and quotas, the control-plane/data-plane split, and resilience + observability — plus scaling, failure modes, and trade-offs. **Out of scope:** the backend services' own business logic (the gateway routes to them; what they do is theirs), building a full identity provider or OAuth authorization server (we *verify* tokens, we don't mint them), the service-mesh sidecar model beyond a comparison, GraphQL federation / response aggregation beyond a mention, and the developer portal — name them so the interviewer sees the boundary, then set them aside. Separate what the system must *do* from how *well* it must do it. The functional list for a gateway is a checklist of cross-cutting concerns; the difficulty, as always, lives in the non-functional column — and here it's dominated by latency and availability, because the gateway is in the path of everything.

Three targets shape every later decision. First, **low added latency**: the gateway runs on every request, so its own work must be a small fraction of total latency — which is why auth and rate-limit state are cached locally and the proxy path is kept lean. Second, **statelessness for horizontal scale and availability**: the gateway keeps no per-request state, so any instance serves any request, the fleet scales by adding replicas, and losing one instance loses nothing — the shared state it does need (rate counters, a token cache) lives in a fast external store. Third, **configurability without redeploys**: routes, keys, and policies change constantly, and you cannot redeploy the whole proxy fleet every time a team adds an endpoint — so a **control plane** owns the config and pushes it to the data plane live. Note what's *relaxed*: the gateway doesn't need its own durable storage on the request path (it's a pass-through), and it can tolerate *eventually* consistent config propagation (a new route appearing a second or two later across the fleet is fine). The system trades durable per-request state for speed, and strict config consistency for live updatability.

The capacity story: a gateway is hard not because any single operation is expensive, but because it must do cross-cutting work for everyone at almost no added latency — which is why it caches aggressively and stays stateless.

<h2 id="entity-model">Entity Model</h2>

Everything else — consumers, counters, registry entries — supports applying a Route to a request.

Three supporting entities complete the picture. A **Consumer** (an API key, client app, or authenticated principal) is the identity a request is attributed to — it carries the credential the gateway authenticates and the **plan/quota** it's rate-limited and authorized against. The **rate-limit counter** is the only mutable per-request state, and it deliberately lives *outside* the stateless gateway in a fast shared store (so every instance sees the same count). And the **service-registry entry** is how routing finds a live target: a list of healthy upstream instances for each service, kept current by health checks. The discipline: the control plane owns durable config, the data plane holds a cached copy and is otherwise stateless, and the one piece of shared mutable state (counters) is externalized.

The **control-plane (admin) API** is the gateway's *own* contract: how operators declare routes, consumers, and policies.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture separates the fast path (the data plane forwarding requests through a filter chain) from the slow path (the control plane managing and pushing config), with shared supporting services for auth, rate limiting, discovery, and telemetry.

<ApiGatewayArchitecture />

- **Client.** Every client of every fronted service; all traffic enters here.
- **Gateway (data plane).** The stateless proxy fleet. Each instance runs the **filter chain** — authN, authZ, rate-limit, transform, route — and forwards to an upstream, holding the route/policy config in memory. Scaled horizontally behind a load balancer.
- **Auth Service.** Verifies credentials at the edge: validates JWT signatures (locally) or introspects opaque tokens (a cached lookup). Offloads auth from every backend.
- **Rate Limiter.** A fast shared counter store so all gateway instances agree on per-consumer usage; the gateway pre-checks locally to avoid a round-trip per request.
- **Backends.** The upstream microservices the gateway routes to — the gateway is transparent to them on success.
- **Service Registry.** Discovery: the set of healthy upstream instances per service, kept current by health checks, used by routing and load balancing.
- **Control Plane + Config Store.** Owns routes, keys, and policies; validates and versions changes; **pushes** config to the data plane so it updates without a redeploy.
- **Telemetry.** The sink for structured logs, RED metrics, and distributed traces emitted for every request.

The decisive boundary is **a stateless data plane that forwards traffic fast, and a separate control plane that configures it live**. The data plane stays simple and quick because it just runs a filter chain over an in-memory config; the control plane carries the complexity of managing and safely rolling out that config; auth and rate-limit state are cached/externalized so the per-request overhead stays in single-digit milliseconds; and the registry, telemetry, and resilience logic let the gateway centralize cross-cutting concerns for every service at once. Every section that follows elaborates one piece: how a request flows through the chain (request lifecycle), how it's matched to a target (routing), how identity is verified (auth), how traffic is throttled (rate limiting), how config is distributed (control/data plane), and how backends are protected and observed (resilience & observability).

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: proxying a request on the happy path, rejecting a bad one at the edge, pushing new config to the data plane, and tripping a circuit breaker when an upstream fails. The first two are the request path; the last two are how the gateway is managed and how it protects what's behind it.

**Proxy a request.** The client sends a request with a token; the gateway runs the filter chain — verify the token (from cache), check the rate limit, match the route — then forwards to the upstream and relays the response, adding only a few milliseconds and a correlation id. This is the hot path the whole design optimizes.

<ProxyRequestSequence />

**Reject at the edge.** A request arrives with a missing or invalid token (or over its rate limit). The gateway's filter chain **short-circuits**: it returns 401 or 429 immediately, without ever touching a backend. Rejecting early is how the gateway shields upstreams from unauthenticated and abusive traffic.

<AuthRejectSequence />

**Push config.** An operator changes a route or policy through the admin API; the control plane validates it, persists a new **version**, and pushes it to the data-plane instances, which **hot-reload** it without dropping connections. Routing and policy change live — no proxy redeploy.

<ConfigPushSequence />

**Trip the circuit breaker.** Requests to an upstream start timing out or erroring; the gateway's failure counter crosses a threshold and the **circuit opens**, so subsequent requests **fail fast** (or return a fallback) instead of piling onto the sick backend; after a cooldown the gateway sends a trial request to see if it can **close** the circuit again. This protects both the failing service and the gateway's own resources.

<h2 id="request-lifecycle">Request Lifecycle & the Filter Chain</h2>

The heart of the data plane is the **filter chain** (a chain-of-responsibility / middleware pipeline): a request flows through an ordered sequence of filters, each doing one cross-cutting job, and any one of them can short-circuit the request. A typical order is **TLS termination → authentication → authorization → rate limiting → request transformation → route matching → forward to upstream → response transformation**, with observability hooks wrapping the whole thing. Filters split into **pre-filters** (run before the upstream call: auth, limits, routing) and **post-filters** (run on the response: header rewriting, metrics, caching). The design elegance is that each concern is an independent, composable filter — you add a new policy by inserting a filter, not by rewriting the proxy — and a route declares *which* filters apply.

**Ordering matters**, and getting it wrong is a classic bug. Authentication must come before authorization (you can't check permissions for an unknown caller) and before rate limiting (you rate-limit *a consumer*, so you must know who they are — and you don't want to spend a limit-check on an unauthenticated request). Cheap, decisive rejections should come early so the gateway does the *least* work for traffic it's going to refuse: a request that fails auth should be rejected before any routing, transformation, or backend call happens. This **short-circuiting** is what makes the gateway an effective shield — most abusive or malformed traffic is bounced at the first failing filter, never reaching an upstream. The principle: a request is a pipeline of small, ordered, independently-testable filters; put the cheap rejections first, know the caller before you authorize or limit them, and only touch the backend once the request has earned it.

<h2 id="routing-discovery">Routing & Service Discovery</h2>

Once a request passes the early filters, the gateway must **match it to an upstream**. Matching is on **host + path + method**, and because it happens on every request it has to be fast: gateways compile all routes into a **radix tree (trie)** on the path so a match is a few microseconds regardless of how many routes exist, rather than a linear scan through a rule list. A route can also carry predicates (headers, query params) and rewrite the path before forwarding. The output of matching is a target *service*, not a specific server — which is where discovery comes in.

**Service discovery** turns "the orders service" into "one of these healthy instances at these addresses." Backends scale and move, so the gateway can't hardcode IPs; instead a **service registry** (Consul, etcd, Kubernetes endpoints) holds the live set of instances per service, kept current by **health checks** that add new instances and evict failed ones. The gateway then **load-balances** across the healthy set — round-robin, least-connections, or latency-weighted — and stops sending traffic to instances that fail health checks. This same machinery enables progressive delivery: **canary** and **blue-green** routing send a fraction of traffic (or traffic matching a rule) to a new version, so rollouts are gradual and reversible at the edge. The principle: match fast with a compiled structure, resolve the target through a live registry rather than static addresses, and balance only across instances currently passing health checks — so routing automatically follows the fleet as it scales, heals, and deploys.

<h2 id="auth">Authentication & Authorization</h2>

Offloading **authentication** to the edge is one of the gateway's biggest wins: verify identity once, at the front door, so backends can trust the request and not each re-implement auth. There are two dominant token styles, and the distinction drives the design. A **JWT** is a signed, self-contained token: the gateway **verifies the signature locally** using the issuer's public key, reads the claims, and is done — **no network call**, so it's extremely fast, which matters on the critical path. The cost is **revocation**: because the token is self-contained and valid until it expires, revoking it early is hard (you need short lifetimes plus a revocation list). The alternative is an **opaque token** validated by **introspection** — the gateway calls the auth service to ask "is this token valid, and who is it?" This is **authoritative** (instant revocation, central control) but adds a lookup per request, so you **cache** the result for the token's short lifetime to keep the cost near zero (the capacity model's 95%-hit cache turning 10 ms into 0.5 ms).

Once identity is established, **authorization** decides whether this caller may do this. The gateway enforces coarse-grained authZ at the edge — **scopes** (an OAuth token granting `orders:read`), **roles**, or per-route ACLs — rejecting forbidden requests before they reach a backend, while fine-grained, data-dependent authorization ("can *this* user edit *that* order?") stays in the service that owns the data. The trade-off to state explicitly: **local JWT verification is fast but revocation-weak; introspection is authoritative but adds a cached lookup** — most systems use short-lived JWTs for speed and accept the revocation window, or introspect with caching when central revocation is required. Either way, the gateway turns "every service must implement auth" into "auth happens once at the edge," which is both faster and more secure because there's one place to get it right.

<h2 id="rate-limiting">Rate Limiting & Quotas</h2>

Rate limiting at the gateway serves two goals: **protecting backends** from overload and abuse, and **enforcing per-consumer plans** (the "gold tier gets 100k requests/day" quota). The algorithms are the ones from the **Rate Limiter** tutorial — **token bucket** for smooth rates with bursts, **sliding window** for accuracy — but the gateway adds a distributed twist: there are ~70 stateless gateway instances, and a consumer's requests can hit any of them, so the **counter must be shared**. The limit state lives in a fast central store (Redis-style) that all instances increment against, so the limit is enforced globally rather than per-instance (70 instances each allowing N would let through 70×N).

The tension is **accuracy versus latency**: a strict global counter means a round-trip to the shared store on *every* request, which is exactly the kind of per-request overhead the gateway must avoid. The common resolution is to **pre-check locally** and reconcile: each instance keeps a small local view (or a leased batch of tokens) and only consults the shared store periodically or when near the limit, trading a little precision for far less latency and load. A second decision is **fail-open vs fail-closed**: if the counter store is unavailable, do you let requests through (fail-open — favor availability, risk overload) or reject them (fail-closed — favor protection, risk an outage)? For most gateways rate limiting **fails open** — a brief inability to count is better than dropping all traffic — while security-critical limits may fail closed. The principle: enforce limits on a shared counter so the stateless fleet agrees, pre-check locally to keep the per-request cost down, and decide deliberately whether each limit fails open or closed when its store is down. (This is the Rate Limiter tutorial's machinery, deployed at the edge for every service at once.)

<h2 id="control-data-plane">Control Plane & Data Plane</h2>

This is the architectural centerpiece: separating the **data plane** from the **control plane**. The **data plane** is the fleet of proxies that actually handles traffic — it runs the filter chain, matches routes, and forwards requests, optimized for raw throughput and low latency, holding all its configuration **in memory**. The **control plane** is the management brain — it owns the routes, API keys, rate-limit policies, and upstream definitions, exposes the admin API, **validates** changes, and **distributes** the resulting config to every data-plane instance. The two are deliberately decoupled: the data plane can keep serving even if the control plane is down (it runs on its last-known config), and the control plane can be a richer, slower, more careful system because it's off the request path.

The reason this split exists is **config changes without redeploying the data plane**. Teams add endpoints, rotate keys, and tune limits constantly; you cannot redeploy ~70 proxy instances every time. Instead the control plane **pushes** versioned config to the data plane, which **hot-reloads** it — updating routing and policy live, with no dropped connections. Two design choices matter here. **Push vs poll**: the control plane can push updates to instances (fast propagation, more coordination) or instances can poll for the latest version (simpler, slightly laggier) — most systems push for speed with a poll as a fallback. And **versioning + safe rollout**: every config change is a new immutable version, validated before distribution and **rolled out gradually** (canary the new config to a few instances first), so a bad route can be caught and **rolled back to the previous version** instantly rather than breaking the whole fleet. The principle: keep the data plane dumb, fast, and stateless-but-for-cached-config; put all the management intelligence in a control plane that validates and versions changes and pushes them out — so the system is both blazing fast on the request path and safely, continuously reconfigurable off it.

<h2 id="resilience-observability">Resilience & Observability</h2>

Because the gateway sits in front of every backend, it's the right place to **protect** them and to **see** what's happening. Protection is a small toolkit. **Timeouts** cap how long the gateway waits on an upstream, so a slow service doesn't tie up gateway resources indefinitely. **Retries** paper over transient blips, but only with a **budget** — retrying everything amplifies load on an already-struggling backend, so you cap retries (often "retry at most once," only on idempotent methods, with backoff). **Circuit breakers** watch an upstream's failure rate and, when it crosses a threshold, **open** — failing fast (or serving a fallback) instead of sending more traffic to a sick service, then probing after a cooldown to close again. **Bulkheading** isolates resources per upstream (separate connection pools), so one failing backend can't exhaust the gateway and take down healthy routes with it. Together these make the gateway a shock absorber: a degraded backend degrades *its* routes, not the whole platform.

**Observability** is the other half, and the gateway is the one place that sees every request, so it's where you instrument. Three pillars: **structured logs** (one record per request with route, consumer, status, latency), the **RED metrics** (Rate, Errors, Duration — per route and per upstream, the vital signs of the platform), and **distributed tracing** — the gateway injects a **correlation/trace id** at the edge and propagates it downstream, so a single request can be followed across every service it touches. This is enormously valuable precisely because it's centralized: instead of stitching together logs from dozens of services, you get a uniform, platform-wide view of traffic, errors, and latency at the front door, and the trace id ties the distributed call graph back to one originating request. The principle: the gateway is the natural home for both protecting backends (timeouts, budgeted retries, circuit breakers, bulkheads) and observing them (RED metrics, structured logs, edge-injected traces), because every request already flows through it.

<h2 id="scalability-evolution">Scalability & Evolution</h2>

Don't build the multi-region, control-plane-driven gateway on day one. Each stage answers a specific pressure and adds a specific cost — name the trigger before taking on the complexity.

<TradeoffTable
  columns={["Stage", "Trigger", "Cost"]}
  rows={[
    ["Per-service libraries (no gateway)", "A few services, simple needs", "Every service re-implements auth/limits/TLS; inconsistent, hard to change globally"],
    ["Single gateway instance", "Want one front door for cross-cutting concerns", "A single point of failure and a throughput bottleneck"],
    ["Stateless replicated data plane + control plane", "Real traffic; config must change without redeploys", "Run a control plane; externalize counters/auth cache; manage config rollout"],
    ["Multi-region edge gateways (+ mesh for east-west)", "Global low latency; service-to-service scale", "Regional config distribution; a service mesh for internal traffic; more operational surface"],
  ]}
  recommendedRow={2}
/>

The progression tracks the pain you're buying out of. Starting with **per-service libraries** is fine for a handful of services but means every team re-implements auth and rate limiting inconsistently, and changing a global policy means changing N codebases. A **single gateway** centralizes that but is an obvious single point of failure and bottleneck. The center of gravity for most platforms is a **stateless, replicated data plane behind a load balancer, driven by a control plane**, with counters and the auth cache externalized — this is where the gateway becomes highly available, horizontally scalable, and reconfigurable without redeploys. Only global products need **multi-region edge gateways**, which pay for locality with regional config distribution, and they often pair the north-south gateway with a **service mesh** for east-west (service-to-service) traffic — a related but distinct shape (sidecars instead of a central edge). Most systems are well-served by "stateless data plane, control-plane-managed config, externalized shared state."

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. The gateway's guiding rules: **never become the single point of failure it was meant to prevent**, and **degrade gracefully** — a partial failure should narrow the blast radius, not widen it.

The theme: **statelessness makes the gateway itself trivially recoverable, and graceful degradation keeps a partial failure partial.** A dead instance is just removed from the pool and replaced — no state lost — because the gateway holds none. A control-plane outage costs *configurability*, not *availability*: the data plane runs on its cached config. Dependency failures are met with deliberate fail-open/fail-closed policies (serve cached auth, approximate limits) so the gateway keeps working in a degraded but safe mode. A sick upstream is contained by timeouts, circuit breakers, and bulkheads so it can't take down healthy routes. And the gateway's own riskiest action — a config change — is made safe by validation, canarying, and versioned rollback. Because the data plane is stateless and runs on cached config, recovery is almost always "replace the instance" or "roll back the config," and the design's whole point — one front door — is protected by making that front door a replicated, gracefully-degrading fleet rather than a single box.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Put a stateless, horizontally-scaled data plane at the edge that runs an ordered filter chain (authN → authZ → rate-limit → transform → route → forward) over in-memory config, with shared state (rate counters, token cache) externalized; manage it with a separate control plane that validates, versions, and pushes config so routing and policy change without redeploys; offload auth to the edge (local JWT verification or cached introspection); rate-limit on a shared counter with local pre-checks; and protect backends with timeouts, budgeted retries, and circuit breakers while emitting RED metrics, logs, and edge-injected traces."
  rationale="The gateway is on the critical path of every request, so its overhead must be minimal and it must never be a single point of failure — which forces statelessness (horizontal scale, trivial recovery) and aggressive caching of auth/limit state. The control/data-plane split is what lets a fast, dumb data plane be safely and continuously reconfigured by a richer control plane without redeploys. Filter ordering and short-circuiting make the gateway an efficient shield. Circuit breakers and bulkheads keep one bad backend from cascading. And because every request flows through it, the gateway is the one place to centralize cross-cutting concerns and observability."
  alternatives="Per-service libraries (no gateway) — no extra hop, but every service re-implements auth/limits inconsistently and global changes are N deploys; a monolithic gateway with config baked in — simpler but every change is a redeploy and there's no safe rollout; a single instance — no horizontal scale and a single point of failure; strict per-request global rate-counter checks — exact but adds a round-trip to every call; a service mesh instead of a gateway — great for east-west service-to-service traffic but not the north-south client edge (often used together); failing closed everywhere — safe but turns a dependency blip into a full outage versus deliberate fail-open for low-risk concerns."
  revisitWhen="East-west (service-to-service) traffic dominates (add a service mesh), global users need regional edges (multi-region config distribution), a single gateway team becomes a bottleneck for many services (federated/per-team gateways), or extreme latency requirements make even a few milliseconds of edge overhead unacceptable (push concerns into the mesh/sidecar or the client)."
/>

Four axes capture the real tension:

- **Centralized gateway vs per-service libraries.** A gateway centralizes cross-cutting concerns (consistent, one place to change) at the cost of an extra hop and a critical shared component; libraries avoid the hop but scatter and duplicate the logic. At more than a few services, the gateway wins.
- **Control/data-plane split vs monolithic gateway.** The split enables live, safe reconfiguration and a fast data plane, at the cost of running a control plane; a monolith is simpler but every change is a redeploy. Split once config changes often.
- **Local JWT verification vs token introspection.** Local is fast (no network call) but revocation-weak; introspection is authoritative but adds a cached lookup. Short-lived JWTs for speed, introspection-with-cache when central revocation matters.
- **Fail-open vs fail-closed.** When a dependency (auth, counter store) is down, fail-open favors availability and fail-closed favors protection. Choose per concern — usually open for low-risk limits, closed for security-critical checks.
