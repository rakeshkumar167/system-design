<h2 id="interview-framing">Interview Framing</h2>

A collaborative document editor lets many people open the same document and type into it at the same time — seeing each other's changes and cursors live — and somehow everyone ends up looking at the exact same document. A user types a character and it appears instantly; a collaborator on another continent sees it a moment later; two people editing the same sentence at once both keep their words. The trap is to picture this as a shared text field behind a websocket: broadcast each keystroke, apply it everywhere, done. The real problem is what happens when edits **collide** — and at human typing speed across many editors, they collide constantly.

The hard part is **conflict-free convergence**. Because a keystroke must show up locally with zero latency, you cannot route every character to a server and wait for a verdict — each editor applies edits to its **own local replica immediately**. But that means replicas drift: while your "hello" is in flight, a collaborator inserted a word ahead of it, so your edit's position is already wrong by the time it lands. The system has to reconcile these concurrent, optimistically-applied edits so that **every replica converges to an identical final document** and **no one's edit is silently lost**. The two canonical answers are **operational transformation (OT)** — a server orders edits and transforms each against the ones it didn't see — and **conflict-free replicated data types (CRDTs)** — edits are made commutative so they merge in any order. The reframe to lead with: *it looks like broadcasting keystrokes, but the hard part is making concurrent, offline-tolerant edits converge without lost updates — OT or CRDTs — over a real-time fan-out fabric.*

Spend the opening minutes scoping. Good questions to ask:

- How many concurrent editors per document — a handful, or a viral doc with hundreds? (It changes the fan-out and ordering story.)
- Plain text or rich text (formatting, tables, embeds)? We'll design for text and note rich text as harder.
- Must edits survive going offline and reconnecting, or only brief network blips?
- Is there a central server we trust to order edits, or must this work peer-to-peer?
- Do we need full version history and undo, or just the live document?

<Callout variant="interview" title="Suggested 45-minute allocation">
**5 min** framing and the convergence reframe · **5 min** requirements and the two guarantees (convergence, intention preservation) · **5 min** capacity (the fan-out and connection numbers fall out here) · **5 min** entity model and the op/websocket API · **10 min** the concurrent-editing problem and OT vs CRDTs — the centerpiece · **10 min** real-time sync, presence, and persistence · **5 min** failure modes and wrap-up. Say your assumptions out loud — the interviewer is grading whether you understand why naive broadcast loses edits and can compare OT and CRDTs, not whether you can recite a transform function.
</Callout>

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** the concurrent-editing problem, operational transformation, CRDTs, the real-time sync protocol (websockets, optimistic apply, broadcast, reconnection), presence/awareness, and op-log persistence with history. **Out of scope:** rich-text schema details, the editor UI and rendering, document search, the sharing/permission system (we note access control but don't design it), deep offline-first mobile sync, and in-document calls — name them so the interviewer sees the boundary, then set them aside.

Two guarantees dominate every later decision. First, **convergence**: once all edits have propagated, every replica must show the *exact same* document — if two people's screens permanently disagree, the product is broken, full stop. Second, **intention preservation**: a concurrent edit must not silently destroy another — last-write-wins on a shared paragraph would quietly delete someone's sentence, which is unacceptable for a document. These two properties are precisely why naive broadcast fails and why OT and CRDTs exist. The third pillar is **instant local editing**: typing must never wait for a server, so edits are applied optimistically to the local replica and reconciled afterward — which is the very thing that creates concurrency in the first place.

<h2 id="capacity-estimates">Capacity Estimates</h2>

Estimates here tell a different story than a storage or streaming system's. The data is tiny — an edit is ~100 bytes — so the constraint isn't bandwidth or disk. It's **fan-out** (every keystroke multiplies across collaborators) and **persistent connections** (millions of open websockets). We size all of it from one set of assumptions.

The takeaways. First, **the load is messages, not bytes**: 4 million operations per second arrive, but each is rebroadcast to the other collaborators on its document, producing **12 million outbound messages per second** — fan-out, not data volume, is the dominant real-time cost, and it's why the system is built around efficient message routing. Second, **connection state is the scaling unit**: the platform holds **2 million persistent websocket connections** open at peak, each consuming memory and needing a home server — so you scale by sharding connections and documents across many servers, not by adding CPU. Third, **the op log grows relentlessly**: ten billion tiny operations a day is only ~**1 TB/day**, but it never stops, so you cannot replay from the beginning of time — periodic **snapshots** plus **log compaction** are mandatory to keep document load times and storage bounded. The shape is the inverse of a content system: minuscule payloads, enormous message and connection counts.

<h2 id="entity-model">Entity Model</h2>

The model centers on the **Document** as the thing users share, but the real unit of work is the **Operation** — the small, ordered edit that the whole convergence story is about. The document's durable truth is not a blob of text; it's the **ordered log of operations** that produced it, periodically materialized into a snapshot.

Three related entities complete the picture. An **Operation** is the unit of change — `(doc_id, version, author_id, type: insert | delete, position, payload)` — and the append-only sequence of operations *is* the document's source of truth; everything else is derived. A **Snapshot** is the materialized document text at a particular version, written periodically so a newly-joining client (or a recovering server) can load the snapshot and replay only the operations since, rather than the entire history. **Presence** is a per-user, in-memory record of a cursor position and selection — deliberately *not* persisted, because it's ephemeral and a lost cursor frame simply self-corrects on the next update.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture routes every editor of one document to a single ordering authority, persists operations durably, and fans them out — while keeping ephemeral presence off the durable path.

<CollaborativeDocEditorArchitecture />

- **Editor (client).** Holds a local replica of the document and applies edits optimistically. It sends operations and presence over a persistent websocket and applies inbound operations from collaborators.
- **WS gateway.** Terminates the millions of persistent websocket connections and routes each editor of a given document to the same collaboration server (sticky by `doc_id`). It's the connection-management tier — the thing that makes two million open sockets survivable.
- **Collaboration service (document session).** The ordering authority for a document: it assigns each operation the next version, transforms or merges concurrent operations (OT/CRDT), appends to the op log, and broadcasts to collaborators. One document's live session lives on one server so its order is unambiguous.
- **Op log.** The append-only source of truth — every operation, in order. The document is, by definition, the replay of this log; nothing is acknowledged until it's durably here.
- **Snapshot store.** Periodic materialized snapshots of the document, so joining clients and recovering servers replay only recent operations instead of all history.
- **Pub/sub.** The fan-out bus that delivers a document's operations to every gateway holding one of its editors, so a doc whose collaborators span many connection servers still converges.
- **Presence service.** Carries ephemeral cursors and selections — high-frequency, lossy, and deliberately never written to the op log.
- **Metadata DB.** Document records, versions, and access-control data.

The decisive boundary is **one ordering authority per document, with a durable log behind it**. Sticky routing guarantees a document's edits are serialized in one place; the op log makes that order durable and replayable; pub/sub spreads it to wherever the collaborators are connected; and presence is kept entirely off that durable path because it doesn't need to be right forever — only right *now*. Every section that follows elaborates how that authoritative order stays convergent under concurrent, optimistic edits.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: a normal edit broadcasting to collaborators, two concurrent edits being reconciled, a presence update, and a client catching up after reconnecting. The conflict-resolution flow is the one that proves the convergence guarantee.

**Edit & broadcast.** An editor applies a keystroke to its local replica instantly, then sends the operation. The collaboration service assigns it the next version, appends it to the log, acks the originator, and broadcasts it to the other editors, who apply it.

<EditBroadcastSequence />

**Conflict resolution.** Two editors change the same region at the same instant, each against a slightly stale view. The collaboration service orders them and **transforms** the later one against the earlier (OT) — or merges them by element id (CRDT) — so both edits survive and every replica ends up identical.

<ConflictResolutionSequence />

**Presence.** An editor moves its cursor and sends a presence update on the ephemeral channel; the presence service fans it out to collaborators. If a frame is dropped, the next one corrects it — presence never touches the durable log.

<PresenceSequence />

**Reconnect & sync.** An editor's connection drops and comes back. It reconnects with the last version it saw, the service reads the missed operations from the log since that version, sends the delta, and the client catches up and resumes live editing.

<h2 id="concurrent-editing">The Concurrent-Editing Problem</h2>

Everything hard about this system lives in one moment: two people edit the same place at the same time. To see why it's hard, watch the naive designs fail. **Last-write-wins** — whoever's edit arrives second overwrites the first — is catastrophic for a document: it silently deletes the loser's sentence, violating intention preservation. **Locking** the document (or a paragraph) while someone types restores correctness but destroys the product: collaboration that makes you wait for a lock isn't real-time collaboration. And **naive broadcast** — just apply every operation everywhere in arrival order — breaks because operations carry **positions**, and positions shift under concurrent edits.

That position problem is the crux. Say the document is `"hat"` and two editors act at once: A inserts `"c"` at position 0 (intending `"chat"`), and B deletes the character at position 2 (the `"t"`, intending `"ha"`). Each applies its own edit locally and ships it. Now B receives A's "insert `c` at 0" and applies it — fine, it shifts everything right. But A receives B's "delete at position 2" and applies it literally — except after A's own insert, position 2 is no longer the `"t"`, it's the `"a"`. A deletes the wrong character, and the two replicas now **permanently disagree**. No edit was lost to a crude overwrite, yet convergence still failed, because an operation's position was interpreted against a document that had changed underneath it.

<Callout variant="info" title="The two properties to guarantee">
Every correct design here must deliver two things. **Convergence:** after all operations propagate, all replicas are byte-for-byte identical — no permanent disagreement like the `"hat"` example. **Intention preservation:** each edit's *meaning* survives — A's insert and B's delete both take effect on the characters they meant, not whichever character happens to sit at a now-stale index. Last-write-wins fails the second; naive broadcast fails the first. Operational transformation and CRDTs are the two families of solution that deliver both, and the rest of this tutorial is about how each one does it.
</Callout>

<h2 id="operational-transformation">Operational Transformation</h2>

Operational transformation is the first family of solution, and the one classic web editors were built on. The idea: keep editing as a stream of **operations**, route them through a **central server that assigns a total order**, and whenever an operation arrives that was generated concurrently with operations the recipient has already applied, **transform** it so it accounts for them. Transformation rewrites an operation's parameters — usually its position — so that applying it to the already-changed document achieves the originator's original intent.

Return to `"hat"`. A inserts `"c"` at 0; B deletes at 2. The server picks an order — say A first — and when B's "delete at 2" is delivered to anyone who has already applied A's insert, the transform shifts its position from 2 to **3**, because A's insertion of one character to the left pushed every later index right by one. Now the delete removes the intended `"t"` everywhere, A's `"c"` is preserved everywhere, and all replicas converge to `"chat"` minus... `"chat"` → delete the `t` → `"cha"`. Every replica reaches `"cha"`. The transform function is what encodes "an insert of length 1 before your position bumps your position by 1," and OT is essentially a careful library of these transform rules for every pair of operation types.

OT's strengths are real: operations are **compact** (a position and a few characters), the model maps naturally to a **server-authoritative** design where one ordering authority already exists, and the document stays a plain string with no per-character bookkeeping. Its weakness is equally real: the **transform functions are subtle and easy to get wrong** — the number of operation-pair cases grows quickly, correctness proofs are famously tricky, and bugs manifest as rare, hard-to-reproduce divergence. This is why OT implementations lean on a central server to bound the ordering, rather than attempting arbitrary peer-to-peer transformation.

<h2 id="crdts">Conflict-Free Replicated Data Types</h2>

CRDTs are the second family, and they attack the problem from the opposite direction: instead of transforming operations to fit a changing document, **make operations commutative so order never matters**. The trick for text is to stop identifying characters by position and start identifying them by a **unique, immutable id** with a total order built in. Each character gets an id that encodes where it sits *between its neighbors* (and a tiebreaker like a site id), so an insert says "add character `c` with id between id-of-`h`-start and id-of-`a`" rather than "insert at index 0." A delete references the **id** of the character to remove, not its index.

Because every operation names stable ids rather than shifting indices, **applying them in any order yields the same document** — they commute. Two concurrent inserts at "the same place" get different ids (their site tiebreakers differ), so they don't collide; they just both land, in a deterministic order. There's no transform function and **no central authority required for correctness**: replicas can exchange operations peer-to-peer, in any order, even after a long offline stretch, and still converge by construction. That property — convergence without coordination — is why CRDTs shine for offline-first and decentralized editing.

<Callout variant="warning" title="CRDTs trade transform-complexity for metadata overhead">
CRDTs remove the subtle transform functions, but they aren't free. Every character carries a unique id (and ordering metadata), which is far larger than the character itself, and a deleted character usually can't just vanish — it becomes a **tombstone** that lingers so that concurrent operations still referencing it remain valid. Over a long-lived, heavily-edited document, ids and tombstones accumulate, so practical CRDTs need **garbage collection / compaction** to reclaim space. The trade is clear: OT keeps the data tiny but the algorithm subtle and server-bound; CRDTs keep the algorithm simple and coordination-free but pay in per-element metadata. Many modern editors choose a sequence CRDT precisely because coordination-free convergence is easier to reason about at scale.
</Callout>

<h2 id="realtime-sync">Real-Time Sync</h2>

With convergence handled by OT or CRDTs, the sync layer is what makes editing feel instant and stay live. Three mechanisms carry it. First, **optimistic local apply**: a keystroke is applied to the local replica immediately and only *then* sent, tagged with the version it was based on and a client sequence number, so the typist never waits a round trip. Second, **authoritative ordering and broadcast**: the operation reaches the document's collaboration server, which assigns the next canonical version, persists it, acks the originator (who reconciles its optimistic state with the confirmed order), and broadcasts the operation — transformed or merged — to every other editor. Third, **catch-up on reconnect**: networks blip, so the client tracks the last version it has, and when it reconnects it asks for everything **since** that version, replays the delta onto its replica, and resumes.

Routing is what makes this scale. Because one document needs **one ordering authority**, all of its editors must land on the **same collaboration server** — so the WS gateway routes stickily by `doc_id`. That sharding-by-document is the key scaling move: a million documents spread across many servers, each server owning the ordering for its slice, and pub/sub bridging the case where a document's editors are connected to different gateway nodes. No global coordination is needed because no operation ever spans two documents.

<h2 id="presence-awareness">Presence & Awareness</h2>

Presence — the live cursors, selections, and "who's here" avatars — is what makes collaboration feel human, and it has the *opposite* engineering profile from editing. It is **high-frequency** (a cursor can move many times a second), **ephemeral** (only the current position matters; history is meaningless), and **loss-tolerant** (a dropped cursor frame is invisible because the next one supersedes it). So it gets its own treatment: presence updates travel on a **separate, non-durable channel**, are fanned out best-effort, and are **never written to the op log**.

Keeping presence off the durable path is a deliberate and important separation. If cursor moves went through the same ordered, persisted pipeline as edits, you'd multiply op-log volume by a huge factor for data that's worthless a half-second later — and you'd slow the one path that must be correct (edits) with a flood that's allowed to be wrong. Treating presence as disposable lets it be cheap and fast: store the latest position per user in memory, broadcast on change, expire it when the user disconnects, and let any loss self-heal on the next update. It's the same instinct as the Video Streaming tutorial keeping the approximate seat-map— here, awareness is approximate by design while the document is exact.

<h2 id="persistence-history">Persistence & History</h2>

The durable heart of the system is the **op log**: the append-only, ordered sequence of every operation applied to a document. It's the source of truth in the strongest sense — the document *is* the replay of its log — and nothing is acknowledged to a client until its operation is safely appended, so an "saved" edit can never be lost. This log-as-truth model is also what makes history and undo almost free: the ordered operations already *are* a complete, replayable timeline, so "the document at 3pm" is just a replay up to that point, and undo is applying an inverse operation.

But an append-only log that only grows is a replay-time and storage problem, which is why **snapshots and compaction** are mandatory rather than optional. Periodically the service materializes the document's current text into a **snapshot** at a known version; a client joining a long-lived document then loads the snapshot and replays only the operations *since* it, instead of ten billion operations from the dawn of the doc. **Compaction** prunes operations already captured by a snapshot (and, for CRDTs, garbage-collects tombstones), keeping both storage growth and load latency bounded. The capacity section's relentless ~1 TB/day is exactly the pressure this answers: the log grows forever, so you continuously fold its old prefix into snapshots and discard what they already cover.

The theme: **the op log is the safety net, and divergence is the enemy.** Because every acknowledged edit is durable in the log and the log is canonical, most failures degrade to *delay* (catch up later) rather than *loss* — a crashed session rehydrates, a dropped client reconnects and replays, fan-out lag self-heals. The one failure to design hardest against is the **split session** (two ordering authorities for one document), because that's how silent divergence happens, and it's prevented by a single-writer lease per document with fencing on failover.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Apply edits optimistically on the local replica, route each document's editors to a single collaboration server that assigns a canonical operation order and reconciles concurrency with operational transformation (or a sequence CRDT), persist every operation in an append-only op log bounded by periodic snapshots and compaction, fan out over websockets plus pub/sub, and carry ephemeral presence on a separate non-durable channel."
  rationale="Optimistic local apply gives instant typing, which is the product, and it's exactly what creates the concurrency that OT or CRDTs then make convergent and intention-preserving. A single ordering authority per document makes the order unambiguous and convergence tractable, while sharding by document scales it horizontally with no global coordination. The op log makes edits durable and turns history and undo into replay, and keeping presence off that log keeps the durable path cheap and fast."
  alternatives="Last-write-wins — trivial but silently destroys concurrent edits, disqualified; document/paragraph locking — correct but kills real-time collaboration; pure peer-to-peer OT — no central order makes transforms intractable; a sequence CRDT instead of OT — coordination-free and offline-friendly but pays per-element id/tombstone overhead; full state sync instead of operations — simpler protocol but enormous bandwidth for large docs."
  revisitWhen="The product needs genuine offline-first or decentralized editing (favor CRDTs), documents become large/rich enough that operation granularity or metadata overhead matters, concurrent-editor counts per doc spike (rethinking fan-out and ordering load), or a single authoritative region no longer meets latency/residency needs."
/>

Three axes capture the real tension:

- **OT vs. CRDT.** OT keeps data compact and maps cleanly to a server-authoritative order, but its transform functions are subtle and bug-prone; CRDTs make convergence coordination-free and offline-friendly at the cost of per-element id and tombstone metadata. Default to whichever your team can reason about — many modern editors pick a sequence CRDT for its simpler mental model.
- **Server-authoritative vs. peer-to-peer.** A central ordering authority makes convergence and persistence straightforward; peer-to-peer removes the central dependency but needs CRDTs and complicates durability and history. We default server-authoritative and reach for P2P only when decentralization is a hard requirement.
- **Snapshot cadence.** Frequent snapshots keep replay short and joins fast but cost more write work and storage churn; infrequent snapshots are cheaper but lengthen replay and recovery. Tune it to document size and how long-lived and heavily-edited the docs are.

<h2 id="interview-summary">Interview Summary</h2>

<Callout variant="interview" title="A 60-second end-to-end answer">
"A collaborative document editor looks like broadcasting keystrokes, but the hard part is making concurrent edits converge without losing anyone's work. Every edit is applied to the local replica instantly — typing can't wait a round trip — which means replicas drift, so I need two guarantees: convergence, all replicas end identical, and intention preservation, no edit silently lost. Naive broadcast fails because operations carry positions and concurrent edits shift them, so an op lands at the wrong index and replicas diverge. The two fixes are operational transformation, where a central server assigns a total order and transforms each op against the ones it didn't see, and CRDTs, where every character gets a unique id so operations commute and merge in any order with no central transform — OT keeps data small but the algorithm subtle, CRDTs keep the algorithm simple but pay in id and tombstone metadata. Around that, the sync layer routes all of a document's editors to one ordering server, sharded by doc id, appends every op to a durable op log that's the source of truth, snapshots and compacts it to bound replay, and fans out over websockets and pub/sub. Presence — cursors — rides a separate ephemeral channel and never touches the log. The load is fan-out and millions of persistent connections, not bytes, and the enemy is silent divergence, which a single-writer-per-doc lease prevents."
</Callout>
