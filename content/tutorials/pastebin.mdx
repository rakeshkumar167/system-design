<h2 id="interview-framing">Interview Framing</h2>

A Pastebin takes a blob of text — a stack trace, a config file, a snippet of code — stores it, and hands back a short, shareable URL like `pst.io/aZ4kP9`. Anyone with the link reads the content back; the paste may be public or private, and it usually carries an expiry so it doesn't live forever. It is the canonical way to share text that is too long to drop into a chat and too transient to deserve a Git repo.

At a glance it looks like a URL shortener — short id in, content out, read-heavy — and a weak candidate will spend the interview re-deriving key generation. The reframe that earns points: a URL shortener stores a *tiny* string (a URL) and the hard part is generating keys; a Pastebin stores a *large* blob (kilobytes to megabytes) that **expires** and has **privacy**, and the hard parts are where that blob lives, how it dies on time, and who is allowed to read it. Spend the opening minutes establishing scope. Good questions to ask:

- How big can a paste be — a few KB of logs, or 10 MB of data? What's the cap?
- Do pastes expire? Is there a one-time "burn after reading" mode?
- What privacy levels — public, unlisted (link-only), private, password-protected?
- Read:write ratio and scale — how many pastes created and read per day?
- Do we render/syntax-highlight content, or just serve raw text?

<Callout variant="interview" title="Suggested 45-minute allocation">
**5 min** framing and scope · **5 min** requirements and the size cap · **5 min** capacity (the blob-vs-metadata split falls out here) · **5 min** entity model and API · **10 min** architecture, storage, and the read path through the CDN · **10 min** expiry, privacy, and failure modes · **5 min** wrap-up. Say your assumptions out loud — the interviewer is grading whether you separate large immutable content from small mutable metadata and how you reason about expiry, not whether you can invent a base-62 encoder.
</Callout>

This tutorial commits to concrete assumptions so the numbers stay consistent end to end. **In scope:** create / read / raw-fetch APIs, the blob-plus-metadata storage split, expiry and TTL (lazy and active deletion, one-time pastes), CDN delivery of immutable content, and access control (public / unlisted / private / password). **Out of scope:** real-time collaborative editing (that's the Collaborative Document Editor problem), the internals of syntax highlighting, full-text search across all pastes, and account/billing productization — name them so the interviewer sees the boundary, then set them aside.

Separate what the service must *do* from how *well* it must do it. For a Pastebin the load-bearing non-functional targets are read latency (it's read-heavy and globally shared), content durability (losing a paste is losing user data), and a hard size cap (it bounds every capacity number that follows).

Three targets shape every later decision. First, **read latency at p99 must stay under ~100 ms globally**, and since paste content never changes after it's written, the way we hit that is a CDN: serve immutable content from the edge so most reads never reach the origin. Second, **content durability is non-negotiable** — a paste is user data, so content belongs in object storage engineered for eleven nines of durability, not on a service node's local disk. Third, the **10 MB size cap** is not a detail; it bounds the largest single object, the upload path, the per-request memory, and — multiplied by volume — the entire storage estimate. Pin it down early.

Estimates are directional — their job is to reveal *which* resource is under pressure. For a Pastebin there are two stories: a modest **request rate** (writes are light, reads are ~10× heavier) and a **storage split** where content and metadata live on wildly different scales. We size both from one set of assumptions.

The takeaways. First, the **request rate is small**: thirty million pastes a month is only about **12 writes per second** on average and **~116 reads per second**, peaking near **580 reads per second** — numbers a single well-provisioned tier handles, and that a CDN shaves down to a trickle at the origin. This is not a QPS problem. Second, the **storage split is the whole story**: 720 million retained pastes hold about **7.2 TB of content** but only **~288 GB of metadata** — content outweighs metadata by ~25× *even at a tiny 10 KB average*, and real pastes (logs, dumps) push that ratio toward 1000×. That gap is the design: **content goes to object storage; metadata — a pointer plus a few attributes — goes to a database.** Third, because content dominates and is immutable, the read path is a **caching problem, not a scaling problem** — the CDN, not the origin, carries peak reads.

The entity model encodes the central decision: a paste is split into a small **metadata** record in a database and a large **content blob** in object storage, joined by a pointer. The metadata row never holds the bytes — it holds a `blob_key` that locates them.

The id doubles as the access token for unlisted pastes, so it must be **long and random enough to be unguessable** — a short sequential counter would let anyone enumerate every "link-only" paste. Everything the read path needs to *authorize and expire* a request — visibility, password hash, `expires_at`, `one_time` — lives in this small row, so those checks are one fast metadata lookup, never a fetch of the multi-megabyte blob.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture is shaped entirely by the blob-vs-metadata split and the read-heavy, immutable workload: a **CDN** in front of reads, stateless **app servers** holding no content, a **metadata database** for the small rows, **object storage** for the large blobs, and an **expiry worker** that reaps dead pastes out of band.

<PastebinArchitecture />

- **CDN.** The front door for reads. Because raw content is immutable, the edge caches it with a long TTL keyed by id, so the vast majority of reads are served near the user and never reach the origin — this is what buys the sub-100 ms p99 globally.
- **App / API servers.** Stateless and horizontally scalable. On create they enforce the size cap, write the blob, then commit metadata; on a cache miss they authorize the request from metadata, fetch the blob, and respond. They hold no durable state, so any instance serves any request.
- **Metadata database.** The small, queryable source of truth: id → blob pointer plus visibility, expiry, and ownership. Partitioned by id; the only thing the authorize-and-expire checks consult.
- **Object storage.** The durable home for content blobs (eleven nines). It holds the bytes and nothing about access logic; a lifecycle policy can even expire objects on its own.
- **Expiry worker.** An out-of-band sweeper that finds pastes past `expires_at` and deletes both the metadata row and the blob, so expired storage is actually reclaimed rather than merely hidden.

The decisive boundary is **small mutable metadata versus large immutable content**. Keeping them in different stores lets each scale on its own axis — the database stays small and fast for point lookups and expiry queries, while object storage absorbs terabytes of blobs cheaply — and lets the CDN treat content as the immutable artifact it is. Every section that follows is a consequence of that split.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: creating a paste, the common cached read, the cache-miss read, and expiry. Trace each so the ordering — especially "blob before metadata" — is explicit.

**Create.** The server enforces the size cap, writes the content to object storage to get a `blob_key`, then commits the metadata row pointing at it. Blob-before-metadata means a committed row never dangles; a blob with no row is just unreferenced garbage the sweeper collects.

**Read — cache hit.** The overwhelmingly common case. The CDN has the immutable raw content at the edge and returns it directly; the origin, the database, and object storage are never touched.

**Read — cache miss.** On a miss the request reaches an app server, which first authorizes and expiry-checks the paste from its small metadata row, then fetches the blob from object storage, returns it, and lets the CDN backfill with a long TTL so the next reader hits the edge.

<h2 id="blob-metadata-split">Blob & Metadata Storage</h2>

This is the core structural decision, so make the reasoning explicit. A paste has two parts with opposite characteristics: **content** is large (KBs–MBs), immutable, and read as an opaque whole; **metadata** is tiny (hundreds of bytes), queried and mutated, and the thing you filter, authorize, and expire on. Storing them together is the classic mistake.

**Why content goes to object storage.** Object stores (S3 and friends) are purpose-built for exactly this: large immutable objects, eleven-nines durability, cheap per-GB pricing, virtually unlimited capacity, and direct CDN integration. They give you durable blob storage without you operating replicas or worrying about a 10 MB row.

**Why metadata goes to a database.** The attributes you query — *find pastes that expired*, *list this owner's pastes*, *is this one private?* — need indexes and fast point lookups on small rows. A database does that well; an object store cannot (it has no secondary indexes, no `WHERE expires_at < now`).

<Callout variant="info" title="Why not just put the blob in the database?">
Storing multi-megabyte blobs as database rows or BLOB columns bloats row size, balloons backups, blows up replication traffic, and evicts the small hot rows you actually query from the buffer cache — all to get durability and scale a database is worse at than an object store. The pointer indirection (`blob_key`) costs one extra fetch on a cache miss and buys cheap, durable, CDN-frontable content storage. It is almost always the right trade.
</Callout>

The join between them is the `blob_key` pointer. One refinement is **content-addressed storage**: name the blob by the hash of its content, so two identical pastes share one object and you get **deduplication** for free — handy when the same stack trace gets pasted a thousand times. The cost is that deletion must be reference-counted (don't delete a blob another paste still points at), so most designs start with one blob per paste and add content-addressing only when dedup measurably pays.

<h2 id="expiry-and-ttl">Expiry & TTL</h2>

Pastes are ephemeral, and getting expiry right is what separates a Pastebin from a URL shortener. The semantics: a paste with a non-null `expires_at` must become unreadable the instant that time passes, and its storage must eventually be reclaimed. Two mechanisms together deliver both halves.

**Lazy deletion** checks `expires_at` on every read and returns `410 Gone` if it has passed, refusing to serve the content even though the bytes still physically exist. This makes expiry *correct immediately* — there is no window where an expired paste is still readable — and costs nothing extra, since the read already loads the metadata row. Its weakness is that it never reclaims storage: a paste that's created and never read again would sit on disk forever.

**Active deletion** closes that gap. An expiry worker periodically queries the `expires_at` index for pastes past due and deletes both the metadata row and the object-storage blob; alternatively, an **object-storage lifecycle policy** can expire blobs by age directly. This *reclaims storage* but runs in batches, so on its own it would leave a paste briefly readable between its expiry time and the next sweep.

The two are complementary, so you run **both**: lazy deletion guarantees an expired paste is never served (correctness, immediate), and active deletion guarantees the storage is actually freed (cost, eventual). Lazy alone leaks storage; active alone has a correctness gap. Together they're tight.

<Callout variant="warning" title="One-time pastes need an atomic read-and-delete">
A "burn after reading" paste must be served exactly once, so the read and the delete have to be a single atomic step. If you check-then-serve-then-delete, two concurrent readers can both pass the check and both get the content — the secret leaks twice. Make the first read a conditional, atomic operation (e.g. a compare-and-delete on the metadata row, or a single-use token consumed transactionally) so exactly one reader wins and the loser gets a 410.
</Callout>

A subtle interaction: **expiry must propagate to the CDN.** If the edge cached the content with a long TTL, the paste can outlive its `expires_at` at the edge. The fixes are to bound the edge TTL to something shorter than typical paste lifetimes and revalidate, or to actively **purge the CDN key** when a paste is deleted, or to make short-lived pastes `no-store` so they're never edge-cached at all. Expiry is only as tight as the most stale copy.

<h2 id="caching-cdn">Caching & CDN</h2>

The read path is a caching problem because paste content has the perfect property for caching: it is **immutable**. Once written, `/{id}/raw` returns identical bytes to every reader forever (until the paste is deleted), so it can be cached at the edge with a long TTL and served near the user — which is exactly how a globally read-heavy service hits sub-100 ms p99 while the origin stays nearly idle.

- **Cache the immutable raw content.** Public and unlisted pastes get `Cache-Control: public, max-age=…, immutable`, keyed by id. The first reader anywhere warms the edge; everyone after hits it. The origin sees only misses and creates.
- **Negative-cache misses.** Cache `404` (unknown id) and `410` (expired) briefly so a flood of requests for a dead or guessed id doesn't stampede the origin and database.
- **Invalidate on death.** When a paste is deleted or expires, the cached copy must go — either by purging the CDN key, or by keeping the edge TTL short enough (with origin revalidation) that staleness is bounded. This is the one crack in "immutable": content never changes, but it can *cease to exist*.
- **Private content bypasses the shared cache.** A private or password paste must never sit in a shared edge cache where another user could be served it. Mark those responses `no-store`, or serve their blobs through **signed, time-limited URLs** that the CDN treats as per-user and uncacheable.

<h2 id="access-control">Access Control & Privacy</h2>

Visibility is a per-paste attribute checked from metadata on every read, before the blob is fetched. Four levels cover the product, and they differ in *where* the secret lives and *whether the content can be edge-cached*.

- **Public** — listed and indexable; anyone can find and read it. Fully CDN-cacheable.
- **Unlisted** — readable by anyone with the link but not listed or indexed. The security is the **id's entropy**: the link itself is the secret, which is why ids must be long and random enough to resist enumeration. Still CDN-cacheable (the URL is the capability).
- **Private** — only the owner, authenticated, may read it. The check hits the origin (an authenticated session), so private content **bypasses the shared CDN cache**.
- **Password-protected** — readable by anyone who supplies the password. The metadata holds a `password_hash`; the origin verifies the supplied password against it before serving, so these also bypass the shared cache.

The design tension is **privacy versus cacheability**: anything that requires a per-user authorization decision can't be served from a shared edge cache, because the cache can't make that decision. So public and unlisted pastes — where the URL *is* the authorization — ride the CDN, while private and password pastes are authorized at the origin and served `no-store` or via **signed, time-limited URLs** that grant one reader temporary access to the blob without making it publicly cacheable.

<Callout variant="info" title="Unlisted is only as private as the id is unguessable">
"Unlisted" leans entirely on the id being infeasible to guess or enumerate. A short sequential id (`/1`, `/2`, `/3`) would let anyone scrape every unlisted paste. Use long, random, high-entropy ids for unlisted/private pastes so the link is a genuine secret, and never expose a sequential counter as the public id.
</Callout>

The progression maps onto the capacity numbers. A single node storing rows and files is fine for a prototype but has no durability story, so the first real move is to **split content into object storage** (durable, cheap, unbounded) and keep small metadata in a database — the split this whole design is built on. Most Pastebins live comfortably at "object storage + DB + CDN"; partitioning and multi-region are justified only by measured scale.

<h2 id="security-abuse">Security & Abuse</h2>

A Pastebin accepts arbitrary text from anonymous users and serves it back to others, which makes it a magnet for abuse. The defenses fall into content safety, platform abuse, and a serving rule that prevents the worst class of attack.

- **Serve raw as `text/plain`, never `text/html`.** The single most important rule. If you serve untrusted paste content with an HTML content type, a paste containing `<script>` becomes **stored XSS** that runs in the reader's browser on your domain. Serve raw content as `text/plain` (and render the highlighted view only through a sandboxed, escaped pipeline, ideally on a separate origin) so pasted markup is inert.
- **Enforce the size cap and rate-limit creates.** The 10 MB cap bounds storage and memory; rate-limiting paste creation per IP/account (the Rate Limiter problem) blunts spammers using the service as free hosting for phishing or malware payloads.
- **Scan for secrets and malware.** Asynchronously scan new pastes for leaked credentials (API keys, tokens) and known-malicious payloads, flagging or quarantining hits — both to protect users who paste secrets by accident and to keep the platform off abuse blocklists.
- **Support abuse reports and takedown.** Provide a report path and a fast takedown mechanism (delete row + blob + purge CDN), since public pastes are routinely abused for phishing pages and pirated content.
- **Don't leak unlisted ids.** Unlisted privacy rests on id entropy, so never expose a sequential or guessable id, and negative-cache `404`s so an attacker can't cheaply probe the id space.

<Callout variant="warning" title="Untrusted content is the threat, and text/plain is the seatbelt">
The defining security fact of a Pastebin is that it stores attacker-controlled content and serves it to victims. Always serve the raw artifact as `text/plain` so it can't execute, isolate any rendered/highlighted view (sandbox, separate origin, strict escaping), and treat every paste as hostile. Most Pastebin security incidents are stored XSS from serving untrusted content with an executable content type.
</Callout>

<h2 id="observability">Observability</h2>

Instrument the two things that define the service's health: the read path's latency and cache effectiveness, and the storage system's growth and expiry hygiene.

- **Read latency (p99) and CDN hit ratio** — the headline SLIs. A falling hit ratio means more reads are hitting the origin and latency will follow; alert on it before users feel it.
- **Create rate and error mix** — creates per second plus the breakdown of `413` (oversize) and `429` (rate-limited), which together reveal abuse spikes and client misbehavior.
- **Expiry backlog and storage growth** — the count of past-due-but-still-present pastes and total bytes stored. A growing backlog means the expiry worker is falling behind and cost is leaking; this is the metric that catches a silently dead sweeper.
- **404 vs 410 rates** — a spike in `404`s can signal id enumeration (an abuse probe); a spike in `410`s is just normal expiry but worth distinguishing.
- **Durability / replication health** — object-storage and metadata replication status, since this is user data and silent replication lag is a latent data-loss risk.

<Callout variant="info" title="The metric that catches the silent failure">
Lazy deletion masks a broken expiry worker: pastes still correctly return 410 on read, so nothing looks wrong — while storage quietly grows forever. Alert on the **count of expired-but-not-yet-deleted pastes** (and on storage growth outpacing create volume) so a dead sweeper becomes a visible, paged event rather than a surprise on the cloud bill.
</Callout>

Define SLOs the way you would for any read-heavy service: an availability SLO on reads, a latency SLO on the cached read path, and an explicit budget for expiry-sweep lag — because every hour the sweeper is behind is storage you're paying for and can't account for.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Store large immutable content in object storage and small queryable metadata in a partitioned database joined by a blob_key pointer, serve immutable raw content through a CDN, and enforce expiry with both lazy deletion (410 on read) and an active sweeper."
  rationale="Content and metadata have opposite profiles, so splitting them lets each scale on its own axis — durable cheap blobs in object storage, fast indexed point lookups and expiry queries in the DB. Immutability makes the read path a caching problem the CDN solves, hitting sub-100 ms p99 while the origin idles. Lazy + active expiry gives immediate correctness and eventual storage reclamation; blob-before-metadata ordering makes the only crash state a harmless orphan blob."
  alternatives="Blob-in-database when pastes are tiny and volume is low (one system, no indirection) — at the cost of bloated backups and replication; content-addressed storage for free dedup when the same content repeats — at the cost of reference-counted deletes; active-only or lazy-only expiry — each gives up either correctness or storage reclamation; no CDN when readership is single-region and small."
  revisitWhen="Average paste size or volume shifts the storage math, dedup would materially cut cost, readership globalizes (add/expand the CDN), metadata outgrows one database (partition + multi-region), or a new privacy tier changes what can be edge-cached."
/>

Three axes capture the real tension:

- **Blob-in-DB vs. object storage.** Keeping content in the database is simpler (one system, no pointer) and fine at tiny scale, but it bloats backups, replication, and cache for none of the durability or cost benefits object storage gives. We split because content and metadata have opposite needs.
- **Lazy vs. active expiry.** Lazy deletion is immediate and correct but never reclaims storage; active deletion reclaims but lags. Neither alone is enough, so we run both — lazy for correctness, active for cost.
- **Privacy vs. cacheability.** Anything needing a per-user authorization decision can't ride a shared edge cache. Public/unlisted pastes (where the URL is the capability) are aggressively cached; private/password pastes are authorized at the origin and served no-store or via signed URLs — trading the CDN's speed for the access control they require.

<h2 id="interview-summary">Interview Summary</h2>

<Callout variant="interview" title="A 60-second end-to-end answer">
"A Pastebin stores a blob of text and returns a short shareable URL, with expiry and privacy. It looks like a URL shortener, but the real problems are large blobs, expiry, and access control — not key generation. So I split storage: the large, immutable *content* goes to object storage for durability and cheap scale, and small *metadata* — id, a blob pointer, visibility, expires_at — goes to a partitioned database that I authorize and expire on with fast point lookups. Because content never changes, the read path is a caching problem: a CDN serves immutable raw content from the edge, so the origin sees a fraction of reads and p99 stays under 100 ms globally. Expiry runs two ways — lazy deletion returns 410 on read so an expired paste is never served, and an active sweeper (or object-storage lifecycle policy) reclaims the storage. On create I write the blob before the metadata row so the worst crash state is a harmless orphan blob. Privacy is a per-paste check: public and unlisted pastes ride the CDN because the URL is the capability, while private and password pastes are authorized at the origin and bypass the shared cache. And I serve raw content as text/plain so a malicious paste can't run as stored XSS."
</Callout>
