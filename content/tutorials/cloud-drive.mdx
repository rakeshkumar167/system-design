<h2 id="interview-framing">Interview Framing</h2>

A cloud drive is a service that stores a user's files durably in the cloud and keeps them in sync across every device they own — the desktop folder, the phone, the web. You drag a file into a folder on your laptop and moments later it appears on your phone; you edit it on your phone and the laptop catches up; you share it with a colleague and they see your changes too. Dropbox, Google Drive, and OneDrive are the canonical examples. The payoff is that your files live everywhere and survive any single device dying. The trap is to picture the whole thing as upload-a-file, download-a-file in front of a big blob store — a glorified FTP server. That model handles the easy 10% and collapses on the hard 90%.

The hard part is not storing bytes; object storage already does that cheaply and durably. The hard part is **sync** and everything it forces. Files are huge and edited constantly, so re-uploading a whole file on every keystroke-save is ruinous — you must split files into **chunks** and move only the chunks that changed (**delta sync**). The same content appears over and over — the same attachment in a thousand inboxes, the same photo across versions — so you must store each unique chunk exactly once (**content-addressed deduplication**), or your storage bill is a multiple of what it should be. Many devices must converge on one truth, so you need a **sync protocol** with a cursor and change notifications, and a way to reconcile two devices that edited the same file offline (**conflicted copies**). And underneath it all sits the defining architectural decision: file *content* is enormous and immutable and belongs in **object storage**, while file *metadata* — the folder tree, versions, permissions, and the list of chunks that make up each file — is small, mutable, and transactional, and belongs in a **separate database**. The reframe to lead with: *it looks like upload/download over a blob store, but the hard part is keeping many devices in sync — split files into chunks so you move and store only what changed, and split the system into a tiny transactional metadata plane and a massive immutable block plane.*

Spend the opening minutes scoping. Good questions to ask:

- How big are files, and how often are they edited? This decides whether chunking and delta sync are essential (they are, at scale) or premature.
- How many devices per user, and how fast must a change propagate — seconds or minutes? That sizes the notification path versus plain polling.
- Do we need to merge concurrent edits to the *same* file, or is "last writer wins with a conflicted copy" acceptable? (Real-time co-editing of one document is a different problem — see the Collaborative Document Editor.)
- How important is deduplication and version history? These shape storage and the metadata model.
- What's the durability bar? For people's only copy of their files, it's the highest in the system.

**5 min** framing and the "sync, not storage" reframe · **5 min** requirements and the durability bar · **5 min** capacity (storage at exabyte scale, dedup savings, and delta-sync bandwidth fall out here) · **5 min** entity model (the file → version → block-list → block chain) and the API · **10 min** chunking, deduplication, and delta sync — the centerpiece · **10 min** the sync protocol (cursor + notifications), metadata/block split, sharing, and conflicts · **5 min** versioning, failure modes, and wrap-up. Say your assumptions out loud — the interviewer is grading whether you reach for chunking + dedup + the metadata/block split, not whether you can name a specific object store.
</Callout>

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** chunking and content-addressed storage, deduplication, delta sync, the sync protocol (cursor + notifications), the metadata store and namespace, sharing and permissions, and versioning and history. **Out of scope:** real-time collaborative co-editing of a single document (that's the Collaborative Document Editor — here a file is the unit and concurrent edits become *conflicted copies*, not merged operations), full-text search of file contents, thumbnail/preview generation, end-to-end encryption and key management beyond a mention, the client UI, and billing — name them so the interviewer sees the boundary, then set them aside.

Separate what the system must *do* from how *well* it must do it. For a cloud drive the functional list is broad — store, sync, share, version — but the non-functional list is dominated by one bar above all others: **durability**. This is, for many people, the only copy of their files.

Three targets shape every later decision. First, **durability is the top bar** — losing a user's file is unforgivable in a way that a brief outage is not — which is why content lives in replicated/erasure-coded object storage and why the design is paranoid about a commit referencing a block that doesn't exist. Second, **bandwidth efficiency is a functional requirement, not a nicety**: at this scale, re-uploading whole files would cost more in network than the storage itself, so chunking and delta sync are load-bearing. Third, **convergence across devices** — every device must eventually reach the same state, and a user must see their own writes immediately on the device that made them — which is the whole job of the sync protocol. Note what's *relaxed*: cross-device propagation is allowed to take a few seconds, and concurrent edits may produce a conflicted copy rather than a clever merge. That latitude is what keeps the system tractable.

Estimates here pin down four things: how much raw data there is (it's enormous), how much **deduplication** claws back, how busy the **metadata** plane is, and how much **delta sync** saves on the wire. We size all of it from one set of assumptions.

The takeaways. First, **raw storage is exabyte-scale**: 500M users at 10 GB each is ~**5 EB** of data. There is no database on earth you put that in — content *must* live in **object storage** (replicated or erasure-coded blob storage), addressed not by a row id but by the hash of its bytes. Second, **deduplication is real money**: storing each unique chunk exactly once removes ~30% here, taking 5 EB down to ~**3.5 EB** — a saving of 1.5 EB that costs nothing but a hash lookup. Third, **metadata is tiny but hot**: the namespace tree, versions, and chunk lists are kilobytes per file, but at ~**58,000 commits/sec** the metadata plane is a high-QPS transactional workload that needs its own sharded store — completely different hardware from the blobs. Fourth, and the headline of the wire: if every edit re-sent the whole file, upload bandwidth would be ~**231 GB/s**; sending only the **changed chunks** cuts it ~10× to ~**23 GB/s**. Dedup shrinks what you *store*; delta sync shrinks what you *move*. Together they're the difference between a viable product and an impossible one — which is why the next sections are about chunks, not servers.

<h2 id="entity-model">Entity Model</h2>

The model has a distinctive shape: a thin, mutable **metadata** layer that points into a fat, immutable **block** layer. The single most important idea is that a file's *content* is never stored as one object — it's an ordered **list of block hashes**, and the blocks themselves are content-addressed and shared.

/>

Four entities complete the picture. A **FileVersion** is an immutable snapshot of a file: an ordered list of block hashes, plus who wrote it and when. Because it's just a list of hashes, creating a version is nearly free and history costs only the metadata. A **Block** is a chunk of file content, addressed by the cryptographic hash of its bytes; it is immutable and stored once, so the same block is reused across every file, version, and user that happens to contain identical bytes — this is what makes deduplication automatic. The **Namespace** is a user's (or a shared folder's) file tree and the unit of sharding and of the change journal. And a **ShareACL** records who may access what, with permission inheritance down a folder subtree. The whole design follows from one observation: separate the mutable pointers (metadata) from the immutable bytes (blocks), and both versioning and dedup come almost for free.

<h2 id="api-design">API Design</h2>

The interface reflects the metadata/block split directly: you upload **content** by hash to the block plane, then **commit** a new version to the metadata plane, and you pull changes with a **cursor**. The defining move is the **two-phase write** — first make sure the server has the bytes, then atomically publish the version that references them — which is what lets dedup and conflict detection work.

The whole surface is: put blocks by hash, commit a version, pull a delta by cursor. Routing content by hash gives idempotent uploads and free dedup; committing against a base version gives conflict detection; pulling by cursor gives efficient, resumable sync. Everything heavy (the bytes) flows through the block plane; everything coordinating (versions, cursors, ACLs) flows through the metadata plane.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture splits cleanly into the two planes the entity model implied: a transactional **metadata** path and a massive immutable **block** path, with a **notification** service tying devices together.

<CloudDriveArchitecture />

- **Client (watcher + chunker).** The desktop/mobile app. It watches a local folder, splits changed files into content-defined chunks, hashes them, probes which are missing, uploads only those, and commits a new version. It also holds the device's sync **cursor**.
- **Sync Gateway.** The entry point: authenticates the device, routes block traffic to the block service and metadata traffic to the metadata service, and terminates the notification channel.
- **Metadata Service → Metadata DB.** Owns the namespace tree, file versions, chunk lists, ACLs, and the per-namespace change journal/cursor. The DB is sharded by namespace and transactional — small rows, high QPS.
- **Block Service → Block Storage.** Owns the content-addressed blocks. Block storage is the exabyte-scale object store (replicated/erasure-coded for durability). Blocks are immutable and written once per unique hash.
- **Notification Service.** When a commit lands, it fans a lightweight "namespace X changed" signal out to the user's other connected devices (long-poll/push), so they call `/delta` immediately instead of polling.

The decisive boundary is **metadata is small, mutable, and transactional; content is enormous, immutable, and content-addressed** — and they live in different stores tuned for opposite workloads. A write is two-phase: blocks go to the block plane (deduplicated by hash, only new ones transferred), then a single transactional commit publishes the version in the metadata plane and advances the cursor; the notification service wakes the other devices, which pull the delta and fetch any new blocks. Every section that follows elaborates one piece: how files become chunks and dedup (chunking), how only changes move (delta sync), how devices converge (sync protocol), how the namespace scales (metadata store), and how sharing and versions layer on top.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: uploading a new file, syncing an edit, propagating a change to another device, and resolving a conflict. The first two are the write path; the last two are how "many devices" converges.

**File upload.** The client chunks and hashes the file, probes which block hashes the server is missing, uploads only those blocks, then commits a version referencing the full ordered list. A file that's really a duplicate uploads almost instantly because every block already exists.

<FileUploadSequence />

**Delta sync.** An existing file is edited. The client re-chunks it, diffs the new chunk list against the last-synced one, finds that only a few blocks changed, uploads just those, and commits a new version that references the unchanged blocks by hash. A one-line edit to a large document moves a few kilobytes, not the whole file.

<DeltaSyncSequence />

**Change notification.** A commit on one device must reach the others. The metadata service raises a change event; the notification service fans it out to the user's connected devices; each device pulls the metadata delta since its cursor, downloads any new blocks, and applies the change locally — convergence in seconds, without constant polling.

<ChangeNotificationSequence />

**Conflict.** Two devices edit the same file offline from the same base version. The first commit wins and becomes the new version; the second arrives with a stale base version, the metadata service detects it, and the second edit is preserved as a **conflicted copy** ("report (Device-B's conflicted copy).docx"). No edit is lost, and no clever merge is attempted — that's the deliberate boundary versus a collaborative editor.

<h2 id="chunking-dedup">Chunking & Deduplication</h2>

This is the centerpiece, and the reason a cloud drive is more than a folder uploaded to a blob store. Instead of storing a file as one object, the client splits it into **chunks** (typically a few MB each), hashes each chunk, and stores blocks **addressed by that hash**. Two consequences follow immediately. **Deduplication:** if two files — or two versions, or two users — contain an identical chunk, its hash is identical, so it's stored exactly **once**; the same attachment in a thousand accounts costs one copy. **Cheap versioning and delta sync:** a file version is just a list of hashes, so an edit that touches one chunk reuses every other chunk by reference.

The subtlety is *where* to cut the chunks. **Fixed-size chunking** (every 4 MB) is simple but brittle: insert a single byte at the front of a file and every subsequent boundary shifts, so every chunk's content — and hash — changes, defeating dedup entirely. **Content-defined chunking (CDC)** fixes this by choosing boundaries based on the *content* itself: a rolling hash over a sliding window declares a boundary wherever the hash hits a pattern, so a boundary "sticks" to the surrounding bytes. Insert a byte and only the one chunk around the insertion changes; every other boundary — and hash — is unchanged. CDC is what makes delta sync actually deliver: it keeps the unchanged 99% of a file addressable by the same hashes after an edit.

<Callout variant="info" title="Why content-addressing makes dedup and integrity automatic">
Addressing a block by the hash of its bytes gives three properties for free. **Dedup:** identical content has one address, so it's stored once — globally, across all users. **Idempotent uploads:** writing a block that already exists is a no-op, so retries are safe and a "new" duplicate file uploads in milliseconds (the probe finds every block present). **Integrity:** the address *is* a checksum — on download the client re-hashes and verifies the bytes, catching any corruption or tampering. The cost is a layer of indirection (a file is a list of hashes, not a contiguous object) and the need to garbage-collect blocks no version references anymore. For a system storing exabytes of often-duplicate data, that trade is overwhelmingly worth it.
</Callout>

<h2 id="delta-sync">Delta Sync</h2>

Delta sync is the direct payoff of chunking: when a file changes, transfer only the chunks that changed, not the whole file. The client re-chunks the edited file, computes each chunk's hash, and diffs the new hash list against the version it last synced. The chunks whose hashes are new are the only bytes that travel; the rest are referenced by hash in the new version. Editing one paragraph of a 50 MB document uploads a few kilobytes; the new `FileVersion` simply lists the one changed block hash alongside the dozens of unchanged ones.

This is where the capacity math pays off: it's the ~10× cut from ~231 GB/s of naive whole-file upload to ~23 GB/s. And it's bidirectional — a device receiving a change downloads only the chunks it doesn't already have locally, because it, too, addresses content by hash and already holds the unchanged blocks from the previous version. Delta sync is not an optimization bolted on at the end; it's the reason the system can afford to sync constantly. It depends entirely on content-defined chunking keeping the unchanged chunks' hashes stable across an edit — which is why the previous section is its foundation.

<h2 id="sync-protocol">Sync Protocol & Notifications</h2>

Sync is how every device converges on one truth, and the mechanism is a per-namespace **change journal** with a monotonic **cursor**. Each commit appends an entry to the namespace's journal and advances the cursor; a device remembers the last cursor it processed and calls `GET /delta?cursor=…` to receive exactly the changes it hasn't seen, then stores the new cursor. This makes sync **incremental and resumable**: a device that's been offline for a week pulls the accumulated delta and catches up; a device that just synced pulls nothing. It also makes the client logic simple — there's no diffing of full trees, just "apply everything after my cursor."

Polling that endpoint on a timer would work but trades latency against load: poll every 30 seconds and changes lag; poll every second and you hammer the metadata service with mostly-empty responses. So a **notification channel** sits alongside it: each device holds a long-poll or push connection, and when a commit lands the **notification service** sends a lightweight "namespace ns7 changed past cursor C" signal to that user's *other* devices, which immediately call `/delta`. The notification carries no file data — it's just a nudge to pull — so it's cheap to fan out and the metadata path stays the single source of truth. If the notification channel drops, the client falls back to periodic polling, so correctness never depends on a delivered push, only latency does.

<h2 id="metadata-store">Metadata Store & Namespace</h2>

The metadata plane is small in bytes but is the system's transactional heart: it holds the folder tree, every file's version history and chunk list, the ACLs, and each namespace's change journal — and it serves the ~58,000 commits/sec from the capacity model plus far more reads (listings, delta pulls). It needs to be a **sharded, transactional** store, and the natural shard key is the **namespace** (a user or a shared folder). Sharding by namespace keeps a user's entire tree, their versions, and their change journal **co-located on one shard**, so a commit — which validates blocks, writes a version, and advances the cursor — is a local transaction, and a `/delta` pull reads one shard's journal. Cross-namespace operations are rare; the common case stays single-shard and fast.

Two operations show why this plane is separate from the blocks. **Listing a folder** is a pure metadata read — it never touches block storage — so browsing your drive is fast even though the content is exabytes away. **Moving or renaming a file or a whole folder** is a *metadata-only* operation: it rewrites path pointers in the tree and **moves zero blocks**, because blocks are addressed by content, not location. A user dragging a 10 GB folder to a new parent triggers a tiny metadata update, not a 10 GB copy. This is the quiet superpower of the metadata/block split: the expensive-looking operations (move, rename, version, share) are cheap because they only shuffle pointers, while the genuinely large thing — the bytes — sits immutable and untouched in object storage.

<h2 id="sharing-permissions">Sharing & Permissions</h2>

Sharing is where the namespace stops being a simple per-user tree. When Alice shares a folder with Bob, that folder must appear in Bob's drive too — pointing at the *same* underlying files, versions, and blocks — without copying anything. The clean way to model this is to make a shared folder its **own namespace** (a "mount") that can be attached into multiple users' trees: Alice and Bob both have a pointer to namespace `ns-shared`, and edits in it are journaled once and synced to everyone mounted on it. Because content is content-addressed, no bytes are duplicated when a file is shared; only an ACL and a mount point are added.

Permissions are expressed as **ACLs** on a file or folder — `viewer`, `editor`, `owner` — and they **inherit** down a subtree: granting Bob `editor` on a folder gives him edit rights on everything beneath it unless a deeper entry overrides. On every access the system resolves the **effective permission** by walking from the item up to the nearest governing ACL, then authorizes the operation: a read resolves to viewer-or-above before serving block hashes; a commit requires editor-or-above. The hard parts are the usual ones for hierarchical authorization — keeping inheritance fast (cache the resolved permission per subtree), handling revocation promptly (a removed share must stop future reads), and avoiding the trap of treating sharing as a copy. Modeling a share as a mounted namespace plus an ACL keeps it a metadata operation: instant, storage-free, and consistent for every participant because they're all reading the one journal.

<h2 id="versioning-history">Versioning & History</h2>

Versioning falls out of the model almost for free. Because every commit creates an **immutable `FileVersion`** that is just an ordered list of block hashes, keeping history costs only the metadata of each version — the blocks are shared across versions, so an edited file's old and new versions overlap on every unchanged chunk. Restoring a previous version is therefore trivial and cheap: point `current_version` back at an older `FileVersion`; all its blocks still exist (as long as they're referenced), so nothing needs to be re-downloaded or reconstructed. The same machinery gives you point-in-time recovery, "see who changed this and when," and undo.

The cost to manage is **retention and garbage collection**. Infinite history is infinite metadata and pins blocks forever, so versions are pruned by policy (keep 30 days, or the last N versions, or longer for paid tiers). Deletes are **tombstones** — the file is marked deleted and moved to trash, content retained for a recovery window — so an accidental delete is undoable; only after the window expires is the version eligible for removal. Reclaiming the underlying blocks requires **reference-counted or mark-and-sweep garbage collection**: a block can be deleted only once *no* version in *any* namespace references its hash — which, with global dedup, means a block one user "deletes" may stay alive because another user's file still contains it. GC must run carefully against live commits so it never collects a block a brand-new version just referenced. Versioning is cheap to offer and the real engineering is in safely letting go.

<h2 id="scalability-evolution">Scalability & Evolution</h2>

Don't build the geo-replicated, globally-deduplicated drive on day one. Each stage answers a specific pressure and adds a specific cost — name the trigger before taking on the complexity.

<TradeoffTable
  columns={["Stage", "Trigger", "Cost"]}
  rows={[
    ["Single app server + local disk/one bucket", "Prototype; a few users; whole-file upload", "No dedup or delta; storage and bandwidth blow up; no real durability"],
    ["Object store for blocks + separate metadata DB", "Data outgrows a disk; need durability and fast listings", "Operating two stores; the two-phase write; content-addressing the blocks"],
    ["Add chunking, dedup, delta sync + notification sync", "Bandwidth and storage costs dominate; multi-device users", "Chunker/CDC on the client; a probe step; a journal/cursor + push channel to run"],
    ["Shard metadata by namespace; geo-replicate blocks", "100s of millions of users; global latency; durability across regions", "Cross-region replication and conflict handling; GC across shards; harder consistency"],
  ]}
  recommendedRow={2}
/>

The progression tracks the cost you're buying down. A single server proves the product but has no durability and re-sends whole files. Splitting into an **object store for blocks and a database for metadata** is the foundational move — it's where durability and fast metadata operations come from. Adding **chunking, dedup, delta sync, and notification-driven sync** is where the economics and the user experience actually arrive, and is where most of the design lives — it's the recommended center of gravity. Only a planet-scale product needs **namespace-sharded metadata and geo-replicated blocks**, and it pays for it in cross-region consistency and the genuine difficulty of garbage-collecting deduplicated blocks across shards and regions. Most cloud drives are well-served by "object store + sharded metadata DB, with chunk-level dedup and delta sync."

<h2 id="resiliency-failure-modes">Resiliency & Failure Modes</h2>

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. A cloud drive's failures are dominated by one rule: **never lose a byte, and never let metadata point at a byte that isn't there.** The two planes fail differently, and the dangerous bugs live at the seam between them.

The theme: **the block plane must never lose or corrupt content, and the metadata plane must never publish a version that points at content the block plane doesn't have.** Durability is bought with replication/erasure coding and integrity scrubbing; the two-phase write (blocks first, atomic commit last, with block-existence validation) keeps the seam safe — a crash leaves harmless orphaned blocks, never a dangling version. Conflicts degrade to a preserved conflicted copy rather than a lost update, the notification channel is allowed to fail because the cursor makes it non-authoritative, and the most subtle danger — GC over a deduplicated store — is handled by conservative reference counting that errs toward keeping a block. Losing a file is the one failure with no acceptable mitigation, so every design choice bends toward never doing it.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Split the system into a transactional metadata plane (namespace tree, immutable FileVersions as ordered block-hash lists, ACLs, per-namespace change journal/cursor) and a content-addressed block plane in exabyte-scale object storage; chunk files with content-defined chunking and deduplicate blocks globally; sync edits with delta sync (upload only changed chunks) and converge devices via a cursor + a notification channel; resolve concurrent offline edits as conflicted copies; and offer near-free versioning from the immutable block-list model."
  rationale="The metadata/block split lets each plane be tuned for opposite workloads — tiny transactional rows vs. huge immutable blobs — and makes moves, renames, shares, and versions cheap metadata operations. Content-defined chunking plus content-addressing make dedup and delta sync automatic, cutting storage ~30% and upload bandwidth ~10× — the difference between viable and impossible at exabyte scale. A cursor/journal gives correct, resumable convergence; notifications make it fast without making it fragile. Conflicted copies keep the system simple and lossless where true merge isn't required."
  alternatives="Store each file as one blob — simple but no dedup, no delta sync, ruinous bandwidth, disqualified at scale; fixed-size chunking — simpler than CDC but breaks dedup on insertion; poll-only sync — works but trades latency for load and empty requests; operational/CRDT merge of concurrent edits — needed for live co-editing but far more complex than conflicted copies for a file-granular drive; per-user (not global) dedup — simpler GC but forgoes the largest dedup savings; one combined store for metadata and blocks — simpler to operate but can't scale the two opposite workloads independently."
  revisitWhen="Files become tiny and numerous (chunking overhead may dominate — consider packing small files), workloads demand real-time co-editing (move to OT/CRDT for those document types), regulatory/residency rules force regional isolation (constrains global dedup and geo-replication), or dedup's GC cost across shards outweighs its savings (consider scoping dedup per namespace)."
/>

Three axes capture the real tension:

- **Content-defined vs. fixed chunking.** Fixed is trivial but loses dedup the instant bytes are inserted; CDC costs a rolling-hash pass but keeps unchanged chunks stable across edits, which is what makes delta sync real. For a sync product, CDC is non-negotiable.
- **Conflicted copies vs. operational merge.** Merging concurrent edits (OT/CRDT) is magical for a shared document but heavy, and it assumes a mergeable data type. A general file drive holds opaque bytes (a video, a binary), so a conflicted copy — lossless and simple — is the right default; reserve true merge for the editor problem.
- **Global vs. per-user dedup.** Global dedup maximizes savings (one copy of a viral attachment for the whole platform) but complicates garbage collection and raises privacy questions; per-user dedup is simpler and isolates GC but stores popular content many times. Most start global within a trust/region boundary and scope it where compliance demands.

<h2 id="interview-summary">Interview Summary</h2>

<Callout variant="interview" title="A 60-second end-to-end answer">
"A cloud drive looks like upload/download over a blob store, but the hard part is sync across many devices. The defining decision is to split the system into two planes: a small, transactional metadata store — the folder tree, immutable file versions, ACLs, and a per-namespace change journal — and a massive, immutable, content-addressed block store in object storage for the actual bytes. Files are chunked with content-defined chunking and each unique chunk is hashed and stored once, which gives global deduplication; at 500 million users and 10 GB each that's 5 exabytes of raw data cut to about 3.5 with dedup. Writes are two-phase: the client probes which chunk hashes the server is missing, uploads only those, then commits a new version — an ordered list of block hashes — against a known base version. Editing a big file uploads only the changed chunks, so delta sync cuts upload bandwidth roughly tenfold. Devices converge with a cursor: each pulls the metadata delta since its last position and downloads any new blocks, and a notification channel triggers that pull in seconds instead of polling — but the cursor is authoritative, so a dropped notification only adds latency. Concurrent offline edits are detected by the stale base version and preserved as conflicted copies, not merged. Versioning is nearly free because a version is just a block-hash list, so old and new versions share unchanged chunks; the real work is garbage-collecting blocks no version references anymore. The cardinal rule throughout is durability: never lose a byte, and never let a commit publish a version that points at a block the store doesn't have."
</Callout>
