<h2 id="interview-framing">Interview Framing</h2>

A notification service is the platform other systems call when they need to tell a user something — "your order shipped," "someone mentioned you," "your code finished deploying." A producer hands it a user and a message; the service figures out *which channels* that user wants (push, SMS, email), renders the right template for each, and delivers them through external providers like APNs/FCM, an SMS gateway, and an email provider. It is shared infrastructure: dozens of teams call it instead of each integrating providers themselves.

The trap is to treat it as "call the SMS API." The real problem is that delivery is **slow, external, and flaky** — a provider can be down, rate-limit you, or take seconds to respond — yet the caller wants an answer in milliseconds and the user must not be spammed with duplicates. That tension forces the whole design: accept fast, deliver **asynchronously** through queues, **retry** failures without notifying the user twice, and fan one logical notification out across **many channels**. Spend the opening minutes scoping. Good questions to ask:

- Which channels — push, SMS, email, more? Can one notification go to several at once?
- How reliable must delivery be — at-least-once? Is a rare duplicate acceptable or forbidden?
- Do users have preferences, opt-outs, or quiet hours we must honor?
- What's the volume, and how bursty (e.g. a marketing blast vs. transactional)?
- What's the latency expectation — instant for the caller, and how fast end-to-end to the user?

<Callout variant="interview" title="Suggested 45-minute allocation">
**5 min** framing and scope · **5 min** requirements and channels · **5 min** capacity (the fan-out and retry amplification falls out here) · **5 min** entity model and the 202 API · **10 min** the async fan-out pipeline and architecture · **10 min** delivery guarantees, idempotency, retries, and the dead-letter queue · **5 min** preferences, failure modes, and wrap-up. Say your assumptions out loud — the interviewer is grading whether you reach for async queues, reason about at-least-once versus exactly-once, and make retries safe with idempotency, not whether you can name an email provider.
</Callout>

This tutorial commits to concrete assumptions so the numbers stay consistent end to end. **In scope:** the asynchronous ingestion → queue → fan-out → channel-worker → provider pipeline, delivery guarantees and idempotency/dedup, retries with backoff and a dead-letter queue, user preferences (channels, opt-out, quiet hours, per-user rate limits), and templates. **Out of scope:** an in-app notification inbox/feed (closer to the News Feed problem), real-time WebSocket presence (the Chat problem), marketing-campaign authoring tools, and building the email/SMS providers themselves — name them so the interviewer sees the boundary, then set them aside.

For a notification service the load-bearing non-functional targets are a fast async accept, at-least-once delivery, and — the one that defines correctness — **no duplicate user-visible notifications** despite retries and at-least-once queues.

Three targets shape every later decision. First, the **accept path must return in under ~50 ms**, which is only possible if it does no provider work: it validates, deduplicates, enqueues, and returns `202 Accepted`, leaving the slow delivery to run asynchronously. Second, delivery is **at-least-once** — across queues and flaky providers you cannot guarantee exactly-once, so messages will sometimes be retried or redelivered. Third, and consequently, the system must be **idempotent** so that at-least-once never becomes "the user got five copies": every delivery is keyed and deduplicated. These three together — async accept, at-least-once, idempotent — are the spine of the design.

<h2 id="capacity-estimates">Capacity Estimates</h2>

Estimates are directional — their job is to reveal *which* tier is under pressure. For a notification service the lesson is **amplification**: a modest accept rate is multiplied by fan-out (channels per notification) and again by retries, so the *delivery* tier carries far more load than ingestion.

The takeaways. First, the **accept path is cheap**: 500 million notifications a day is only about **5,800 accepts per second** — a thin API that returns `202` and enqueues handles this trivially, and it is never the bottleneck. Second, **fan-out doubles the work**: at two channels each, that's **~11,600 deliveries per second** to push through queues and workers, and **retries add ~20%** on top, so the channel workers actually perform close to **~13,900 delivery attempts per second**. Third, you **provision for peak, not average**: at a 4× peak the delivery tier must absorb **~46,000 deliveries per second**, and across a day that's **a billion provider calls**. The headline: size the *delivery* tier and provider throughput — the part amplified by fan-out and retries — not the cheap accept path. That asymmetry is exactly why the architecture is asynchronous.

Three related entities round out the model and live in their own stores. A **NotificationRequest** is the logical send the API accepted (id, user, template, payload, requested channels, idempotency key) — the parent of the per-channel `Notification` rows. <h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture is an **asynchronous pipeline**: a thin accept tier puts work on a queue, a fan-out stage explodes one notification into per-channel deliveries, and pools of channel workers drain per-channel queues into external providers. Stores for preferences and deduplication, plus a dead-letter queue, sit alongside.

<NotificationArchitecture />

- **Ingestion API.** Stateless and thin: authenticate the sender, validate, check the idempotency key, write the request durably, enqueue it, and return `202`. It does no provider work, so it stays fast and cheap to scale.
- **Ingestion queue.** Decouples accept from delivery. It absorbs bursts and lets the slow, failure-prone downstream run at its own pace without ever blocking callers.
- **Fan-out service.** Consumes the ingestion queue, resolves the user's **preferences** (which channels are enabled, quiet hours), dedupes against the **dedup store**, renders templates, and publishes one message per enabled channel onto the **per-channel queues**.
- **Per-channel queues + workers.** One queue (and an autoscaled worker pool) per channel, so a slow or rate-limited channel can't head-of-line block the others. Each worker calls its external **provider** and applies retries.
- **Providers.** External, owned by someone else: APNs/FCM for push, an SMS gateway, an email provider. They are the slow, flaky dependency the whole async design exists to tolerate.
- **Dead-letter queue (DLQ).** Where a delivery lands after exhausting its retry budget or being identified as poison — captured for inspection and replay rather than retried forever or dropped silently.

The decisive boundary is **accept versus deliver**: everything before the ingestion queue is fast and synchronous to the caller; everything after is asynchronous, retryable, and idempotent. That split is what lets a millisecond accept coexist with a provider that takes seconds and sometimes fails — and every section that follows elaborates one stage of this pipeline.

<h2 id="detailed-flows">Detailed Flows</h2>

Four flows cover the system: the happy-path fan-out, a transient failure retried with backoff, exhaustion into the dead-letter queue, and idempotent suppression of a duplicate request.

**Send and fan-out.** The API accepts the request and returns `202` immediately; the fan-out service later consumes it, filters by preference, and publishes one message per enabled channel onto the per-channel queues, where workers pick them up.

<SendFanoutSequence />

**Retry with backoff.** A worker calls the provider and gets a transient failure — a timeout or a `5xx`. Rather than dropping it, the worker re-enqueues the delivery with an **exponential backoff and jitter**; a later attempt succeeds. The user sees one notification, late by seconds, instead of none.

<RetryBackoffSequence />

**Dead-letter.** Some failures don't recover — a permanently bad address, or a message that keeps failing. After the retry budget is exhausted, the worker stops retrying and moves the delivery to the **dead-letter queue**, where it's alerted on and can be inspected and replayed after a fix, instead of looping forever and clogging the channel.

<DeadLetterSequence />

**Idempotent accept.** A caller times out and retries the same request with the same `Idempotency-Key`. The first request did a check-and-set on the dedup store and proceeded; the second finds the key already present and is suppressed, returning the same notification id without enqueuing anything new.

<h2 id="fanout-pipeline">Fan-out Pipeline</h2>

The pipeline is asynchronous and queue-based for one reason: **delivery is slow and unreliable, and the caller is neither.** If the accept path called providers inline, a single sluggish SMS gateway would tie up request threads and a provider outage would fail the caller's own request. Putting a **queue** between accept and deliver decouples them — the queue absorbs bursts, smooths load, and lets delivery retry at its own pace while the caller already has its `202`.

Fan-out is the stage that turns one logical notification into N physical deliveries. The fan-out service consumes the ingestion queue, resolves which channels the user actually wants, and **publishes one message per channel** onto separate per-channel queues. This is where the capacity amplification happens — and where a crucial topology decision lives.

<Callout variant="info" title="Why per-channel queues, not one shared queue">
Channels have wildly different throughput, latency, and failure profiles: email providers accept thousands per second, an SMS gateway might rate-limit you, push has its own quirks. If every channel drained one shared queue, a slow or backed-up channel would **head-of-line block** the others — push notifications would pile up behind stuck SMS messages. Giving each channel its own queue and its own autoscaled worker pool isolates them: a backed-up SMS queue grows without ever delaying push or email, and each pool scales to its channel's provider throughput independently.
</Callout>

Each per-channel worker is a small loop: pull a message, render the final payload, call the provider, and on the result either mark it delivered or apply the retry logic. Workers are stateless and horizontally scalable, so the pool autoscales on queue depth. When a channel's queue backs up faster than its workers (and provider) can drain it, **back-pressure** is the safety valve — the queue buffers, low-priority traffic can be shed or deprioritized, and ingestion can be throttled — rather than letting an overwhelmed provider melt down.

<h2 id="delivery-guarantees">Delivery Guarantees & Idempotency</h2>

Start from an uncomfortable truth: **exactly-once delivery is unattainable** here. The worker calls an external provider over a network; if the call times out, the worker cannot know whether the provider sent the message or not. Retry and you risk a duplicate; don't retry and you risk a drop. No amount of cleverness removes that ambiguity across a third-party boundary. So you choose **at-least-once** — always retry on uncertainty, accepting that a message may be delivered more than once — and then make duplicates harmless.

The tool that makes duplicates harmless is **idempotency**. Every delivery carries an idempotency key (for the accept, the client-supplied `Idempotency-Key`; for a per-channel delivery, something like `hash(request_id + channel)`). Before acting, the service does an atomic **check-and-set** against a **dedup store**: if the key is already present, the work has been done (or is in flight) and is suppressed; if not, it's set and the work proceeds. At-least-once delivery plus idempotent suppression yields **effectively-once** — the user sees exactly one notification even though the pipeline may have tried several times.

<Callout variant="warning" title="The dedup check must be atomic">
If the check ("is this key present?") and the set ("mark it present") are separate steps, two concurrent retries of the same message can both check, both see "absent," and both deliver — the exact duplicate you were preventing. Use a single atomic operation: a conditional insert on the unique `idempotency_key`, or a `SET key … NX` in the dedup store, so exactly one actor wins the key and everyone else observes it already taken. This is the same check-then-act race the Rate Limiter tutorial fixes with an atomic increment.
</Callout>

Two refinements matter in interviews. First, **provider-side idempotency**: some providers accept an idempotency key of their own, so even if your retry reaches them twice they suppress the duplicate — use it when available, as it closes the gap exactly where the ambiguity lives. Second, the **dedup store's retention**: keys only need to live as long as duplicates can plausibly arrive (minutes to hours), so the store is a TTL'd cache, not a permanent ledger — which keeps it small and fast.

<h2 id="retries-dlq">Retries & Dead-Letter Queue</h2>

Retries are where reliability is won, but naive retries cause more harm than the failures they paper over. The discipline has three parts: classify the failure, back off correctly, and bound the effort.

**Classify transient vs. permanent.** A `5xx`, a timeout, or a `429` rate-limit is **transient** — retrying later will likely succeed, so retry. A `4xx` like an invalid address or an unsubscribed recipient is **permanent** — retrying will fail identically forever, so don't waste attempts; fail it immediately. Treating every error the same either gives up too early on recoverable failures or hammers the provider with hopeless ones.

**Back off with exponential delay and jitter.** Retry immediately and you stampede a provider that's already struggling; retry on a fixed schedule and every worker retries in lockstep, creating synchronized spikes. **Exponential backoff** (1s, 2s, 4s, 8s…) gives a recovering provider room, and **jitter** (randomizing each delay) desynchronizes the fleet so retries spread out instead of thundering together.

**Bound the effort and dead-letter the rest.** Every delivery has a **retry budget** — a maximum attempt count or total time. When it's exhausted, the message is not dropped and not retried forever; it's moved to the **dead-letter queue**. The DLQ is also where **poison messages** go — a malformed or buggy message that crashes its worker on every attempt would otherwise loop forever, blocking the channel; capping attempts and routing it to the DLQ isolates the damage. Messages in the DLQ are alerted on, inspected, and — after a fix or a provider recovery — **replayed** back through the pipeline.

<h2 id="user-preferences">Preferences & Rate Limiting</h2>

A notification service that ignores what users want is a spam cannon. Preferences are resolved during fan-out, **before** any per-channel delivery is created, so an unwanted channel is never even enqueued.

- **Channel preferences and opt-out.** Per user and per notification type, which channels are enabled. Some opt-outs are not just product preference but **legally mandatory** — an unsubscribe from marketing email, a `STOP` reply to SMS — and must be honored absolutely; when in doubt about an opt-out, **fail closed** (don't send) rather than risk a compliance violation.
- **Quiet hours.** Respect the user's local time: defer non-urgent notifications out of the middle of their night, holding them until quiet hours end. Urgent/transactional notifications may bypass this — a security alert shouldn't wait.
- **Per-user rate limiting and digesting.** Cap how many notifications a single user receives in a window so a misbehaving producer (or a noisy event) can't flood them — the same token-bucket idea as the Rate Limiter tutorial, keyed per user. Beyond a threshold, **collapse** many notifications into a single **digest** ("12 new mentions") rather than 12 separate pushes.

<Callout variant="info" title="Preferences turn 'send' into 'send if appropriate'">
Fan-out is not "create one delivery per requested channel" — it's "create one delivery per channel the user *will accept, right now, without being flooded*." Resolving preferences, opt-outs, quiet hours, and rate limits at fan-out time means the expensive delivery machinery only ever runs for notifications that should actually reach the user, and it keeps the service a good citizen instead of a source of complaints and unsubscribes.
</Callout>

The progression maps onto the capacity numbers. Most services live comfortably at "async, per-channel queues, autoscaled workers"; partitioning and multi-region are justified only by measured scale.

<h2 id="security-abuse">Security & Abuse</h2>

A notification service can reach users directly on their phones and inboxes, which makes it both a high-value abuse target and a compliance surface.

- **Authenticate and authorize senders.** Only trusted services may send, and only to the users and templates they're permitted to — otherwise a compromised caller becomes a phishing channel with your reputation behind it.
- **Per-sender rate limits.** Bound how much any one producer can send (the Rate Limiter problem) so a buggy or compromised sender can't blast millions of messages, burn provider quota, or flood users.
- **Minimize and protect PII.** Payloads carry personal data (names, order details, phone numbers). Minimize what's stored, encrypt it at rest and in transit, and keep retention short — a delivered notification's payload rarely needs to live long.
- **Honor unsubscribe and compliance.** Unsubscribe links, SMS `STOP`, and regional rules (CAN-SPAM, GDPR, TCPA) are legal obligations, not features; opt-out state must be authoritative and fail-closed.
- **Guard against template injection.** Templates interpolate caller-supplied data; escape it per channel so a malicious payload can't inject markup into an email or smuggle content into a push — the same untrusted-input discipline as serving user content elsewhere.
- **Protect provider credentials.** The provider API keys are the keys to your sending reputation; store them in a secrets manager, rotate them, and scope them tightly.

<h2 id="observability">Observability</h2>

Instrument the pipeline end to end, because in an async system "it was accepted" and "the user got it" are far apart, and the gap between them is where problems hide.

- **Accept latency and accept rate** — the cheap front door; should stay flat and fast. A spike in `429`s flags a sender hitting its limit.
- **End-to-end delivery latency** — from accept to provider-confirmed delivery, per channel. This is the SLI users actually feel; a rising p95 usually means a backed-up queue or a degraded provider.
- **Per-channel success and failure rates** — broken down by provider error class, so a degrading provider is visible before it becomes an outage.
- **Retry rate and DLQ depth** — a climbing retry rate signals provider trouble; a growing **DLQ depth** is a paging signal that notifications are being abandoned and need a human.
- **Queue depth and consumer lag** — the leading indicator of delivery latency; alert before the backlog turns into missed SLAs.
- **Dedup hit rate** — how often duplicates are being suppressed; a sudden jump can reveal a caller stuck in a retry loop or a redelivery storm.

<Callout variant="warning" title="The DLQ is your most important alert">
A dead-letter queue that nobody watches is just a place notifications go to die silently. Every message in the DLQ is a notification a user should have received and didn't. Alert on DLQ depth and growth rate — not just log it — so abandoned deliveries become an actionable, paged event with a clear owner, and define an explicit budget for how many dead-letters are acceptable before it's an incident.
</Callout>

Define SLOs the way you would for any async pipeline: an availability SLO on the accept path, an end-to-end delivery-latency SLO per channel, and an explicit budget for DLQ volume — because every dead-lettered message is a notification you promised to deliver and didn't.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Accept synchronously with a 202 and enqueue; fan out through a message queue into per-channel queues drained by autoscaled channel workers that call external providers; deliver at-least-once made safe by an idempotency key plus an atomic dedup store (effectively-once); retry transient failures with exponential backoff and jitter and dead-letter the rest; and resolve user preferences, opt-outs, quiet hours, and rate limits at fan-out time."
  rationale="Async accept decouples a millisecond caller from slow, flaky providers, so a provider's bad day is latency, not a failed caller request. Per-channel queues stop a slow channel from head-of-line blocking the others and let each scale to its provider. At-least-once is the only honest guarantee across an external boundary, and idempotency makes its duplicates harmless. Backoff-with-jitter retries recover transient failures without stampeding, and the DLQ bounds the rest. Resolving preferences before delivery keeps the service a good citizen."
  alternatives="Synchronous direct send for trivial volume (simplest, but couples the caller to providers); exactly-once via distributed transactions (impractical and slow across third-party providers); a single shared queue (simpler, but a slow channel blocks all others); fail-open vs fail-closed dedup/preferences depending on whether a duplicate or a missed opt-out is the worse outcome."
  revisitWhen="Volume or user locality outgrows one region (partition + multi-region, add provider redundancy), a channel needs strict ordering, duplicate tolerance changes (re-tune fail-open vs fail-closed), or a new channel with different throughput/latency joins the fan-out."
/>

Three axes capture the real tension:

- **Synchronous vs. asynchronous.** Calling providers inline is simplest and gives the caller an immediate delivered/failed answer, but couples a fast caller to slow, failure-prone providers. We go async because decoupling is the only way to keep accept fast and delivery reliable — at the cost of a queue to operate and the idempotency it forces.
- **At-least-once vs. exactly-once.** Exactly-once is unattainable across an external provider boundary, so we choose at-least-once and neutralize its duplicates with idempotency. The trade is accepting occasional retries in exchange for never dropping a notification, with dedup ensuring the user still sees it once.
- **Single vs. per-channel queues.** One queue is simpler to run; per-channel queues cost more moving parts but isolate channels so one slow provider can't delay all notifications. At any real scale the isolation is worth the operational cost.

<h2 id="interview-summary">Interview Summary</h2>

<Callout variant="interview" title="A 60-second end-to-end answer">
"A notification service delivers messages to users across push, SMS, and email. The hard part isn't calling provider APIs — it's that delivery is slow and flaky while the caller wants a millisecond answer and the user must never be spammed. So I make the accept asynchronous: the API validates, dedupes on an idempotency key, enqueues, and returns 202. A fan-out service consumes that queue, resolves the user's channel preferences and opt-outs, and publishes one message per channel onto per-channel queues — separate queues so a slow channel can't head-of-line block the others. Autoscaled channel workers drain those queues and call the providers. Delivery is at-least-once, because exactly-once is impossible across an external boundary, and I make it safe with idempotency: an atomic check-and-set on a dedup key so retries and redeliveries notify the user only once. Transient failures retry with exponential backoff and jitter; permanent ones fail fast; and anything that exhausts its retry budget or is poison goes to a dead-letter queue to be alerted on and replayed. At 500 million sends a day, fan-out and retries amplify that to over a billion provider calls, so I size the delivery tier, not the cheap accept path."
</Callout>
