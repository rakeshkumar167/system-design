<h2 id="interview-framing">Interview Framing</h2>

A leaderboard ranks players (or posts, or drivers, or songs) by a score and shows who's on top. Every game, fitness app, trading platform, and social feed has one. From the outside it's the most familiar query in computing: `SELECT player, score FROM scores ORDER BY score DESC LIMIT 10`. Point that at a thousand players and you're done. Point it at fifty million players whose scores change thousands of times a second, and it falls apart in two distinct ways — the sort becomes a repeated full scan, and, worse, the moment someone asks *"what's **my** rank?"* you're computing `COUNT(*) WHERE score > mine` over the entire table on every request.

The hard part is **maintaining a live ranking at scale that answers two different questions cheaply**: the **top-K** (the leaders — one shared, cacheable answer) and an **arbitrary player's exact rank** (a personal answer, different for every one of millions of users, e.g. "you're #4,271,908 of 50 million"). A batch `ORDER BY` is fine for the first and catastrophic for the second, because rank is a *global* property — it depends on everyone else's score — so a naive rank lookup scans or counts the whole dataset. The reframe to lead with: *it looks like `ORDER BY score DESC`, but sorting isn't the problem — the problem is keeping a ranking continuously up to date under a firehose of score changes while answering both "who are the leaders" and "where does this specific player stand" in milliseconds. The move is to hold the ranking in a **sorted set** (a skip list, as in a Redis ZSET) that stays ordered incrementally — giving O(log N) updates, O(log N) rank lookups, and O(log N + K) top-K reads — and then to solve what that data structure makes hard: global rank across shards, time-windowed boards, and durability behind an in-memory index.*

Spend the opening minutes scoping. Good questions to ask:

- What are we ranking, and is the score **absolute** (set to a value) or **incremental** (add points as they're earned)? It changes the write API (`ZADD` vs `ZINCRBY`).
- Do we need only the **top-K**, or also an **individual player's rank** and the players **around** them? The second is what forces a sorted set over a cached top list.
- How many players and how often do scores change? This sizes the write load and whether one node suffices.
- **Time windows** — all-time only, or daily / weekly / per-season boards? Windows multiply the write and storage story.
- How **fresh** and how **exact** must rank be? A rank that's approximate or a second stale is usually fine — a crucial relaxation at scale.

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** the sorted set and its operations, the read and write paths, top-K and per-player rank (with neighbors), sharding and **approximate global rank**, **time-windowed** boards, a durable **source of truth** behind the in-memory index, plus scaling, failure modes, and trade-offs. **Out of scope:** the game or app that *produces* the scores and its anti-cheat/validation (we assume scores arrive already validated — a manipulated score is an upstream concern), the social graph for friends-only boards beyond noting it as a filtered view, and the analytics warehouse that mines historical scores. Three targets shape every later decision. First, **both query shapes must be fast**: the top-K is one shared answer (easy to cache), but a **per-player rank** is different for every user and can't be precomputed for all of them — so the data structure itself must answer "how many players outrank X" in O(log N), which is exactly what a sorted set does and a plain list or a cached top-10 does not. Second, **the load is real-time and read-heavy**: scores stream in continuously (so the ranking must update incrementally, not rebuild), and rank/top-K views vastly outnumber updates, so reads must be cheap and cacheable. Third, and most liberating, **rank can usually be approximate**: telling a player they're "#4,271,908" versus "#4,271,950" out of 50 million is indistinguishable to them, and this single relaxation is what makes **sharded** leaderboards tractable — exact global rank across shards is expensive, approximate is cheap. Note what's **relaxed**: the ranking is **eventually consistent** (a score appearing a second late is fine), and exact global rank is sacrificed at scale. The system trades perfect, instantaneous, exact global rank for fast incremental updates and cheap top-K and per-player rank.

The takeaways. First — the clarifying one — **the whole ranked set fits in memory**: 50 million players at ~100 bytes each (member id, score, and skip-list pointers) is only ~**5 GB**, small enough to live comfortably in the RAM of a single node. Storage is *never* the binding constraint for a leaderboard; that's why an in-memory sorted set is the natural home, not a disk-based database. Second, **it's throughput-bound and read-heavy**: ~5 billion score updates a day is ~**57,870 writes/sec**, and with ~10 rank/top-K views per update the read load is ~**578,704/sec** — roughly **10× the writes** — for ~**636,574 ops/sec** total. Because every sorted-set operation is **O(log N)** (a few dozen pointer hops even at 50M entries), a single node sustains ~100k ops/sec, so ~**7 nodes** of a replicated in-memory set carry the entire load. Third, and the design consequence: since the data is small and reads dominate, you **scale reads by replicating** the set and **cache the hot top-K** — and the genuinely hard problems that remain are **hot boards** (one viral leaderboard exceeding a node's write throughput), **cross-shard global rank**, and **time-window fan-out**, none of which is about raw storage. The capacity story: a leaderboard is easy to *store* and hard to *serve fast under a stream of updates* — plan for throughput, not terabytes.

Two supporting ideas complete the picture. The **ScoreEvent** is the durable atom: one submitted or incremented score (`player_id`, `board_id`, value, timestamp), appended to the source-of-truth store — this is what makes a Redis loss survivable, since the sorted set can be replayed from it. The discipline: the sorted set is a **derived, rebuildable, read-optimized index** holding just identity and score; the durable **source of truth** is the append-only ScoreEvent record; and display data is **hydrated at read time**, never stored in the hot ranking structure.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture is an **in-memory sorted set** doing the ranking, a **cache** absorbing the hottest reads, and a **durable store** behind it as the source of truth — with an async path keeping the two in sync.

<LeaderboardArchitecture />

- **Game Client.** Submits scores as they're earned and requests the top-K and the player's own rank.
- **Leaderboard Service.** Stateless; translates API calls into sorted-set operations (`ZINCRBY`, `ZREVRANGE`, `ZREVRANK`) and hydrates results with profile data.
- **Top-K Cache.** Fronts the service for the global top-K — one shared answer, so a ~1s cache absorbs the bulk of read traffic.
- **Sorted Set.** The ranked structure held **in memory** (a Redis ZSET / skip list). Small (~5 GB), replicated for read scale, and the thing that makes updates and rank queries O(log N).
- **Score Stream.** The async channel carrying every score change from the service toward durable storage.
- **Score DB.** The **durable source of truth** — every score event recorded so the sorted set can be rebuilt after a loss.

The decisive boundary is that **the sorted set serves live ranking from memory while the database durably records the truth asynchronously** — the fast index and the durable log are deliberately separated, and the sorted set is treated as a rebuildable derived structure, not the system of record. Every section that follows elaborates one piece: how the sorted set works, how updates and the two read shapes map to O(log N) operations, how sharding and approximate rank scale it, how time windows are keyed, and how the durable store makes the in-memory index recoverable.

**Submit a score update.** The client submits a score; the service applies it to the sorted set with an O(log N) `ZINCRBY` and emits an event that is durably persisted to the database off the hot path, then acknowledges the client. The in-memory index updates synchronously; durability is asynchronous.

<h2 id="sorted-sets">Sorted Sets: The Core Data Structure</h2>

The sorted set is the whole design, and understanding *why* it's the right structure is the centerpiece of a strong answer. A **sorted set** stores unique members each associated with a **score**, and keeps them **continuously ordered by that score**. The canonical implementation — Redis's ZSET — is a **skip list** (plus a hash map from member to score). A skip list is a linked list with multiple express lanes: probabilistically, each node appears in higher levels less often, so you can "skip" over large stretches and locate any position in **O(log N)** expected time, while still supporting ordered traversal. The companion hash map gives O(1) "what's this member's score" lookups.

That structure is exactly matched to the four things a leaderboard does, and every one is **O(log N)** or better:

- **`ZADD` / `ZINCRBY`** — set or increment a member's score, re-positioning it in the order: **O(log N)**.
- **`ZREVRANGE start stop`** — read a contiguous slice by rank, e.g. the top 10: **O(log N + K)** for K results.
- **`ZREVRANK member`** — the killer feature: return a member's **rank** (its position in the order) directly: **O(log N)**.
- **`ZSCORE member`** — a member's score: **O(1)** via the hash map.

Contrast the alternatives. A **SQL table** with `ORDER BY score` needs an index to avoid a full sort, but computing an arbitrary row's *rank* still means `COUNT(*) WHERE score > x` — an index range scan that's O(rank), potentially tens of millions of rows, per query. A plain **sorted array** gives O(log N) rank via binary search but **O(N)** inserts (shifting elements) — hopeless under a write stream. A **heap** gives you the top element cheaply but not arbitrary rank or ordered ranges. The skip list is the one structure that makes *all four* operations logarithmic simultaneously, which is why "use a sorted set / Redis ZSET" is the expected core of the answer. The principle: a leaderboard needs incremental ordered updates **and** arbitrary-position rank queries at once, and a skip-list-backed sorted set is the data structure that delivers both in O(log N) — turning "recompute the ranking" into "read the already-maintained order."

<h2 id="score-updates">Real-Time Score Updates</h2>

The write path is a single sorted-set operation, but the details matter. The choice between **`ZADD` (set)** and **`ZINCRBY` (increment)** follows the scoring model: incremental games ("+150 points") use `ZINCRBY`, which atomically adds to the current score and re-orders the member in one O(log N) step; "record the best result" boards use a **conditional set** (update only if the new score is higher — `ZADD GT`), so a worse later attempt doesn't lower a high score. Atomicity matters: `ZINCRBY` is a single atomic operation, so concurrent increments to the same player don't lose updates the way a read-modify-write would.

Two scaling concerns ride on the write path. The first is the **hot key / hot board** problem: a single wildly popular leaderboard (a global board, or one player being hammered with updates) concentrates writes on one node, and a sorted set for one board lives on one node — so that board's write throughput is capped by that node. Mitigations: **shard the board** (next section) so writes spread across nodes, **batch/coalesce** rapid updates to the same player in a short window before applying, or absorb bursts through the **score stream** and apply them asynchronously. The second is the **durability timing**: the write updates the in-memory set first for a fast response, then records to the durable store. **Write-behind** (ack after the in-memory update, persist asynchronously) is fast but risks losing the last few updates if a node dies before they're persisted; **write-through** (persist before ack) is safe but slower. Leaderboards usually accept write-behind because a lost score is quickly re-earned and eventual re-sync corrects it — but a *money* leaderboard (a trading contest) would choose write-through. The principle: apply updates as **single atomic sorted-set operations** (increment or conditional-set), and consciously choose the **durability point** (write-behind for speed, write-through for safety) while planning for **hot boards** via sharding, batching, or stream-buffered application.

<h2 id="rank-queries">Top-K & Rank Queries</h2>

The two reads look similar but have very different characteristics, and treating them differently is what keeps the system fast. The **top-K** is `ZREVRANGE 0 K-1` — read the K highest-scoring members in order, O(log N + K). Its defining property is that it's the **same answer for everyone**, so it's aggressively **cacheable**: a 1-second cache on the global top-10 collapses a flood of identical reads into one sorted-set operation per second per replica. The top-K is the easy read.

The **per-player rank** is the interesting one, because it's a **different answer for every user** and therefore *not* cacheable across users. This is where the sorted set earns its place: `ZREVRANK player` returns that player's exact position — "you are #4,271,908" — directly in O(log N), with no scan and no per-user precomputation. To show the player in context (the "you're here" strip with players just above and below), a `ZREVRANGE` around the returned rank fetches the neighbors. Both operations are logarithmic, so even a rank query for the 40-millionth player is a few dozen pointer hops. The subtlety at scale is that on a **single** sorted set the rank is **exact and trivial**; the difficulty only appears once the board is **sharded** across nodes, where no single node knows the global position — the next section's problem. The principle: serve the shared **top-K from a short-lived cache** over a range read, and serve **per-player rank directly from the sorted set** with `ZREVRANK` plus a neighbor range — the per-player rank being precisely the query that justifies a sorted set over a cached top list, since it can't be precomputed for millions of distinct users.

<h2 id="sharding-scale">Sharding & Approximate Rank</h2>

One sorted set on one node handles a remarkable amount — millions of players, hundreds of thousands of ops/sec — but eventually a board outgrows a single node's memory or, more often, its **write throughput** (the hot-board ceiling). Then you **shard**: partition the players across N sorted sets on N nodes, typically by hashing `player_id`. Sharding fixes write and memory scaling, but it **breaks the two reads in different ways**, and articulating that asymmetry is the mark of a strong answer.

**Top-K survives sharding cheaply.** The global top-K is the merge of each shard's *local* top-K: ask every shard for its top K (each an O(log N + K) read), then merge the N small lists and take the overall top K. With N shards and small K that's trivial, so leaders are still fast.

**Exact global rank does not.** A player's global rank is "how many players, across *all* shards, outscore them" — but each shard only knows its own members, and the player lives on just one shard. Computing it exactly means fan-out: send the player's score S to **every** shard, have each return `ZCOUNT (S, +inf)` (how many of its members beat S), and sum the counts plus one. That's correct but costs a scatter-gather to every shard on every rank query — expensive at scale and getting worse as shards grow. So production systems **approximate**: maintain a coarse **score-bucket histogram** per shard (counts of players in score ranges, cheap to keep and periodically refreshed), and estimate a rank by summing the histogram buckets above the player's score across shards — an O(buckets) lookup, no per-query fan-out. Since a rank of "#4,271,908" versus "#4,271,950" is meaningless to the user, this approximation is essentially free accuracy-wise and hugely cheaper. The principle: **shard by player to scale writes and memory**, accept that **top-K merges cheaply** while **exact cross-shard rank requires a costly fan-out count**, and lean on the requirement that **rank may be approximate** — using per-shard score histograms to estimate global rank without scatter-gather.

<h2 id="time-windows">Time-Windowed Leaderboards</h2>

Real products rarely want only an all-time board; they want **daily**, **weekly**, **monthly**, and **seasonal** boards ("today's top players", "this week's leaders"). The clean model is **one sorted set per window**, keyed by the period: `board:daily:2026-07-20`, `board:weekly:2026-W29`, `board:global`. A score submission then **fans out** to the currently-active windows — a single "+150" writes to the daily, weekly, and all-time sets — so each window maintains its own independent ranking, and a query just targets the relevant key.

Two consequences follow. First, **write amplification**: with W active windows, each score update becomes W sorted-set writes. This is usually fine (W is small — daily, weekly, all-time), but it's the cost to name, and it's why you don't create windows you won't query. Second, **expiry**: past daily boards mostly stop being interesting, so window sets are given a **TTL** and expire automatically (yesterday's daily board evaporates after a retention period), or are archived to the durable store and dropped from memory — which keeps total memory bounded even though windows multiply the number of sets. **Rolling windows** ("last 24 hours" rather than "calendar today") are harder: a true sliding window means old scores must *leave* the set as they age, which a plain sorted set doesn't do on its own — so they're approximated with fine-grained buckets (e.g. per-hour sets summed over the last 24) or handled with a time-decayed score. The principle: model each time window as its **own keyed sorted set** with a **TTL**, **fan out** each score to the active windows (accepting bounded write amplification), and approximate true rolling windows with bucketed sub-periods rather than trying to age individual entries out of one set.

<h2 id="persistence-truth">Persistence & the Source of Truth</h2>

An in-memory sorted set is fast but **volatile** — a node restart or crash can lose it — so the design must be explicit that **the sorted set is an index, not the system of record**. Every score change is durably written to a **source-of-truth store** (a database or an append-only log of ScoreEvents), and the sorted set is treated as a **derived structure that can be rebuilt** from that record. This is the same discipline as a search index or a cache in front of a database: the fast structure is disposable, the durable log is authoritative.

That makes recovery conceptually simple: if a sorted-set node is lost, **replay the scores from the durable store** to reconstruct the ranking (load current scores per player and `ZADD` them back), or fail over to a **replica**. Redis-style systems also offer their own persistence (snapshots and append-only files) that speed recovery by reloading a recent state rather than replaying from scratch — but the external source of truth remains the backstop, because you never want the *ranking* store to also be the *only* copy of the scores. The tension is the familiar one from the write-path section: **write-behind** persistence keeps writes fast but risks losing the last few unpersisted updates on a crash (acceptable when a lost point is quickly re-earned), while **write-through** guarantees durability at higher write latency (for stakes where a lost score matters). The principle: treat the sorted set as a **rebuildable derived index** and keep an independent **durable source of truth** for scores, so a node loss is recovered by **replaying or failing over** rather than by data loss — choosing the persistence timing to match how costly a lost update is.

<h2 id="scalability-evolution">Scalability & Evolution</h2>

Don't build the sharded, histogram-approximated, multi-window system on day one. Each stage answers a specific pressure and adds a specific cost — name the trigger before taking on the complexity.

<TradeoffTable
  columns={["Stage", "Trigger", "Cost"]}
  rows={[
    ["SQL table + ORDER BY", "A small board, low update rate", "Full sort per top-K and an O(rank) COUNT for per-player rank — dies under updates and rank queries"],
    ["Single in-memory sorted set", "Real-time updates; need per-player rank", "One node caps write throughput and holds the whole board; volatile without a durable backstop"],
    ["Replicated sorted set + cache + durable store", "Read-heavy load; durability", "Replicas for read scale, a top-K cache, and async persistence to the source of truth"],
    ["Sharded sets + approximate rank + time windows", "A board too hot/large for one node; windowed boards", "Shard by player; cross-shard top-K merge; histogram-approximated global rank; per-window fan-out"],
  ]}
  recommendedRow={2}
/>

The progression tracks the wall you're hitting. A **SQL `ORDER BY`** is fine for a small board but re-sorts per query and computes per-player rank with an O(rank) count — it collapses under both updates and rank lookups. A **single in-memory sorted set** is the pivotal move: real-time O(log N) updates and, crucially, O(log N) per-player rank — but one node caps write throughput and the set is volatile. The center of gravity for a real system is a **replicated sorted set with a top-K cache and a durable store behind it** — reads scale across replicas, the hot top-K is cached, and the durable log makes the in-memory index recoverable. Only genuine scale — a board too hot or too large for one node, or many time windows — needs **sharding with approximate cross-shard rank and per-window fan-out**, paying scatter-gather-or-approximate rank and write amplification for it. Most leaderboards live happily at "replicated sorted set, cache, durable store."

<h2 id="resiliency-failure-modes">Resiliency & Failure Modes</h2>

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. The leaderboard's guiding rule: **the sorted set is a rebuildable index, so most failures degrade or recover rather than lose data — provided the durable source of truth is intact.**

  rows={[
    { failure: "Sorted-set node crashes", impact: "A board (or shard) unavailable", detection: "Health check / replica lag", mitigation: "Replica failover; the set is derived, not the source of truth", recovery: "Promote replica or rebuild from the durable store" },

The theme: **derive, don't depend; degrade, don't lose.** Because the sorted set is a derived index over a durable log, a crashed node is recovered by promoting a replica or replaying scores — no data loss as long as the source of truth holds. Persistence lag is ridden out on the in-memory set while the stream backlog drains. Hot boards and shard imbalance are throughput problems answered by sharding, coalescing, and rebalancing. The top-K cache is protected from stampedes by single-flighting its refresh and serving slightly stale results. Correctness hazards — duplicate or out-of-order score events — are contained by **idempotent, conditional, versioned application** so a replayed or late event can't double-count or lower a score, with the durable log as the ultimate arbiter. And the deliberately-accepted "failure," an approximate rank being slightly off, isn't really one: it's the design trade, corrected on the next histogram refresh. Because scores are durably logged and the ranking is rebuildable, recovery is almost always "fail over, replay, or rebalance."

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Maintain each leaderboard as an in-memory sorted set (a Redis ZSET / skip list) so updates (ZINCRBY), top-K reads (ZREVRANGE), and per-player rank (ZREVRANK) are all O(log N); front the shared top-K with a short-lived cache; record every score durably to a source-of-truth store via an async stream so the sorted set is a rebuildable derived index; replicate for read scale; and when a board outgrows one node, shard by player — merging per-shard top-K and approximating global rank from per-shard score histograms — while modeling time windows as separate keyed sorted sets with TTLs."
  rationale="The hard part is a live ranking that answers both the shared top-K and an arbitrary player's rank cheaply under a stream of updates, and a skip-list sorted set makes all of update, range, and rank logarithmic — which SQL and arrays can't do simultaneously. The board is tiny (~5 GB), so it lives in RAM and the system is throughput-bound and read-heavy, met by replication plus a top-K cache. A durable store keeps the volatile index recoverable. Sharding scales writes/memory, and because rank may be approximate, histograms avoid per-query scatter-gather. Windows are independent sets so each has its own clean ranking."
  alternatives="SQL ORDER BY + COUNT for rank — simple but a per-query sort and an O(rank) count, dying under updates and rank lookups. A cached top-K list only — serves leaders but can't answer an arbitrary player's rank, which is the defining query. A sorted array — O(log N) rank but O(N) inserts under a write stream. Exact cross-shard rank via fan-out on every query — correct but a scatter-gather per request that scales poorly, so histogram approximation is preferred. Making the sorted set the only store — fast but loses scores on a crash; a durable source of truth is required. One giant set for all windows — loses the clean per-window ranking that separate keyed sets give."
  revisitWhen="A board's write rate or size exceeds a single node (shard it); exact rank becomes a product requirement (accept the fan-out cost or tighten histogram granularity); true rolling windows are needed (bucket sub-periods or time-decay scores); scores carry real stakes like money (switch write-behind to write-through); or friends/segment boards dominate (materialize filtered sub-boards rather than ranking the global set per query)."
/>

Four axes capture the real tension:

- **Sorted set vs SQL.** The sorted set makes update, range, and rank all O(log N); SQL sorts per query and counts O(rank) rows for per-player rank. The sorted set wins decisively for live ranking.
- **Exact vs approximate rank.** Exact cross-shard rank needs a per-query fan-out count; approximate rank from histograms is far cheaper and indistinguishable to users at scale. Approximate wins once sharded.
- **Write-behind vs write-through.** Write-behind is fast but can lose the last few updates on a crash; write-through is durable but slower. Choose by how costly a lost score is.
- **One set per window vs one set.** Per-window sets give clean independent rankings and easy TTL expiry at the cost of write fan-out; a single set can't express multiple windows. Per-window wins when you need windows.
