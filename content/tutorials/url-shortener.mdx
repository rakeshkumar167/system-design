<h2 id="interview-framing">Interview Framing</h2>

A URL shortener turns a long URL into a short code — `https://example.com/very/long/path?utm=...` becomes `https://sho.rt/aB3xK9q` — and redirects anyone who visits the short link back to the original. It looks trivial, which is exactly why it is a classic interview opener: the interesting parts are scale, latency, and the distributed-systems decisions hiding underneath a two-endpoint API.

Spend the first few minutes establishing scope rather than drawing boxes. Good questions to ask:

- What is the expected traffic, and what is the read-to-write ratio? (It is almost always read-heavy.)
- How long must links live — forever, or with expiry?
- Do we need custom aliases (vanity links)?
- Do we need analytics on clicks?
- What are the latency and availability targets for redirects?
- Who can create and delete links — anonymous users, or authenticated accounts?

<Callout variant="interview" title="Suggested 45-minute allocation">
**5 min** framing and scope · **5 min** requirements and capacity estimates · **5 min** API and data model · **10 min** high-level architecture and key generation · **15 min** deep dives (caching, partitioning, consistency, failure) · **5 min** trade-offs and wrap-up. Say your assumptions out loud and write them down — the interviewer is grading how you reason, not whether you memorized an answer.
</Callout>

This tutorial commits to a concrete set of assumptions so the numbers stay consistent end to end. **In scope:** create, redirect, custom alias, optional expiry, and aggregate click analytics. **Out of scope:** user-facing dashboards, billing, and link-preview generation — mention them, then set them aside.

<h2 id="requirements">Requirements</h2>

Separate what the system must *do* from how *well* it must do it. The non-functional targets are what actually drive the architecture.

Two requirements dominate every later decision. First, the system is **overwhelmingly read-heavy**: redirects outnumber creations roughly 100 to 1, so the redirect path is what we optimize and cache. Second, **redirects must be fast and always available** — a slow or failing redirect is a broken link on someone else's website, so we favor availability and low latency over strict global consistency on the read path.

<h2 id="capacity-estimates">Capacity Estimates</h2>

Estimates are directional, not precise. Their job is to tell us *which* component is under pressure. Start from one assumption — new links per month — and derive the rest.

<h2 id="key-generation">Key Generation</h2>

This is the heart of the problem. We need short codes that are unique, hard to guess in bulk, cheap to generate, and that use the key space efficiently. A 7-character Base62 code (`0-9A-Za-z`) yields **62⁷ ≈ 3.5 trillion** combinations — far more than the ~6 billion we'll create in five years, so 7 characters give us comfortable headroom.

<TradeoffTable
  columns={["Strategy", "Collisions", "Predictable?", "Coordination", "Notes"]}
  rows={[
    ["Hash + truncate", "Possible — must check", "Somewhat", "None", "Hash the URL, take 7 chars; needs collision retry"],
    ["DB auto-increment + Base62", "None", "Highly (sequential)", "Single sequence", "Simple but a write bottleneck and enumerable"],
    ["Distributed ID + Base62", "None", "Low (with sharding)", "Coordinated, cheap", "Snowflake/range-allocated IDs encoded to Base62"],
    ["Random token + check", "Rare — retry on hit", "No", "None", "Generate random 7 chars; check-and-set on write"],
  ]}
  recommendedRow={2}
/>

<DecisionRecord
  title="Key generation"
  choice="Allocate unique numeric IDs from a distributed generator (Snowflake-style or a range/ticket server), then Base62-encode them to 7-character codes."
  rationale="IDs are unique by construction, so there is no collision check on the write path. A range/ticket server hands each write node a block of IDs (e.g. 1,000 at a time) so ID allocation is not a per-request bottleneck. Base62 keeps codes short and URL-safe."
  alternatives="Random tokens with a conditional 'insert if not exists' are a fine, slightly simpler alternative when unguessability matters more than density — at the cost of occasional retries as the key space fills."
  revisitWhen="You need unguessable links (e.g. private shares): switch to random tokens or add random padding so codes can't be enumerated by incrementing."
/>

A subtlety: purely sequential IDs make links **enumerable** — anyone can walk `aaaa0`, `aaaa1`, … and scrape every link. Interleaving shards or adding a few random bits before encoding breaks the obvious sequence while keeping uniqueness.

  answer={1}
  explanation="Auto-increment is unique and simple, but every create must touch one sequence, which caps write throughput and adds a failure point. Distributed generators (or range allocation) let many nodes mint IDs independently while staying globally unique."

<h2 id="high-level-architecture">High-Level Architecture</h2>

The defining move is to **split the write path from the read path**. They have wildly different traffic (1:100), different latency budgets, and different scaling needs, so they get separate services that scale independently and share only the mapping store.

<ArchitectureDiagram />

- **Edge / CDN + API Gateway** terminate TLS, route, and apply rate limits. Redirects can often be served right at the edge.
- **Write Service** validates the URL, reserves a key from the **ID Generator**, persists the mapping, and kicks off asynchronous abuse scanning.
- **Redirect Service** is the hot path: it reads the **Distributed Cache** first and only falls back to the **Mapping Store** on a miss. It emits a click event to the **Event Stream** and returns the redirect — it never waits on analytics.
- **Analytics Worker** consumes the stream, aggregates counts, and writes the **Analytics Store**, fully decoupled from redirects.

<h2 id="detailed-flows">Detailed Flows</h2>

**Creating a link.** Validate, reserve an ID, encode, persist, return. Abuse scanning is fire-and-forget.

<CreateUrlSequence />

**Redirect — cache hit (the ~95% case).** One cache read, then respond. The click event is emitted asynchronously.

<RedirectCacheHitSequence />

**Redirect — cache miss.** Read the store, backfill the cache with a TTL, then respond. A negative cache entry for unknown codes prevents repeated misses from hammering the store.

<RedirectCacheMissSequence />

**Recording analytics.** Fully decoupled. If this pipeline lags or fails, redirects are unaffected and events are replayed on recovery.

<AnalyticsSequence />

For a **disabled or expired** link, the redirect service treats a `disabled` status or a past `expires_at` as a non-resolution and returns `410 Gone` (expired) or `404` (deleted), after the cache entry has been invalidated.

<h2 id="storage-partitioning">Storage & Partitioning</h2>

The access pattern is dead simple: **look up one row by primary key**. There are no joins, no range scans on the hot path, and the data is enormous. That points away from a single relational database and toward a horizontally partitioned key-value or wide-column store.

<TradeoffTable
  columns={["Option", "Fit", "Why / why not"]}
  rows={[
    ["Single relational DB", "Poor at scale", "Great consistency and queries we don't need; one node can't hold ~3 TB / 11k QPS"],
    ["Sharded key-value (e.g. DynamoDB/Cassandra)", "Strong", "Point lookups by key, horizontal scale, tunable consistency — matches the pattern exactly"],
    ["Wide-column", "Strong", "Same benefits; good if we later add per-owner listing or time-series columns"],
  ]}
  recommendedRow={1}
/>

<DecisionRecord
  title="Mapping store"
  choice="A sharded key-value store partitioned by short_code (hash partitioning)."
  rationale="Every read and write is keyed by short_code, so hashing the code spreads load evenly across partitions and makes lookups a single-partition operation. Replication (typically 3×) gives durability and read availability."
  alternatives="A relational primary with read replicas works at small scale and is simpler to operate, but it caps out and forces a painful migration later."
  revisitWhen="You add access patterns the key alone can't serve (e.g. list all links for an owner) — then add a secondary index or a separate owner→codes table."
/>

Because the partition key is the code itself and codes are uniformly distributed, **hot partitions** are unlikely from key skew — but a single viral link is hot on *reads*, which the cache (not the store) absorbs. Rebalancing is handled by the store's consistent-hashing ring; backups are periodic snapshots plus a change log for point-in-time restore.

<h2 id="caching">Caching</h2>

Caching is what makes the 100:1 read ratio affordable. We use a **cache-aside** pattern: the redirect service checks the cache, and on a miss reads the store and populates the cache with a TTL.

- **TTL** bounds staleness and lets cold entries age out; the ~600 GB hot set fits in memory.
- **Negative caching** stores "this code does not exist" briefly so floods of bad codes don't stampede the store — with a short TTL so a newly created code becomes resolvable quickly.
- **Invalidation** on disable/delete proactively evicts the entry so dead links stop resolving fast (this is also why we avoid 301s).
- **Hot-key replication** spreads a single viral code across cache nodes so one key doesn't overload one node.

<Callout variant="warning" title="Two failure modes to name explicitly">
**Cache stampede:** when a hot key expires, thousands of concurrent misses hit the store at once. Mitigate with request coalescing (single-flight) so only one miss recomputes while others wait, plus jittered TTLs. **Cache unavailable:** the redirect service must fall back to the store and keep serving — degraded latency, not an outage. Size the store to survive cache loss, or accept temporary latency breaches.
</Callout>

<h2 id="consistency-concurrency">Consistency & Concurrency</h2>

We deliberately choose **strong consistency where correctness demands it and eventual consistency everywhere else.**

- **Read-after-write:** a creator should see their new link resolve immediately. Write to the store, then optionally warm the cache, before returning the short URL. Reads of a brand-new code on a different region may briefly miss until replication catches up.
- **Alias uniqueness:** two users racing for the same custom alias is resolved by a **conditional insert** on `AliasReservation` — exactly one wins with a `409` for the other. No locks, no race.
- **Stale cache:** bounded by TTL and eliminated on disable/delete by explicit invalidation.
- **Disable/delete propagation:** update status in the store, invalidate the cache; replicas converge within replication lag. Using 302 (not 301) ensures browsers don't keep a private permanent cache.
- **Analytics consistency:** intentionally eventual — counts may lag by seconds. That is the right trade for keeping redirects fast.

<h2 id="security-abuse">Security & Abuse</h2>

A URL shortener is an attractive tool for attackers: it hides malicious destinations behind a trusted-looking domain. Treat safety as a first-class requirement.

- **Malicious destinations:** scan new URLs against a safe-browsing/threat feed asynchronously; quarantine or block links that resolve to malware or phishing.
- **Open-redirect & unsafe schemes:** allow only `http`/`https`; reject `javascript:`, `data:`, and links back into your own auth flows.
- **Enumeration:** sequential codes are scrapeable — add random bits or use random tokens for private links; rate-limit the redirect endpoint per IP.
- **Abuse & quotas:** rate-limit creation per account/IP; cap links per free account; require auth for high-volume creation.
- **Alias squatting:** reserve a deny-list of sensitive aliases (brand names, `admin`, `login`); consider verification for vanity namespaces.
- **Authorization:** only the owner can disable/delete or view a private link's stats.
- **Privacy & retention:** store click analytics coarsely (country, not precise location), and expire raw events on a short schedule.

<Callout variant="warning" title="Why 302, restated as a security property">
A `301 Moved Permanently` is cached by browsers and proxies, so a link flagged as malicious *after* creation may keep resolving from caches you don't control. A `302` keeps you in the loop on every click, which is exactly what you want for revocation and abuse response.
</Callout>

<h2 id="observability">Observability</h2>

Define the signals that tell you the system is healthy before users do. Track them per region.

- **Redirect latency (p50/p99)** — the headline SLI; alert on p99 > 50 ms.
- **Redirect success rate** — fraction of 3xx vs 4xx/5xx.
- **Cache hit rate** — a drop predicts store pressure and latency.
- **Creation error rate** — surfaces ID-generator or validation problems.
- **Store latency & error rate** — primary dependency health.
- **Replication lag** — bounds read-after-write across regions.
- **Event-stream consumer lag** — analytics freshness.
- **Abuse blocks / rate-limit rejections** — security posture.
- **Availability by region** — drives failover decisions.

<DepthSection depth="advanced">
Pair RED metrics (Rate, Errors, Duration) on every service with the four golden signals on dependencies. Emit a trace ID through the redirect path so a slow p99 can be attributed to cache, store, or network. Synthetic probes from multiple regions catch regional failures faster than waiting for real-user error spikes.
</DepthSection>
