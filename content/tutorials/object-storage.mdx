<h2 id="interview-framing">Interview Framing</h2>

Object storage is the system behind Amazon S3, Google Cloud Storage, and Azure Blob — a service that stores an essentially unlimited number of **immutable blobs** (files, images, videos, backups, data-lake objects) under string keys and serves them over HTTP. Half the internet's data sits in one. From the outside the API is trivially simple — `PUT` bytes under a key, `GET` them back — which is exactly why the naive framing ("it's a giant key-value store for files") misses the entire problem. Anyone can store a file; the reason object storage is a hard system is that it must keep **exabytes of data across millions of disks safe for years**, where disks are failing, corrupting, and being replaced *constantly*, and lose essentially nothing — the headline promise is **eleven nines of durability** (99.999999999%), meaning if you store ten million objects you'd expect to lose one every ten thousand years.

The hard part, then, is **durability at scale, affordably** — plus the machinery that scale forces. Four things dominate, and the API is the easy one. First, **durability**: at exabyte scale, keeping three full copies of everything (simple replication) is financially brutal, so the system uses **erasure coding** (Reed-Solomon) to get *better* durability than 3× replication at roughly **1.5×** the storage, by splitting each object into data and parity fragments spread across independent failure domains. Second, **metadata at massive scale**: the namespace is **flat** (a key maps to an object; there are no real directories), but there are **trillions** of keys, so a separate **metadata service** must index them — small per entry, enormous in count. Third, **huge objects**: a single object can be terabytes, so uploads use **multipart** — parallel, independently-retryable parts — because you can't stream a terabyte over one flaky connection. Fourth, **silent corruption**: bits rot on disk undetectably, so a **background scrubber** continuously re-reads and checksums fragments and **repairs** them, which is what actually *sustains* durability over time — the initial write is not enough. The reframe to lead with: *the API is a key-value PUT/GET, but the system is a durability engine — erasure coding for affordable 11-nines durability across failure domains, a sharded metadata service over a flat namespace of trillions of keys, multipart upload for enormous objects, and continuous background scrubbing and repair against the relentless disk failures and bit rot that would otherwise erode the data. Storing a blob is easy; guaranteeing you'll still have it, uncorrupted, in ten years, at exabyte scale and affordable cost, is the design.*

Spend the opening minutes scoping. Good questions to ask:

- What are we storing — **immutable blobs** of arbitrary size, addressed by key? (Yes — object storage, not a filesystem with in-place edits or a block device.)
- What's the **durability and availability** target? (11 nines durability is the headline; availability is typically ~4 nines — the two are different guarantees.)
- What **object-size range** — kilobytes to terabytes? It drives multipart and how small objects are packed.
- **Consistency** — is strong read-after-write for new objects required? (Modern object stores provide it.)
- **Scale** — how many objects and how much data? This decides erasure-coding vs replication and how the metadata is sharded.

This tutorial commits to concrete assumptions so the design stays consistent end to end. **In scope:** the flat-namespace metadata service, durability via erasure coding across failure domains, the write path and multipart upload, the read path with reconstruction, background scrubbing/repair/rebalancing, and immutability/versioning/consistency — plus scaling, failure modes, and trade-offs. **Out of scope:** the applications *built on top* of object storage — a file-sync product (**Cloud Drive**) or a video platform (**Video Streaming**) both *use* a blob store but add their own metadata, sharing, and delivery on top; here we build the blob store itself. Also out of scope: POSIX filesystem semantics (in-place edits, byte-range writes, hard links — object storage deliberately doesn't do these), block storage (virtual disks), and the CDN edge caching that fronts object storage for hot reads. This problem is the storage substrate the **Cloud Drive** and **Video Streaming** tutorials assume exists.

Three targets shape every later decision. First, **durability is the whole product**: 11 nines means the design must survive constant disk failures, whole-rack and availability-zone losses, and silent bit rot without losing data — which forces **erasure coding across independent failure domains** plus **continuous scrubbing and repair**, because durability isn't a property of the write, it's a property you *maintain* against entropy forever. Second, **scale demands cost efficiency**: at exabytes, the difference between 3× replication and 1.5× erasure coding is *half your entire storage bill* — literally exabytes of hardware — so erasure coding isn't an optimization, it's the only affordable way to hit the durability target. Third, **immutability simplifies everything**: objects never change in place — a "modification" writes a **new version** — which makes caching trivial (an object at a key+version never changes), enables deduplication, and sidesteps the hardest distributed-systems problem (concurrent in-place mutation). Note what's **relaxed**: object storage is **not** a filesystem — no partial in-place writes, no rename-as-move, no strong cross-object transactions — and **availability is a weaker guarantee than durability** (a brief inability to *serve* an object is tolerable; *losing* it is not). The system trades filesystem flexibility and some availability for extreme, affordable durability at unlimited scale.

The takeaways. First — the signature decision — **erasure coding roughly halves the storage bill versus replication**. A trillion 1 MB objects is **1000 PB (1 EB)** of logical data. Store it with **8+4 Reed-Solomon** (split each object into 8 data + 4 parity fragments) and you use **1500 PB** — a **1.5× overhead** — while surviving **any 4** simultaneous fragment losses. Store it with **3× replication** for comparable durability and you need **3000 PB**. Erasure coding **saves ~1500 PB** — an entire exabyte and a half of hardware — which at scale is the difference between a viable business and an impossible one. That's why every large object store uses erasure coding for its main tier. Second, **durability is about tolerating failures, not copies**: 8+4 tolerates 4 fragment losses, and if you spread those 12 fragments across 12 **independent failure domains** (racks, AZs), no single rack or zone failure can take out more than one, so correlated failures stay well under the 4-loss budget — this is how you reach 11 nines. Third — the part the byte math hides — **metadata is small but numerous**: a trillion objects means a *trillion* metadata entries (key → size, checksum, version, fragment locations), maybe a few hundred bytes each, so the metadata service is a large distributed database in its own right, sharded across the keyspace and kept entirely separate from the multi-exabyte data plane. The capacity story: size the data plane for **erasure-coded durability across failure domains** (the affordable path to 11 nines), and size the metadata plane for **trillions of tiny entries**, not bytes.

<h2 id="high-level-architecture">High-Level Architecture</h2>

The architecture is a **thin metadata plane** pointing into a **vast data plane**, with an **erasure coder** between the API and storage and a **background scrubber** keeping the data honest.

<ObjectStorageArchitecture />

- **Client.** Any HTTP client issuing PUT/GET/DELETE against a bucket and key.
- **Object API.** Stateless front end: authenticates, checksums, and orchestrates encode-on-write / fetch-and-reconstruct-on-read. Scales horizontally behind a load balancer.
- **Metadata Service.** A sharded, replicated datastore mapping (bucket, key, version) → size, checksum, and **fragment locations**. Small records, trillions of them, entirely separate from the bytes.
- **Erasure Coder.** Splits an object into **k data + m parity** fragments on write and reconstructs it from any **k** on read (often colocated in the API/storage tier, shown separately for clarity).
- **Storage Nodes.** The dumb, vast data plane: disks holding erasure-coded **fragments**, deliberately spread across **independent failure domains** (disks, racks, AZs).
- **Scrubber / Repair.** A continuous background service that re-reads fragments, verifies checksums, reconstructs corrupt or lost ones, and rebalances on node changes.

The decisive boundary is the **separation of a small, indexed metadata plane from a huge, dumb data plane**, joined by erasure coding: metadata knows *where* every fragment is, the data plane just stores fragments durably, and the coder turns objects into fragments and back. Every section that follows elaborates one piece: the metadata service and flat namespace, durability via erasure coding, the write and read paths, background scrubbing/repair, and consistency/versioning.

<h2 id="metadata-service">The Metadata Service & Flat Namespace</h2>

The first thing to get right is that object storage has a **flat namespace**, not a filesystem tree. A key like `photos/2026/sunset.jpg` looks hierarchical, but the slashes are just characters in an opaque string — there is no `photos` directory object, no `2026` folder, no tree to traverse. What looks like folders in a UI is produced by **listing keys with a common prefix** and a delimiter (`prefix=photos/2026/, delimiter=/`), which the system answers as a **prefix scan** over a sorted key index. This flatness is a deliberate simplification: no directory locks, no recursive rename, no tree rebalancing — a billion keys under one prefix is fine, because there's no actual directory to bottleneck.

That flat namespace is indexed by the **metadata service**, which is a serious distributed database in its own right, separate from the data plane. Each entry is small — (bucket, key, version) → size, checksum, storage class, and the **fragment locations** — but there are **trillions** of them, so the metadata service is **sharded** across the keyspace (by hash of the key for even distribution, or by key range to keep prefix-listing efficient) and **replicated** for its own durability and availability. It's on the critical path of every request (a GET is a metadata lookup then a fragment fetch), so it must be fast and highly available, and its own scaling — sharding, replication, hot-shard handling — mirrors the **Distributed Key-Value Store** problem. The separation is the key architectural decision: keeping metadata apart from data means you can **scale, index, and strongly-consistency the small metadata independently** of the vast, dumb, eventually-consistent bytes. The principle: object storage is a **flat key→object map** where prefixes only *simulate* folders (LIST is a prefix scan), and the **metadata service is a sharded, replicated index** of trillions of small records — kept separate from the data plane so the two scale on their own terms.

<h2 id="durability-erasure-coding">Durability & Erasure Coding</h2>

This is the centerpiece. To not lose data when disks constantly fail, you need **redundancy** — but *which* redundancy scheme decides whether the system is affordable. The simple option is **replication**: keep N full copies (typically 3) on different nodes, so you survive N−1 losses. It's simple and fast, but at exabyte scale **3× is ruinous** — you're buying three exabytes of hardware to store one exabyte of data.

**Erasure coding** gets *more* durability for *far less* overhead. With **Reed-Solomon (k, m)**, you split an object into **k data fragments**, compute **m parity fragments** from them, and store all **k+m**. The mathematical guarantee: **any k of the k+m fragments** are enough to reconstruct the original object, so you can lose **any m** fragments and still recover perfectly. The overhead is just **(k+m)/k**. With **8+4** (our capacity model): split into 8 data + 4 parity = 12 fragments, tolerate **any 4** losses, at **1.5×** overhead — versus 3× replication tolerating only 2 losses. Erasure coding gives **better durability at half the storage cost**, which is why it's the main tier of every large object store.

The subtlety that makes it actually durable is **failure domains**. Tolerating "any 4 fragment losses" only helps if losses are **independent** — so you place the 12 fragments across **12 different failure domains**: different disks, different nodes, different racks, ideally different availability zones. Then a whole rack dying takes out at most one fragment, a whole AZ failing takes out a bounded few, and correlated failures stay comfortably under the 4-loss budget — which is how you get to **11 nines**. The costs are real and worth naming: erasure coding spends **CPU** to encode on write and **decode/reconstruct** on degraded reads, and a **small-object penalty** (splitting a tiny object into 12 fragments has overhead), so very small objects are often replicated or packed together. The principle: use **erasure coding (Reed-Solomon k+m)** — any k of k+m fragments reconstruct, tolerating m losses at (k+m)/k overhead — instead of replication, and spread fragments across **independent failure domains** so correlated failures can't exceed the parity budget, buying **11-nines durability at ~1.5×** the storage cost of the data.

<h2 id="write-path-multipart">The Write Path & Multipart Upload</h2>

The **write path** turns the durability design into a careful sequence. On a PUT, the API first **checksums** the incoming bytes (both to verify the client's upload and to store as the ETag), the erasure coder **splits** them into k+m fragments, and those fragments are **written across storage nodes in separate failure domains**. Critically, the **metadata is committed last**: only after the fragments are confirmed durably stored does the API write the (key → fragment locations) record. This ordering gives **atomic visibility** — until metadata is committed, the object simply doesn't exist to any reader, so a crash mid-write never exposes a half-written object; the orphaned fragments left behind are just **garbage-collected** by a background sweep. The write isn't acknowledged until enough fragments (a write quorum) are durable, so a returned 200 genuinely means "safe."

For large objects, a single PUT is impractical — you can't reliably stream a terabyte over one HTTP connection that might drop at 90% — so object stores provide **multipart upload**.

<MultipartUploadSequence />

The client **initiates** the upload (getting an `uploadId`), splits the object into **parts** (each ~5 MB to 5 GB), uploads the parts **in parallel** and **independently** (each part is its own request with its own retry), and finally issues a **complete** call listing the parts and their ETags, which the system **assembles** into one logical object. The wins are large: parts upload **concurrently** for throughput, a **dropped part is re-sent alone** rather than restarting the whole transfer, and an abandoned upload is **abortable and garbage-collected** so a failed multi-TB transfer costs nothing permanent. It also enables resumable uploads and uploading an object whose total size you don't know upfront. The principle: on write, **erasure-code and distribute fragments across failure domains, then commit metadata last** so objects appear atomically and partial writes are invisible and GC'd; and for large objects use **multipart upload** — parallel, independently-retryable parts assembled on completion — so huge transfers are fast, resilient to flaky networks, and cheap to abort.

<h2 id="read-path-reconstruction">The Read Path & Reconstruction</h2>

The **read path** is where erasure coding quietly pays off in latency and availability, not just cost. On a GET, the API **looks up the object's fragment locations** in the metadata service, then **fetches fragments** from the storage nodes. Here's the elegant part: it needs **any k** of the k+m fragments. In the common case, all k **data** fragments are present, so the object is just their concatenation — a fast path with no math. But if a data fragment is **missing** (a dead node) or **slow** (a struggling disk, a hot node), the API doesn't wait — it fetches a **parity** fragment instead and **reconstructs** the missing data via the Reed-Solomon decode.

This "any k" property is a double win. For **failure tolerance**, a read succeeds as long as any k of the k+m fragments are reachable, so several nodes can be down and reads still work — availability for free from the same coding that gives durability. For **tail latency**, the API can issue requests to more than k fragments and **use the first k to respond**, treating the slow ones as if they'd failed and reconstructing around them — so one slow disk out of twelve never slows the read (the same hedged-request idea that tames tail latency elsewhere). Reads also support **byte ranges** (fetch just the fragments covering the requested range) for partial reads like video seeking and resumable downloads, and hot objects are fronted by a **CDN/cache** so most reads never touch storage at all. The principle: a read needs only **any k of the k+m fragments**, so the API fetches from the **fastest responders and reconstructs from parity** around slow or dead nodes — turning erasure coding into failure tolerance and tail-latency insurance on top of its storage savings.

<h2 id="background-integrity">Scrubbing, Repair & Rebalancing</h2>

Here's the truth that separates a real durability design from a naive one: **writing the data durably is not enough — durability decays, and must be actively maintained.** Disks suffer **silent corruption** (bit rot): a bit flips on a platter or SSD cell with no error reported, so a fragment that was written correctly is now subtly wrong, and you won't know until you read it — by which time other fragments may also have rotted, and you could be below the recovery threshold. And nodes fail permanently every day, each taking its fragments with it. If you did nothing, redundancy would erode fragment by fragment until objects crossed the m-loss threshold and were **lost forever**.

<ScrubRepairSequence />

The defense is a **background scrubber** that runs continuously: it systematically **re-reads every fragment** and **verifies its checksum**. When it finds a fragment that's **corrupt** (checksum mismatch) or **missing** (its node is gone), it **reconstructs** that fragment from the surviving fragments (the same Reed-Solomon decode as a degraded read) and **rewrites** it to a healthy node in a proper failure domain — **restoring full redundancy** before enough fragments are lost to threaten the object. This is what actually delivers 11 nines *over years*: durability is a **rate** — you must repair lost redundancy faster than it decays. Closely related is **rebalancing**: when nodes are **added** (spread data onto them), **removed**, or **fail** (their fragments must be re-replicated elsewhere), the system moves fragments to keep placement even and failure domains intact. The trade-off is background **I/O and CPU cost** — scrubbing constantly re-reads the entire dataset and repair consumes bandwidth — which is tuned against how fast you need to detect and heal corruption. The principle: durability is **maintained, not achieved once** — a continuous **scrubber** re-reads and checksums fragments and **repairs** silent corruption and lost fragments from parity, restoring redundancy faster than disks and nodes destroy it, with **rebalancing** keeping placement healthy as the fleet changes.

<h2 id="consistency-versioning">Consistency, Immutability & Versioning</h2>

Object storage leans hard on **immutability**, and it buys a lot. Objects are never modified in place — a PUT to an existing key doesn't edit bytes, it writes an entirely **new version** of the object, and the metadata service flips the key to point at it. This single choice dissolves the hardest problems in distributed storage: there's no concurrent in-place mutation to coordinate, an object at a given (key, version) **never changes** so it's **infinitely cacheable** (a CDN can hold it forever), and identical content can be **deduplicated** safely. "Editing" an object is re-uploading it; the old bytes remain their own immutable version.

On **consistency**, modern object stores provide **strong read-after-write for new objects**: the moment a PUT returns success, a subsequent GET is guaranteed to see it — achieved by making the **metadata commit the single linearization point** (the object exists exactly when its metadata is committed, and the metadata service provides read-your-writes on that record). Historically some stores were only *eventually* consistent here (a just-written object could 404 briefly), but the metadata-commit-last design makes strong read-after-write natural. **Overwrites and deletes** are typically read-after-write consistent on the metadata too, while **LIST** operations may lag slightly (the index catches up). **Versioning** builds directly on immutability: with it enabled, every PUT keeps the prior versions (so you can recover an overwritten or deleted object), and a **DELETE** just writes a **tombstone / delete-marker** rather than erasing data — the bytes are reclaimed later by lifecycle policy. The principle: objects are **immutable** (a write is a new version), which makes them cacheable, dedup-able, and simple to reason about; **strong read-after-write** consistency for new objects falls out of committing **metadata as the linearization point**; and **versioning + tombstoned deletes** give recoverability on top of the same immutable-version model.

<h2 id="scalability-evolution">Scalability & Evolution</h2>

Don't build the exabyte-scale, erasure-coded, multi-region system on day one. Each stage answers a specific pressure and adds a specific cost — name the trigger before taking on the complexity.

  rows={[

    ["Replicated across nodes", "Need to survive disk/node loss", "3× storage overhead; survives a couple of losses but expensive at scale"],
    ["Metadata-sharded + erasure-coded across failure domains", "Petabytes+; cost matters; 11-nines target", "Sharded metadata service; erasure coding (CPU); placement across racks/AZs; scrubbing/repair"],

The progression tracks the wall you're hitting. A **single-node blob store** is a toy — one disk, no durability. **Replication across nodes** is the first real durability step and is genuinely fine at small-to-medium scale, but its 3× overhead becomes untenable as you grow. The center of gravity for a real object store is a **sharded metadata service plus erasure coding across failure domains with background scrubbing** — this is where affordable 11-nines durability lives, and where most of this tutorial's design sits. Only at the largest scale do you add **multi-region and storage tiering**: replicate objects and metadata across regions for geo-durability and locality, and offer **storage classes** (hot "standard" on fast media, "cold/glacier" on cheap high-latency media) with **lifecycle policies** that transition objects to cheaper tiers as they age — trading retrieval latency for cost. Most systems are well-served by "erasure-coded across failure domains, metadata sharded, scrubbed continuously," adding regions and tiers as durability geography and cost pressure demand.

<h2 id="resiliency-failure-modes">Resiliency & Failure Modes</h2>

Reason about every dependency in the same matrix — blast radius, detection, mitigation, recovery. Object storage's guiding rule is stark: **durability is sacred and non-negotiable; availability can wobble, but data must never be lost** — so every failure mode is answered by redundancy, reconstruction, and background repair.

<FailureMatrix
  rows={[
    { failure: "Disk / storage-node failure", impact: "Some fragments lost", detection: "Health checks / read errors", mitigation: "Erasure coding: any k of k+m still reconstruct; reads unaffected", recovery: "Scrubber reconstructs lost fragments to new nodes; rebalance" },
    { failure: "Silent bit rot (corruption)", impact: "A fragment is wrong but 'present'", detection: "Checksum verification during scrub/read", mitigation: "Reject the bad fragment; reconstruct from parity", recovery: "Rewrite the repaired fragment; redundancy restored" },

/>

The theme: **redundancy tolerates, reconstruction heals, background repair sustains.** A dead disk or node is a non-event for reads because any k of k+m fragments reconstruct the object, and the scrubber rebuilds the lost fragments elsewhere. Silent corruption is caught by **checksums** (the only defense against bit rot — the disk won't tell you) and repaired from parity. Rack and AZ failures are survived because fragments are **spread across failure domains** so a whole-zone loss stays under the m-fragment budget, and the dangerous case — correlated loss *approaching* m — is met with **prioritized repair** of low-redundancy objects, racing to rebuild before crossing the line (durability as a rate again). Metadata shard loss is handled like any replicated datastore (fail over to a replica). Half-written uploads are harmless because **metadata-commit-last** means they're never visible, and GC reclaims the orphans. Hot objects are absorbed by a **CDN/cache** (trivial because objects are immutable). Because the data is erasure-coded across domains and continuously scrubbed, recovery is almost always "reconstruct from the survivors and rewrite" — and the one line never crossed is losing an object.

<h2 id="tradeoffs-alternatives">Trade-offs & Alternatives</h2>

Every recommendation here bought something and gave up something. Stating both is the difference between an answer and a strong answer.

<DecisionRecord
  title="The one-sentence design"
  choice="Store immutable objects under a flat (bucket, key, version) namespace, splitting each object with Reed-Solomon erasure coding (k+m, e.g. 8+4) into fragments spread across independent failure domains for ~11-nines durability at ~1.5× overhead; keep a small, sharded, replicated metadata service mapping keys to fragment locations, separate from the vast data plane; commit metadata last on write so objects appear atomically, use multipart upload for large objects, and reconstruct from any k fragments on read (bypassing slow/dead nodes); run a continuous background scrubber that checksums and repairs fragments to sustain durability, rebalancing as the fleet changes; and rely on immutability + metadata-as-linearization-point for cacheability and strong read-after-write, with versioning and tombstoned deletes for recoverability."
  rationale="Durability at exabyte scale must be affordable, and erasure coding across failure domains delivers better-than-3×-replication durability at ~1.5× cost — the decisive economic choice.   alternatives="3× replication instead of erasure coding — simpler and lower-CPU but double the storage cost at scale, untenable for exabytes (though still used for very small objects, where EC's per-fragment overhead hurts). Storing objects whole rather than as fragments — no reconstruction complexity but no cheap durability or any-k read resilience. Combining metadata and data — simpler but couples their very different scaling and consistency needs. A filesystem/POSIX interface — flexible in-place edits but reintroduces the coordination immutability removes. No background scrubber — durable at write time but decays to data loss as bit rot and node failures accumulate. Eventual consistency for new objects — simpler but surprises clients with missing-then-present reads."
  revisitWhen="Object-size distribution skews tiny (replicate or pack small objects to avoid per-fragment overhead); durability geography requires surviving a whole region (add cross-region replication); cost pressure warrants tiering (add cold storage classes + lifecycle policies); read patterns concentrate on hot objects (widen fragment placement / lean on CDN); or the metadata service becomes the bottleneck (reshard, add caching, or adopt a stronger distributed KV store underneath)."
/>

Four axes capture the real tension:

- **Erasure coding vs replication.** Erasure coding gives better durability at ~half the storage cost but spends CPU on encode/decode and hurts on tiny objects; replication is simple and low-latency but 3× the storage. Erasure code the main tier; replicate tiny objects.
- **Metadata/data separation vs unified.** Separating them lets each scale and be consistent on its own terms (the winning choice); unifying is simpler but couples very different needs.
- **Immutable vs mutable objects.** Immutability removes mutation coordination and enables caching/dedup at the cost of no in-place edits; object storage embraces it fully.
- **Strong vs eventual read-after-write.** Strong (via metadata-commit-last) is what clients expect and modern stores provide; eventual is simpler but surprises callers with briefly-missing new objects.
